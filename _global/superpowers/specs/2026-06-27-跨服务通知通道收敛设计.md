# 跨服务通知通道收敛设计

## 背景

goto 当前异步任务链路同时存在 RabbitMQ、Redis Stream 和 WebSocket 三类通道，职责边界已经混杂：

- RabbitMQ 仍承担通用任务和示例数据任务的服务间投递。
- RabbitMQ 仍承担 task 到 system 的任务完成通知转发。
- Redis Stream 已在 UserDraw 任务、取消和通知链路中使用，但尚未覆盖通用任务和示例数据。
- WebSocket 当前被前端大量使用，负责 system 到浏览器的实时通知。
- `freedom-public` 共享模块中仍保留 `NoticeType.RABBITMQ`、RabbitMQ 通知分支和 RabbitMQ 工具调用，导致上层项目可以继续引入旧通道。

本设计目标是先把消息通道的职责收敛清楚，再为后续长任务稳定性设计中的 Job Ledger、租约、fencing token、通知版本过滤提供统一基础。

## 目标

- Redis Stream 成为服务间唯一异步消息总线。
- WebSocket 仅保留为 system 到浏览器的最后一跳实时推送。
- RabbitMQ 在 `goto` 和 `freedom-public` 中全部下线，不保留兼容开关。
- 所有跨服务通知事件通过 Redis Stream 进入 system，再由 system 读取最新状态并推送 WebSocket。
- 前端现有基于 `commonNoticeUUID` 的异步回调能力保留，避免一次性重写所有页面为轮询。
- 为后续长任务状态机提供统一事件模型，支持 `statusVersion`、`fencingToken` 和旧通知过滤。

## 非目标

- 本设计不直接实现 Job Ledger 表结构和状态机细节；该部分由长任务稳定性设计覆盖。
- 本设计不把 WebSocket 替换为轮询、SSE 或 HTTP 长轮询。
- 本设计不保留 RabbitMQ 与 Redis Stream 双通道灰度。
- 本设计不迁移 RabbitMQ 中历史积压消息。
- 本设计不处理 `.worktrees/*` 中历史副本代码。

## 目标架构

统一后的链路：

```text
前端 / system HTTP 请求
→ system 写业务任务记录
→ system 投递 Redis Stream 任务命令
→ task 消费任务命令
→ task 执行业务任务
→ task 写业务状态或 Job Ledger
→ task 投递 Redis Stream 状态事件
→ system 消费状态事件
→ system 查询 DB 最新任务快照
→ system 通过 WebSocket 推送浏览器
```

职责边界：

```text
Redis Stream：服务间可靠投递、消费者组、Pending、重试、XAUTOCLAIM、DLQ
数据库：业务事实、任务状态、幂等记录、状态版本
WebSocket：浏览器实时展示，不承担服务间可靠性
RabbitMQ：删除
```

## 当前主要链路判断

### 通用任务链路

当前：

```text
system
→ CommonTaskRabbitMQUtils.publish
→ RabbitMQ QUEUE_COMMON_TASK_NOTICE
→ task CommonTaskConsumer
→ MessageHandler 执行业务
→ RabbitMQ QUEUE_COMMON_MESSAGE_NOTICE
→ system CommonTaskMessageConsumer
→ PublicWebSocketMultiple
→ goto-web commonNoticeUUID 回调
```

目标：

```text
system
→ CommonTaskCommand Stream
→ task CommonTaskCommandStreamConsumer
→ MessageHandler 执行业务
→ CommonTaskNotification Stream
→ system CommonTaskNotificationStreamConsumer
→ 查询最新任务状态
→ PublicWebSocketMultiple
→ goto-web commonNoticeUUID 回调
```

### 示例数据链路

当前：

```text
system
→ PlotTypeTaskRabbitMQUtils.publish
→ RabbitMQ QUEUE_GENERATE_SAMPLE_DATA_TASK_NOTICE
→ task GenerateSampleTaskConsumer
→ GenerateSampleDataThreadPool
→ RabbitMQ QUEUE_GENERATE_SAMPLE_DATA_MESSAGE_NOTICE
→ system GenerateSampleMessageConsumer
→ GenerateSampleDataWebsocket
→ 前端设置页
```

目标：

```text
system
→ GenerateSampleTask Stream
→ task GenerateSampleTaskStreamConsumer
→ GenerateSampleDataThreadPool
→ GenerateSampleNotification Stream
→ system GenerateSampleNotificationStreamConsumer
→ GenerateSampleDataWebsocket
→ 前端设置页
```

### UserDraw 链路

当前 UserDraw 已主要迁移到 Redis Stream：

```text
system
→ STREAM_USER_DRAW_TASKS / STREAM_USER_DRAW_CANCEL_TASKS
→ task UserDrawStreamConsumer / UserDrawCancelStreamConsumer
→ STREAM_USER_DRAW_NOTIFICATION
→ system UserDrawNotificationStreamConsumer
→ UserDrawWebsocket
→ 前端绘图页面
```

目标是保留该方向，但补齐统一事件信封、状态版本、旧通知过滤和命名规范。

## WebSocket 保留边界

WebSocket 当前不能整体删除，因为前端仍依赖以下连接：

- `public_multiple`：goto 主页面通用通知连接，使用 `commonNoticeUUID` 路由到统计、数据处理、工具箱、文件标记等子页面。
- `user_draw`：绘图任务完成后即时展示结果。
- `generate_sample_data`：示例数据生成结果通知。
- 登录、部署等独立 WebSocket 如仍有页面入口，应作为独立能力评估。

WebSocket 的新约束：

- 只能由 system 服务面向浏览器使用。
- task 服务不得直接面向浏览器推送。
- Redis Stream 消费者不得盲信通知事件中的最终展示内容。
- WebSocket 推送前必须读取数据库最新状态或最新任务快照。
- 推送消息必须携带可用于前端过滤乱序的版本字段。

## Redis Stream 事件模型

为避免不同业务继续各自定义消息结构，服务间事件建议统一成信封：

```json
{
  "eventId": "uuid",
  "eventType": "COMMON_TASK_SUCCEEDED",
  "businessType": "statistics",
  "businessId": "job-10001",
  "idempotencyKey": "statistics:run:pca:job-10001",
  "producerService": "goto-task",
  "occurredAt": "2026-06-27T10:08:00",
  "fencingToken": 2,
  "statusVersion": 8,
  "commonNoticeUUID": "frontend-page-uuid",
  "payload": {}
}
```

字段约束：

- `eventId`：事件自身幂等键。
- `eventType`：状态事件类型，例如提交成功、进度变化、任务成功、任务失败、任务取消。
- `businessType`：业务类型，例如 `common-task`、`generate-sample`、`user-draw`、`statistics`。
- `businessId`：业务主键或任务 ID。
- `idempotencyKey`：业务幂等键。
- `fencingToken`：长任务执行版本；短任务可为空。
- `statusVersion`：状态版本；system 和前端用来过滤乱序。
- `commonNoticeUUID`：前端页面级回调路由，保留现有页面体验。
- `payload`：只放最小必要事件数据，不作为最终展示真相源。

system 消费通知事件后应执行：

```text
读取事件
→ 按 eventId 幂等去重
→ 查询 DB 最新任务状态
→ 过滤旧 fencingToken / 旧 statusVersion
→ 构建 WebSocket 展示快照
→ 推送到 public_multiple / user_draw / generate_sample_data
→ ACK Redis Stream
```

## freedom-public 改造边界

`freedom-public` 是共享模块，RabbitMQ 能力不能只在 goto 中绕开，必须从共享抽象中下线。

需要删除或替换的共享能力：

- `NoticeType.RABBITMQ` 枚举值。
- `NoticeRelationData` 中 `exchange`、`routingKey` 作为 RabbitMQ 概念的公开字段。
- `common-thread` 的 `ThreadPool` 中 RabbitMQ 通知分支。
- `common-thread-r` 的 `CommonStage` 中 RabbitMQ 通知分支。
- 共享模块中对 `RabbitMQUtils`、`RabbitTemplate`、AMQP 类型的依赖。

替代设计：

- `NoticeType` 保留 `WEBSOCKET` 和 `REDIS_STREAM`。
- `NoticeRelationData` 改为面向通用事件通道：

```text
noticeType
personalizationNoticeType
streamKey
eventType
businessType
webSocketPath
```

- 线程池和 Stage 不直接理解 RabbitMQ。
- 线程池和 Stage 对 Redis Stream 的通知发送只依赖统一事件发布接口，例如 `TaskEventPublisher`。
- WebSocket 仍由 system 层消费事件后触发，公共线程模块不应直接决定跨服务投递实现细节。

## goto 改造边界

### 需要新增的 Redis Stream 消费者

- `CommonTaskCommandStreamConsumer`：替代 `CommonTaskConsumer`。
- `GenerateSampleTaskStreamConsumer`：替代 `GenerateSampleTaskConsumer`。
- `CommonTaskNotificationStreamConsumer`：替代 `CommonTaskMessageConsumer`。
- `GenerateSampleNotificationStreamConsumer`：替代 `GenerateSampleMessageConsumer`。

### 需要替换的生产者

- `CommonTaskRabbitMQUtils.publish` 替换为 `CommonTaskStreamPublisher.publish`。
- `PlotTypeTaskRabbitMQUtils.publish` 替换为 `GenerateSampleTaskStreamPublisher.publish`。
- `RabbitMQUtils.publish` 的通知调用替换为统一 `TaskEventPublisher.publish`。
- `SystemRebootInitUtils` 中 `NoticeType.RABBITMQ` 配置改为 `NoticeType.REDIS_STREAM`。

### 需要删除的 RabbitMQ 文件和配置

- `RabbitMQConstant`。
- `RabbitmqExchangeQueueConfig`。
- `RabbitTemplateConfig`、`CommonRabbitTemplateConfig`。
- RabbitMQ listener factory 配置。
- RabbitMQ 消费者包。
- RabbitMQ 发布工具类。
- Spring AMQP 依赖。
- application 中 RabbitMQ 配置。
- 测试中的 RabbitMQ mock 和说明。

## WebSocket 推送规则

WebSocket 不再承载服务间状态判断，只做最后展示。

通用推送规则：

```text
Redis Stream 状态事件
→ system 消费
→ 查询最新业务状态
→ 检查 commonNoticeUUID 或用户连接
→ 生成 BasicMessage 兼容前端
→ 推送 WebSocket
```

为了兼容现有前端，可以短期继续输出 `BasicMessage`，但内部事件模型不再以 `BasicMessage` 为核心。

前端过滤规则：

- 保留 `commonNoticeUUID` 页面路由。
- 新增或逐步使用 `statusVersion`。
- 收到旧版本消息时忽略。
- 终态消息不能被旧运行中或旧进度消息覆盖。

## 删除 RabbitMQ 的实施顺序

推荐分阶段：

1. 定义统一 Redis Stream 命令和事件信封。
2. 新增通用任务、示例数据任务的 Redis Stream 生产者和消费者。
3. 新增 task 到 system 的 Redis Stream 通知消费者。
4. 将 `SystemRebootInitUtils`、`CommonStage`、`ThreadPool` 的通知分支切到 Redis Stream。
5. 将 system 中所有 `CommonTaskRabbitMQUtils`、`PlotTypeTaskRabbitMQUtils` 调用点切到 Redis Stream。
6. 迁移测试 mock 和断言，确认前端 WebSocket 回调行为不变。
7. 删除 goto 中 RabbitMQ 消费者、生产者、配置、常量和 AMQP 依赖。
8. 删除 freedom-public 中 RabbitMQ 枚举、字段、分支和依赖。
9. 全局搜索 `RabbitMQ`、`Rabbitmq`、`RabbitTemplate`、`RabbitListener`、`org.springframework.amqp`，确保无业务残留。

删除最终动作必须在 Redis Stream 替代链路全部跑通后执行。

## 风险与约束

- `freedom-public` 是共享模块，删除 `NoticeType.RABBITMQ` 可能影响除 goto 外的其他项目；实施前必须全局搜索依赖方。
- 前端仍依赖 WebSocket，不应把 WebSocket 与 RabbitMQ 一起删除。
- Redis Stream 通知消费者必须保留 WebSocket 最后一跳，否则统计、数据处理、工具箱等页面会失去异步完成回调。
- 旧 RabbitMQ 测试用例需要改为 Redis Stream publisher/consumer mock，并按项目规则补齐中文测试注释。
- 如果短期未实现 Job Ledger，至少也要用业务状态和 `eventId` 幂等保护通知重复。

## 验收标准

- `goto` 和 `freedom-public` 源码中不再出现 Spring AMQP 业务依赖。
- `NoticeType` 不再包含 `RABBITMQ`。
- 通用任务、示例数据、UserDraw 的服务间通信都走 Redis Stream。
- task 服务不直接推送浏览器 WebSocket。
- system 作为唯一 WebSocket 推送服务，保留现有前端实时通知体验。
- 统计、数据处理、工具箱、文件标记、绘图、示例数据页面的异步回调不因 RabbitMQ 删除而中断。
- Redis Stream 通知事件重复投递时，system 不重复推送旧状态。
- 全局搜索 RabbitMQ 关键字时只允许历史文档或迁移说明中出现，不允许业务代码、配置和依赖残留。

## 与长任务稳定性设计的关系

本设计是上层通道收敛设计，先回答“服务间用什么通道、WebSocket 是否保留、RabbitMQ 删除到什么范围”。

长任务稳定性设计是下层可靠性设计，继续回答“单个任务如何防止重复执行、如何处理宕机、如何过滤旧 owner 通知”。

两者最终组合为：

```text
Redis Stream 统一通道
+ Job Ledger 状态机
+ owner_id / fencing_token / lease_until
+ system 端 WebSocket 推送
+ 前端 statusVersion 过滤
```
