# goto-web Dialog 标题栏空白区拖拽设计

## 1. 背景

`goto-web` 当前已经在统一 `Dialog` 组件中接入了基于 `size="sm|md|lg"` 的响应式布局能力，负责弹窗的：

1. 宽高按视口比例收敛
2. 水平、垂直居中
3. 小屏与矮视口下的 `safe-edge` 安全边距模式
4. 内容高度变化后的自动重算

在此基础上，用户希望进一步增强交互性：允许通过鼠标长按标题栏空白区域拖动弹窗位置，但不能破坏现有的自动布局基线。

本轮目标不是引入新的弹窗体系，也不是为所有弹层增加任意拖拽能力，而是在统一 `Dialog` 入口上提供一个可控、可回退、不会和响应式布局冲突的拖拽行为。

## 2. 已确认约束

本次设计基于以下已确认约束：

1. 拖拽只允许从标题栏空白区域启动
2. 标题文字区域不触发拖拽
3. 关闭按钮及标题栏中的其他交互元素不触发拖拽
4. 拖拽位置只在当前打开周期内有效
5. 弹窗关闭后，下次重新打开恢复默认居中
6. 当窗口 `resize` 或内容高度变化时，直接退出拖拽位置，恢复自动居中 / `safe-edge`
7. 拖拽能力建立在现有统一 `Dialog` 组件上，不改造各业务调用点

## 3. 目标

### 3.1 交互目标

- 用户可以通过按住标题栏空白区域拖动当前弹窗
- 拖拽行为不会抢占标题栏中已有的点击行为
- 拖拽后的弹窗仍保持在视口可见范围内
- 关闭、重开、窗口变化、内容变化后，弹窗都能自然回到组件自动布局

### 3.2 工程目标

- 所有拖拽逻辑集中在 [dialog.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/dialog.vue)
- 不要求业务页面感知拖拽实现细节
- 不破坏现有 `size`、`fullscreen`、`safe-edge`、内容高度监听逻辑

### 3.3 维护目标

- 拖拽能力作为自动布局的覆盖层，而不是新的定位体系
- 后续若需要禁用某些弹窗拖拽，可通过统一组件开关扩展
- 避免把拖拽状态持久化到业务页面或全局 store

## 4. 非目标

本轮不做以下事情：

1. 不支持记忆上次拖拽位置
2. 不支持触摸手势拖拽
3. 不支持从 body、footer 或任意区域拖拽
4. 不重做 Element UI 弹窗 DOM 结构
5. 不引入第三方拖拽库

## 5. 方案对比与决策

### 5.1 方案 A：直接改 `left/top`

优点：

- 直观
- 看起来最像传统拖拽实现

缺点：

- 容易和现有居中 / `safe-edge` / 视口收敛冲突
- 回退到自动布局时需要恢复更多定位状态
- 对 Element UI 默认定位耦合更深

### 5.2 方案 B：叠加 `transform: translate(...)`

优点：

- 自动布局仍然是基础坐标系
- 拖拽只表现为附加偏移，重置简单
- 与现有响应式布局兼容性最好

缺点：

- 需要显式处理视口边界裁剪
- 需要在自动布局重算时清空偏移

### 5.3 方案 C：引入拖拽库

优点：

- 少写底层鼠标事件

缺点：

- 本次需求过窄，引库收益低
- 适配 Element UI 结构和现有布局规则的成本不值得

### 5.4 最终决策

采用 **方案 B：叠加 `transform: translate(...)`**。

原因：

1. 最符合当前 `Dialog` 已有的自动布局架构
2. 拖拽状态可以在任意自动布局重算时清零，回退最自然
3. 实现边界清晰，不需要把 `Dialog` 变成手写绝对定位组件

## 6. 组件职责

拖拽能力只由 [dialog.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/dialog.vue) 负责：

- 识别是否命中标题栏空白区域
- 记录拖拽起点与原始偏移
- 在拖拽期间更新偏移量
- 对偏移量做视口边界裁剪
- 在关闭、重开、窗口变化、内容变化时重置拖拽状态

业务调用侧不承担拖拽状态管理职责。

## 7. 状态模型

建议状态只保留最小集合：

- `dragging`：当前是否正在拖拽
- `dragOffsetX` / `dragOffsetY`：当前打开周期内的拖拽偏移
- `dragStartX` / `dragStartY`：拖拽开始时鼠标位置
- `dragOriginX` / `dragOriginY`：拖拽开始时已有偏移

可选扩展：

- `draggable`：统一组件开关，默认 `true`

本轮若未显式引入 `draggable` 对外接口，也要保留内部结构可扩展性。

## 8. 事件模型

### 8.1 启动拖拽

触发条件：

- `mousedown` 发生在标题栏空白区域
- 当前不是 `fullscreen`
- 当前弹窗处于可见状态

启动时记录：

- 鼠标初始坐标
- 当前拖拽偏移

并绑定：

- `document.mousemove`
- `document.mouseup`

### 8.2 拖拽中

`mousemove` 期间：

1. 计算鼠标位移
2. 与拖拽开始时偏移叠加
3. 按当前视口对偏移做边界裁剪
4. 将裁剪后的偏移写入弹窗显示状态

为避免轻微抖动造成误触，建议引入拖拽阈值：

- 鼠标移动距离未超过 `4px` 前，不进入正式拖拽态
- 超过 `4px` 后才开始更新拖拽偏移

### 8.3 结束拖拽

`mouseup` 时：

- 退出拖拽态
- 解绑全局鼠标事件

## 9. 命中规则

只允许标题栏空白区域触发拖拽。

以下命中对象必须排除：

- `.el-dialog__title`
- `.el-dialog__headerbtn`
- `button`
- `input`
- `a`
- 带 `data-no-drag` 的元素

也就是说：

- 标题文字不可拖
- 关闭按钮不可拖
- 任何标题栏内未来新增的交互元素都应可通过 `data-no-drag` 主动退出拖拽命中

## 10. 渲染策略

### 10.1 自动布局仍是基础层

现有 `Dialog` 自动布局继续负责：

- `size` 对应的宽高
- 居中
- `safe-edge`
- 内容区滚动

### 10.2 拖拽只叠加偏移

拖拽期间不重写基础定位，只额外叠加：

- `transform: translate(dragOffsetX, dragOffsetY)`

默认值为：

- `translate(0, 0)`

这样可以保证：

1. 关闭或重算时只需清空偏移
2. 自动布局仍然是事实上的权威坐标系
3. `safe-edge` 切换时不会残留绝对定位副作用

## 11. 边界规则

拖拽偏移必须被限制在视口可见范围内。

最低要求：

- header 始终可见，保证用户可以再次抓住拖动
- 弹窗不能整体拖出屏幕

推荐实现上按当前弹窗 `getBoundingClientRect()` 和视口边界做裁剪：

- 左侧至少保留 `24px` 可见区域
- 右侧不能超出视口右边界
- 顶部至少保留 `48px` 标题栏可见区域
- 底部至少保留 `24px` 可见区域

若当前布局已经进入 `safe-edge`，拖拽边界仍以“视口可见优先”执行。

## 12. 回退规则

以下情况必须清空拖拽偏移并退出拖拽状态：

1. 弹窗关闭
2. 弹窗重新打开
3. 窗口 `resize`
4. 内容高度变化触发自动布局重算
5. 自动布局判断进入 `safe-edge`

这组规则保证：

- 拖拽不会覆盖自动布局的权威性
- 每次重新打开弹窗时，用户都从统一默认布局开始

## 13. 实施边界

本轮实施限定为：

- 修改 [dialog.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/dialog.vue)
- 增加或更新对应的单元测试 / source-contract tests

本轮不应修改业务调用点。

## 14. 验收标准

必须满足：

1. 默认打开仍保持当前的自动居中 / `safe-edge` 行为
2. 按住标题栏空白区域可以拖动
3. 点击标题文字不会触发拖拽
4. 点击关闭按钮不会触发拖拽
5. 拖拽释放后，当前位置在当前打开周期内保持
   仅在未触发 Section 12 回退条件时成立
6. 关闭再打开恢复默认居中
7. `resize` 后恢复自动布局
8. 内容异步变高后恢复自动布局
9. 长内容进入 `safe-edge` 时不会与拖拽状态冲突

## 15. 风险与注意事项

### 15.1 Element UI 内部结构耦合

拖拽命中区域和位移应用都需要依赖 Element UI 的 dialog/header DOM 结构。

因此实现时要尽量把结构耦合集中在一处，避免把 selector 散落在多个方法中。

### 15.2 transform 与现有布局叠加

当前 `Dialog` 已使用运行时样式和 class 驱动布局。拖拽偏移必须只作为附加层，不应再次引入新的定位体系。

### 15.3 内容重算时的状态切换

若拖拽状态和内容重算时序处理不好，容易出现：

- 弹窗闪回
- 偏移残留
- `safe-edge` 与拖拽叠加冲突

因此必须把“自动布局重算优先于拖拽偏移”作为硬约束。
