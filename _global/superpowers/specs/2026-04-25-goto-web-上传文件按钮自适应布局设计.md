# goto-web 上传文件按钮自适应布局设计

## 1. 背景

当前绘图侧边栏的“上传文件”区域位于 [PlotSidePanel.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/components/PlotSidePanel.vue)，按钮本体由 [uploaderBtn.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/components/form/draw/uploaderBtn.vue) 渲染。

现状中，上传文件区域被强制为纵向堆叠：

1. 父容器宽度较大时，两个上传按钮仍然上下排列
2. 横向空间利用率不足
3. 与同侧边栏内其他“宽时并列、窄时折行”的布局感受不一致

本轮希望只优化“上传文件”这一组按钮的排列行为：

- 父容器较窄时，每个按钮单独占一行，宽度为 `100%`
- 父容器较宽时，按钮自动并列显示

本轮不是按钮视觉重做，而是一次局部布局增强。

## 2. 已确认约束

本次设计基于以下已确认约束：

1. 只改“上传文件”这一组按钮
2. 不改“预览与输出”等其他参数模块
3. 不影响其他位置的参数格式与样式
4. 如果现有通用 class 存在波及风险，则必须新增上传文件专用样式类
5. 布局切换依据父容器可用宽度自动完成，不使用固定断点作为核心机制
6. 已上传态、未上传态、删除按钮、点击上传等现有交互保持不变

## 3. 目标

### 3.1 交互目标

- 上传按钮区域在窄宽度下自动切换为单列
- 上传按钮区域在宽宽度下自动切换为并列
- 按钮本体始终占满自己的网格单元，避免局部留白

### 3.2 工程目标

- 布局能力收敛在上传文件区域容器，不扩散到公共表单布局
- 按钮组件继续只负责按钮外观与内部内容排布
- 不修改 `DrawGroupOptional` 或其他公共参数表单栅格规则

### 3.3 维护目标

- 上传文件区域拥有独立命名的局部样式，避免未来误复用
- 后续若上传项数量增加，布局规则仍可自动扩展

## 4. 非目标

本轮不做以下事情：

1. 不改 `GUploaderBtn` 的视觉设计语言
2. 不调整“预览与输出”或其他参数区块的响应式行为
3. 不重构整个绘图侧边栏的表单布局体系
4. 不引入新的全局断点体系

## 5. 方案对比与决策

### 5.1 方案 A：保持当前纵向堆叠

优点：

- 风险最低
- 不需要额外布局规则

缺点：

- 无法利用宽容器空间
- 不符合本轮目标

### 5.2 方案 B：上传区域使用自适应网格

优点：

- 宽时并列、窄时单列，最符合目标
- 不依赖固定媒体查询断点
- 更适合侧边栏这类可变宽度容器
- 可通过容器类局部生效，隔离性好

缺点：

- 需要梳理当前上传区域已有的强制纵向样式
- 需要确保内部 `Uploader` 结构不会破坏网格宽度计算

### 5.3 方案 C：固定双列配合媒体查询

优点：

- 行为容易理解
- 对特定设计稿可控性高

缺点：

- 依赖人工选择断点
- 在不同侧边栏宽度场景下不够自然

### 5.4 最终决策

采用 **方案 B：上传区域使用自适应网格**。

原因：

1. 最贴合“根据父容器可用宽度自动换行”的已确认需求
2. 可将改动限制在上传文件区域，不波及其他参数模块
3. 对未来上传项数量变化更友好

## 6. 作用范围

本轮实现只允许触达以下区域：

- [PlotSidePanel.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/components/PlotSidePanel.vue) 中“上传文件”折叠面板对应的模板与局部样式

可选最小触达：

- [uploaderBtn.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/components/form/draw/uploaderBtn.vue) 保持按钮自身 `width: 100%` 能力；若当前已满足，则不修改

明确不触达：

- 公共参数表单栅格
- 其他参数组件模板
- 其他折叠面板内容区布局

## 7. 组件职责划分

### 7.1 上传区域容器职责

上传文件区域容器负责：

- 决定当前是单列还是多列
- 控制按钮项之间的间距
- 控制宽容器下的并列数量

建议新增上传专用类名，例如：

- `plot-upload-grid`
- `plot-upload-item`

这类类名只服务当前上传文件区域，不复用到其他参数区块。

### 7.2 上传按钮职责

[uploaderBtn.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/components/form/draw/uploaderBtn.vue) 继续只负责：

- 已上传 / 未上传状态视觉
- 图标、文件名、删除按钮排布
- 自身宽度占满父级单元

按钮组件不负责判断并列还是单列。

## 8. 布局策略

### 8.1 当前问题

当前 [PlotSidePanel.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/components/PlotSidePanel.vue) 中上传区域样式存在以下特征：

- `.file-settings-container` 被强制为 `flex-direction: column`
- `.file-settings` 被强制为 `width: 100%`
- 内部 `Uploader` 相关包装层也被设置为块级并撑满宽度

其中后两项可以保留，但第一项会阻止宽容器下并列。

### 8.2 新布局原则

新的布局遵循以下原则：

1. 外层容器改为上传专用的自适应 grid
2. 每个上传项仍保留 `width: 100%`
3. 内部 `Uploader` 结构继续撑满所在网格单元
4. 不依赖固定媒体查询断点作为主切换逻辑

建议形式：

```scss
grid-template-columns: repeat(auto-fit, minmax(240px, 1fr));
```

含义：

- 当容器足够宽时，上传项自动并列
- 当容器不足以容纳两列最小宽度时，自动退化为单列

`240px` 只是设计阶段建议值，落地时应以当前按钮内容、图标、文字和删除区的最小可读宽度为准做微调。

### 8.3 样式隔离

为避免影响其他参数模块，建议：

1. 模板层为上传文件容器和上传项新增专用 class
2. 新的 grid 规则只挂在这些专用 class 上
3. 原有通用类如果已被其他位置依赖，不直接改变其语义

也就是说，不把“上传文件自适应布局”抽象成公共参数样式，而是把它作为上传区特例管理。

## 9. 风险与规避

### 9.1 风险：影响其他参数布局

原因：

- 当前文件中已有一些通用命名的容器类
- 如果直接改这些类的布局语义，可能波及其他内容

规避：

- 新增上传专用类名
- 样式选择器只绑定到上传文件模板节点

### 9.2 风险：按钮内部内容在窄宽度下挤压

原因：

- 已上传态包含文件名和删除按钮
- 未上传态包含图标、文案和加号

规避：

- 保留 `uploaderBtn.vue` 现有 `width: 100%`
- 维持文字区域 `flex: 1` 与省略样式
- 通过 `minmax` 下限避免单元过窄

### 9.3 风险：折叠展开后宽度计算不稳定

原因：

- `el-collapse` 动画可能影响初始宽度

规避：

- 使用纯 CSS 自适应 grid，不引入运行时测量逻辑
- 尽量避免依赖 JS 监听容器宽度

## 10. 验证方案

实现完成后需要验证以下内容：

1. 宽容器下两个上传按钮并列显示
2. 窄容器下两个上传按钮自动变为上下排列
3. 单列时按钮占满整行 `100%`
4. 已上传态、未上传态、删除按钮和点击上传行为不受影响
5. “预览与输出”、普通参数区块、列表参数区块样式不发生变化
6. 折叠面板展开 / 收起后布局稳定，无明显抖动

## 11. 实施摘要

落地时建议按以下顺序执行：

1. 在 [PlotSidePanel.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/components/PlotSidePanel.vue) 模板中为上传区域添加专用 class
2. 将上传区域容器由纵向 flex 切换为自适应 grid
3. 保留上传项与 `Uploader` 内层的 `width: 100%` 规则
4. 本地验证宽窄场景与回归范围

## 12. 结论

本轮设计的核心不是修改按钮本身，而是让“上传文件”区域具备局部、自适应、可隔离的布局能力。

最终方案将：

- 只改变上传文件区域的排列方式
- 宽时并列，窄时单列
- 不影响其他参数模块
- 通过上传专用样式类控制影响范围

这满足当前交互目标，也符合“只做局部布局优化、不重写公共表单体系”的工程边界。
