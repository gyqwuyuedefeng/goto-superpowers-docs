# goto-web 开始分析前列名标记校验设计

## 1. 背景

当前绘图页入口位于 [draw.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/draw.vue)，用户点击“开始分析”后会直接调用前端 `custom_draw_plot` 接口。

现状中，“列名标记”模块自身已经在 [fileSetting.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/fileMark/fileSetting.vue) 内对部分操作做了必填校验，但该校验没有覆盖 `draw.vue` 的“开始分析”入口。同时，后端 [PlotServiceImpl.java](/mnt/f/IdeaProjects/goto-software/goto/system/src/main/java/com/freedom/service/impl/PlotServiceImpl.java) 的 `customDrawPlot(...)` 也没有对“必选列名标记是否已完成”做最终兜底校验。

这会导致两类问题：

1. 用户在前端未完成必选列名标记时，仍可能直接发起分析
2. 即使前端后续补了拦截，接口仍可以被绕过直接调用

本轮目标是在前后端同时补齐“开始分析前必须完成必选列名标记”的校验能力，并统一失败提示文案。

## 2. 已确认需求

本次设计基于以下已确认口径：

1. 校验范围是“各个上传文件对应的列名标记模块”
2. 只检查 `fileMarkContainerList` 中 `type = SELECT_COLUMN` 且 `required = true` 的项
3. 只要任意一个必选列名标记未完成，就阻止分析
4. 前端拦截时只弹出 `请先标记列名`
5. 前端不需要自动展开、切换或定位到缺失标记的文件面板
6. 后端失败提示统一为 `请先标记列名`
7. 若某个上传文件没有列名标记配置，或者没有 `required = true` 的项，则不拦截

## 3. 目标

### 3.1 交互目标

- 用户点击“开始分析”时，若缺少必选列名标记，立即得到统一提示
- 未完成标记时不发起分析请求
- 已完成标记时保持现有分析流程不变

### 3.2 接口目标

- `POST /api/plot/custom_draw_plot` 在服务端也执行同口径校验
- 防止脚本直调、旧页面缓存、绕过前端校验等场景提交非法分析请求

### 3.3 维护目标

- 前后端判断口径一致
- 不引入新的请求字段或数据库结构
- 尽量复用现有数据结构与位标记机制

## 4. 非目标

本轮不做以下事情：

1. 不改“列名标记”模块的保存交互
2. 不改“列名标记”模块内现有逐项提示文案
3. 不新增自动展开、自动定位或高亮缺失项的交互
4. 不调整 `custom_draw_plot` 的入参结构
5. 不对 `UPLOAD_FILE` 类型的 file mark 容器做必填分析校验

## 5. 现有结构分析

### 5.1 前端入口

[draw.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/draw.vue) 中：

- `handleStartOrStop()` 是“开始分析 / 取消分析”统一入口
- `handleSubmit()` 当前直接调用 `doSubmit()`
- `doSubmit()` 会执行 `drawReCreateParamSettings()`，随后发起 `crudPlot.customDrawPlot(params)`

现状中，这里没有针对“必选列名标记是否完成”的统一拦截。

### 5.2 前端现有列名标记校验

[fileSetting.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/fileMark/fileSetting.vue) 中已经存在：

- `beforeSave()`：校验当前文件的必选 `SELECT_COLUMN` 标记是否已选中
- `uploadClick()`：在部分上传场景中校验某个必选标记是否已完成

这说明前端已经认可“`required=true` 的 `SELECT_COLUMN` 必须先标记”的业务规则，但该规则尚未覆盖“开始分析”动作。

### 5.3 后端可复用数据结构

后端当前已经具备做该校验所需的完整数据：

- [ChartUploadFileSettings.java](/mnt/f/IdeaProjects/goto-software/goto/common/src/main/java/com/freedom/model/ChartUploadFileSettings.java)
  - `fileMarkContainerList`
  - `headerInfoList`
- [FileMarkContainer.java](/mnt/f/IdeaProjects/goto-software/goto/common/src/main/java/com/freedom/model/FileMarkContainer.java)
  - `type`
  - `field`
  - `positionParam`
  - `required`
- [HeaderInfo.java](/mnt/f/IdeaProjects/goto-software/goto/common/src/main/java/com/freedom/model/HeaderInfo.java)
  - `status`
  - `columnName`

[PlotTypeUtils.java](/mnt/f/IdeaProjects/goto-software/goto/common/src/main/java/com/freedom/util/PlotTypeUtils.java) 现有逻辑已经使用 `HeaderInfo.status` 配合 `positionParam` 的位信息来表达列名标记状态，因此本轮不需要引入新的状态字段。

## 6. 方案对比与决策

### 6.1 方案 A：仅前端拦截

优点：

- 改动小
- 用户反馈快

缺点：

- 可被绕过
- 后端接口契约仍不完整

### 6.2 方案 B：前后端各自独立校验，但不强调统一口径

优点：

- 两端都能阻止非法提交

缺点：

- 规则容易分叉
- 后续维护成本高

### 6.3 方案 C：前端快速拦截，后端按同口径最终兜底

优点：

- 用户体验最好
- 接口最安全
- 满足本轮完整目标

缺点：

- 需要把判断规则清晰固化到两端

### 6.4 最终决策

采用 **方案 C：前端快速拦截，后端同口径兜底**。

原因：

1. 前端可以在点击时立即阻止无效提交
2. 后端可以保证接口不会被绕过
3. 现有数据结构已经足够支持这套校验，无需额外扩展协议

## 7. 校验口径

### 7.1 校验范围

遍历当前 `userDraw.fileSettings`，逐个检查每个上传文件的：

- `fileMarkContainerList`
- `headerInfoList`

### 7.2 校验对象

仅检查满足以下条件的 file mark 容器：

- `type = SELECT_COLUMN`
- `required = true`

### 7.3 通过条件

对于每一个必选容器，当前文件的 `headerInfoList` 中必须至少存在一个列，其 `status` 在该容器对应的 `positionParam` 位上已被激活。

### 7.4 不拦截条件

以下情况不视为失败：

1. 当前文件没有 `fileMarkContainerList`
2. 当前文件的 `fileMarkContainerList` 中没有 `required = true` 的 `SELECT_COLUMN`

其中，“当前文件没有启用列名标记能力”在本设计中的可执行定义为：

- `fileMarkContainerList` 为空或不存在
- `fileMarkContainerList` 中没有任何 `SELECT_COLUMN` 项

也就是说，本轮不引入额外的独立启用开关，前后端都只基于 `fileMarkContainerList` 的实际内容判断是否需要进入列名标记校验。

### 7.5 失败结果

- 前端：阻止提交，提示 `请先标记列名`
- 后端：直接返回失败响应，提示 `请先标记列名`

## 8. 前端设计

### 8.1 作用范围

本轮前端改动仅触达：

- [draw.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/goto/plot/type/draw.vue)

### 8.2 落点

在“开始分析”提交流程前增加统一校验，建议落在：

- `handleSubmit()` 中调用统一校验后再进入 `doSubmit()`

或等价地落在 `doSubmit()` 最前面，但必须保证在以下动作之前完成拦截：

- `drawReCreateParamSettings()`
- `this.drawing = true`
- `crudPlot.customDrawPlot(params)`

### 8.3 建议实现方式

新增一个统一方法，例如：

- `validateRequiredFileMarks()`

该方法职责：

1. 遍历 `this.plot.userDraw.fileSettings`
2. 对每个 `fileSetting.fileMarkContainerList` 过滤出必选 `SELECT_COLUMN`
3. 检查 `fileSetting.headerInfoList` 是否存在满足位标记条件的列
4. 只要发现缺失，立即弹出 `请先标记列名` 并返回 `false`

### 8.4 前端失败行为

校验失败时：

- 不调用 `drawReCreateParamSettings()`
- 不设置 `drawing = true`
- 不发起 `customDrawPlot` 请求
- 只提示一次 `请先标记列名`

### 8.5 前端不改动项

明确保持不变：

1. `fileMark/fileSetting.vue` 中已有的保存校验逻辑
2. 现有 tab 切换逻辑
3. 面板展开收起状态
4. 结果页与下载逻辑

## 9. 后端设计

### 9.1 作用范围

本轮后端改动仅触达：

- [PlotServiceImpl.java](/mnt/f/IdeaProjects/goto-software/goto/system/src/main/java/com/freedom/service/impl/PlotServiceImpl.java)

如实现需要，可在服务类内增加小型私有方法；若已有公共位判断工具适合复用，可直接使用，但不建议为此新增大的公共抽象。

### 9.2 落点

在 `customDrawPlot(PlotType plotType)` 中：

1. 保留现有基础参数非空校验
2. 在 `UserDrawUtils.userDrawUpdate(componentBean, plotType)` 之前增加列名标记校验

这样可以保证：

- 非法请求不会更新任务状态
- 非法请求不会进入后续 Redis Stream 提交流程

### 9.3 建议实现方式

新增一个服务内私有校验方法，例如：

- `hasMissingRequiredFileMark(List<ChartUploadFileSettings> fileSettings)`

职责：

1. 遍历每个 `fileSetting`
2. 查找其中 `required=true` 的 `SELECT_COLUMN`
3. 检查该文件的 `headerInfoList` 是否有列在对应 `positionParam` 位上已标记
4. 发现任意缺失则返回 `true`

在 `customDrawPlot(...)` 中：

- 若校验失败，直接 `return ResponseUtils.failureMsg("请先标记列名")`

### 9.4 后端失败行为

校验失败时必须保证以下动作不发生：

1. 不执行 `UserDrawUtils.userDrawUpdate(...)`
2. 不执行参数重建后的任务提交流程
3. 不发送 Redis Stream 消息

## 10. 测试设计

### 10.1 前端测试场景

1. 存在必选列名标记且未标记，点击“开始分析”后不发请求，并提示 `请先标记列名`
2. 存在必选列名标记且都已标记，可以正常调用 `custom_draw_plot`
3. 存在列名标记模块但没有 `required=true` 项，不拦截
4. 某个上传文件没有 `fileMarkContainerList`，不拦截

### 10.2 后端测试场景

1. 请求中缺少必选列名标记，`customDrawPlot(...)` 返回失败消息 `请先标记列名`
2. 必选列名标记齐全，接口继续原有正常提交流程
3. 没有列名标记配置或没有必选项时，不影响现有正常流程
4. 存在必选 `SELECT_COLUMN`，但 `headerInfoList` 为空或缺失时，按“未完成标记”处理并返回 `请先标记列名`

### 10.3 回归验证重点

1. 现有“取消分析”流程不受影响
2. 非列名标记相关的参数校验不受影响
3. 文件上传、列名标记保存、结果查看与下载流程不受影响

## 11. 风险与规避

### 11.1 风险：前后端判断规则不一致

原因：

- 两端都需要按 `status + positionParam` 判断标记状态

规避：

- 设计阶段明确固定同一口径
- 实现时尽量让前后端都围绕相同字段组合判断

### 11.2 风险：误把非必选项当成阻塞条件

原因：

- `fileMarkContainerList` 中可能存在非必选项或非 `SELECT_COLUMN` 项

规避：

- 明确只检查 `required = true` 的 `SELECT_COLUMN`

### 11.3 风险：错误影响历史图表类型

原因：

- 历史配置可能没有 `fileMarkContainerList`，或者虽有列名标记但未标记为必选

规避：

- 对无配置或无必选项的情况一律放行

## 12. 实施摘要

建议后续实现按以下顺序推进：

1. 在前端 `draw.vue` 中增加统一提交前校验方法
2. 在后端 `PlotServiceImpl.customDrawPlot(...)` 中增加同口径兜底校验
3. 补充前后端对应测试或最小可验证用例
4. 回归验证“开始分析”“取消分析”“文件上传”“列名标记保存”链路

## 13. 结论

本轮设计通过在前后端同时补齐“必选列名标记完成度”校验，解决“未标记也能开始分析”和“接口可被绕过”两个问题。

最终效果是：

- 前端点击“开始分析”时能即时阻止无效提交
- 后端接口能作为最终兜底保证契约完整
- 提示文案统一为 `请先标记列名`
- 不影响未启用或未配置必选列名标记的历史图表
