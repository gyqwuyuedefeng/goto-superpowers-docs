# 文档集中迁移设计

## 背景

`goto-software` 工作区内历史上同时存在根目录级 `docs/plans`、根目录级 `docs/superpowers`，以及具体项目目录下的 `docs/superpowers`。这些目录分散保存了 superpowers 与 skill 生成的设计文档、实施计划和相关工作记录，后续检索和规范约束不够集中。

本次设计的目标是建立统一的中心文档目录 `goto-superpowers-docs`，并将主工作区内已确认范围的 `plans`、`specs` 文档迁移到该目录。后续所有 superpowers 与 skill 生成的 `specs`、`plans` 也应写入该中心目录。

## 迁移范围

本次迁移只处理主工作区中的文档源，不迁移临时工作区副本。

纳入迁移的源目录：

- `/mnt/f/IdeaProjects/goto-software/docs/plans`
- `/mnt/f/IdeaProjects/goto-software/docs/superpowers/plans`
- `/mnt/f/IdeaProjects/goto-software/docs/superpowers/specs`
- `/mnt/f/IdeaProjects/goto-software/goto-web/docs/superpowers/plans`
- `/mnt/f/IdeaProjects/goto-software/goto-web/docs/superpowers/specs`

明确排除的目录和文件：

- `/mnt/f/IdeaProjects/goto-software/.worktrees/*`
- 任意 `.git/*`
- 任意 `node_modules/*`
- 任意 `docs/superpowers/test-results/*`
- 任意非 `plans`、非 `specs` 的 `docs/superpowers` 子目录

## 目标目录结构

全局文档使用 `_global` 作为隔离目录；单个项目的文档使用项目目录名作为隔离目录。

目标结构如下：

```text
/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs/
├── _global/
│   └── superpowers/
│       ├── plans/
│       └── specs/
└── goto-web/
    └── superpowers/
        ├── plans/
        └── specs/
```

## 路径映射规则

根目录级文档视为全局文档，统一迁移到 `_global`：

- `/mnt/f/IdeaProjects/goto-software/docs/plans/*`
  → `/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs/_global/superpowers/plans/*`
- `/mnt/f/IdeaProjects/goto-software/docs/superpowers/plans/*`
  → `/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs/_global/superpowers/plans/*`
- `/mnt/f/IdeaProjects/goto-software/docs/superpowers/specs/*`
  → `/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs/_global/superpowers/specs/*`

项目级文档使用项目名称隔离：

- `/mnt/f/IdeaProjects/goto-software/<项目名>/docs/superpowers/plans/*`
  → `/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs/<项目名>/superpowers/plans/*`
- `/mnt/f/IdeaProjects/goto-software/<项目名>/docs/superpowers/specs/*`
  → `/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs/<项目名>/superpowers/specs/*`

本次已发现的项目级迁移来源只有 `goto-web`。

## 冲突处理

迁移前先检查目标路径是否存在同名文件。若目标路径不存在同名文件，直接移动。

若发生同名文件冲突，不覆盖已有文件。冲突文件保留两份，并在迁入文件名末尾追加来源标识，例如：

```text
2026-03-09-rabbitmq-to-redis-stream-design-from-docs-plans.md
```

该策略用于避免静默覆盖历史文档，同时让后续人工审查可以根据来源判断是否合并。

## 项目规范更新

需要更新项目级规范文件：

```text
/mnt/g/Obsidian/code/01_Project_Specs/goto_software_spec.md
```

新增规则应说明：

- 后续所有 superpowers 与 skill 创建的 `specs`、`plans` 统一写入 `/mnt/f/IdeaProjects/goto-software/goto-superpowers-docs`。
- 全局文档写入 `_global/superpowers/specs` 和 `_global/superpowers/plans`。
- 单个项目文档写入 `<项目名称>/superpowers/specs` 和 `<项目名称>/superpowers/plans`。
- 禁止继续写入旧位置 `docs/plans`、`docs/superpowers`、`*/docs/superpowers`。
- `.worktrees/*` 下的历史文档不作为本次迁移对象。

## 验收标准

迁移完成后应满足以下条件：

- 除 `.worktrees`、`.git`、`node_modules` 和 `test-results` 外，主工作区不再存在待迁移的 `docs/plans`、`docs/superpowers/plans`、`docs/superpowers/specs` 文件。
- 目标目录中包含预期迁移文件：全局文档 102 个，`goto-web` 文档 10 个，共 112 个文件。
- 若迁移过程中发生同名冲突，冲突文件均被保留，并带有清晰来源后缀。
- `goto_software_spec.md` 已记录新的 specs/plans 写入规则。
- `git status --short` 中与本任务相关的变更只包含中心文档目录新增、旧文档删除和项目级规范更新，不包含无关源码修改。

## 实施边界

本设计只定义文档迁移和规范更新，不修改业务代码、不执行 Maven、不访问数据库、不处理 `.worktrees` 中的分支副本文档。
