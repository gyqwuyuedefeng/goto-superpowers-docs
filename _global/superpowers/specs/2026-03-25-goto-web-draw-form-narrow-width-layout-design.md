# goto-web draw 表单窄宽度布局设计

## 1. 背景

`goto-web` 的 `draw` 表单当前依赖自动网格和多套字段组件共同完成布局。在正常宽度下，大多数配置项显示正常；但当容器变窄时，部分 `label + select/input` 组合会进入危险宽度，出现以下问题：

1. 字段标签和输入控件被拆裂
2. 复合字段中的局部标签漂移，例如 `y` 留在上一行
3. 某些组件看起来像多出一个输入框，实际是容器被挤裂后的视觉错位

用户首先明确指出了图例设置中的问题：

- `图例位置`
- `内部相对坐标`
- `排列方式`

其中“排列方式”在窄容器下会与相邻复合字段发生错位，视觉结果与正常宽度下的预期布局明显不一致。

这不是单一页面补丁问题，而是 `draw` 表单同类字段组件缺少统一的窄宽度契约：外层网格决定了列数，但字段内部没有稳定的“整组换行、组内不拆”规则。

## 2. 已确认约束

本次设计基于以下已确认约束：

1. 修复范围不只限于图例设置，需要顺带统一 `draw` 表单里同类 `label + select/input` 组合的窄宽度行为
2. 窄宽度下采用“按组换行”而不是强行保持所有字段同排
3. `label` 始终单行显示，超出时省略，不允许为了适配窄宽度而改成多行
4. 复合字段组内部尽量保持一行，例如 `x + input`、`y + input` 组成的坐标输入，不应在组内自行拆裂
5. 本轮只处理布局与样式契约，不修改业务逻辑、字段联动或参数可见性

## 3. 目标

### 3.1 视觉目标

- 窄容器下，`draw` 表单中的同类字段表现一致
- 字段标签与对应控件始终保持绑定关系，不出现拆裂
- 复合字段在窄宽度下仍保持可读，不出现孤立 `x/y` 标签或错位输入框

### 3.2 工程目标

- 将窄宽度行为沉淀到 `draw` 表单的共享布局层和字段组件层
- 避免只在 `legendSettingsParam.vue` 里做孤立补丁
- 为后续同类字段提供统一行为基线，降低继续出现窄宽度回归的概率

### 3.3 维护目标

- 将“整组换行、组内不拆”的规则变成可复用约定，而不是页面特例
- 让后续新增 `draw` 字段组件可以直接复用既有布局规则

## 4. 非目标

本轮不做以下事情：

1. 不重构 `draw` 表单的业务结构或参数组织方式
2. 不统一所有控件实现为同一个组件
3. 不覆盖 `radio`、`checkbox`、`slider`、`color picker` 等结构明显不同的控件
4. 不扩展到非 `draw` 目录下的后台表单
5. 不修改字段的显示条件、接口字段或交互流程

## 5. 现状与根因

当前布局由三层共同构成：

1. 外层网格：`src/assets/styles/draw/_grid.scss` 中的 `.draw-group-attribute-line`
2. 基础表单结构：`src/assets/styles/draw/_form-base.scss` 与 `src/assets/styles/draw/_modern-lab-theme.scss`
3. 字段组件模板：`src/views/goto/components/form/draw/` 下的各类 `select/input` 组件

当前根因不是某一个组件单独写错，而是三层之间缺少统一契约：

- 外层网格会继续压缩列宽，直到字段进入不可读宽度
- 普通字段组件和复合字段组件对 `label`、`form-content`、内部 `container` 的收缩策略不一致
- 某些组件只声明了“宽度 100%”，但没有声明“字段组不可拆”

结果就是：网格仍然试图维持更多列，字段内部又没有稳定的不可拆约束，最终导致窄宽度时错位。

## 6. 方案决策

### 6.1 采用“共享窄宽度契约”方案

本轮采用的不是单点补丁，也不是只调外层网格阈值，而是建立 `draw` 字段的共享窄宽度契约：

1. 外层网格只负责“字段组之间如何换行”
2. 基础表单层只负责“字段组自身不可拆”
3. 复合字段组件只负责“组内如何保持稳定”

原因：

- 只修图例设置会继续留下其他同类字段的回归风险
- 只调外层网格无法保证字段内部不拆裂
- 共享契约最符合当前仓库对 `draw` 样式问题的处理方式：修到公共层，避免页面特例

### 6.2 采用“整组换行”而不是“极限压缩”

窄宽度下不强求继续保持三列同排，而是让字段组在进入危险宽度之前整组换到下一行。

原因：

- 这是用户明确选择的退化方式
- 这种行为最稳，不会出现标签和控件分离
- 相比横向滚动或极限压缩，更符合当前 `draw` 面板的阅读方式

### 6.3 复合字段遵循“组内不拆，组外可换”

像 `内部相对坐标` 这种字段遵循两层规则：

1. 整个字段组相对外部其它字段可以换行
2. 字段组内部的 `x/y` 小项尽量保持在一行

原因：

- 这是用户明确确认的行为
- 复合字段的局部标签一旦独立换行，视觉破坏比普通字段更明显

## 7. 责任分层

### 7.1 网格层

文件：

- `goto-web/src/assets/styles/draw/_grid.scss`

职责：

- 控制单个字段组占几列
- 决定字段组在什么宽度下整组换行
- 为复合字段预留更合理的最小宽度或更大的列跨度

不负责：

- 字段内部 label 是否换行
- 输入框或下拉框内部如何收缩

### 7.2 表单基础层

文件：

- `goto-web/src/assets/styles/draw/_form-base.scss`
- `goto-web/src/assets/styles/draw/_modern-lab-theme.scss`

职责：

- 为 `form-group / form-label / form-label-wrapper / form-content / form-content-item` 定义共享窄宽度行为
- 保证 label 单行、省略号、不可拆裂
- 保证 `form-content` 在可收缩时不会把字段拆成两半

不负责：

- 某个具体业务字段是否跨整行
- 某个复合控件内部 `x/y` 结构如何排列

### 7.3 字段组件层

优先纳入本轮统一的组件：

- `goto-web/src/views/goto/components/form/draw/input.vue`
- `goto-web/src/views/goto/components/form/draw/selectObject.vue`
- `goto-web/src/views/goto/components/form/draw/selectMultipleObject.vue`
- `goto-web/src/views/goto/components/form/draw/selectSimple.vue`
- `goto-web/src/views/goto/components/form/draw/selectDictImage.vue`
- `goto-web/src/views/goto/components/form/draw/coordinatePointSingleParam.vue`

职责：

- 接入共享字段组契约
- 对复合字段补充组内稳定性约束
- 避免各组件继续用不一致的 label 模版和收缩规则

## 8. 统一行为规则

### 8.1 普通字段组

适用于 `label + select/input` 类字段：

1. `label` 永远单行
2. `label` 超出时显示省略号
3. `label` 和控件视为同一个字段组
4. 当可用宽度不足时，整个字段组换到下一行
5. 不允许出现 label 仍在上一行、控件掉到下一行的拆裂状态

### 8.2 复合字段组

适用于坐标类或类似复合字段：

1. 整个字段组和普通字段一样，按组参与外层换行
2. 组内 `x/y` 小项尽量保持一行
3. 组内小项可以弹性收缩，但不能彼此拆裂
4. 不允许出现孤立 `x/y` 标签、被挤到上一行的局部标签或视觉上像重复控件的错位

### 8.3 共享样式契约

本轮要求共享层明确建立如下约束：

1. 字段组默认 `min-width: 0`，但不能只靠 `width: 100%` 解决问题
2. `form-content` 需要允许内部控件收缩，同时保持字段组整体稳定
3. 横向布局字段在窄宽度下允许整组掉行，不再坚持极限压缩
4. 核心布局规则不能依赖 `:has()`，应以稳定类名为主

## 9. 覆盖范围

### 9.1 必须验证的入口

实现后至少验证以下场景：

1. `legendSettingsParam.vue` 中的 `图例位置 / 内部相对坐标 / 排列方式`
2. 普通单选下拉字段
3. 普通数字输入字段
4. 多选下拉字段
5. 图片选择字段
6. 复合坐标字段

### 9.2 明确不纳入的入口

本轮不要求统一以下结构：

1. `radio`
2. `checkbox`
3. `slider`
4. `color picker`
5. 非 `draw` 目录下后台表单

## 10. 风险与控制

### 10.1 可能风险

1. 某些现有字段可能会比以前更早换行
2. 横向布局字段组件可能受共享样式影响
3. 多选和图片选择这类特殊下拉在接入共享规则后，可能出现局部尺寸回归

### 10.2 控制策略

1. 先在共享层定义最小必要契约，不引入大范围视觉重构
2. 对复合字段单独补规则，不把特例压到全局所有字段
3. 验证时覆盖普通字段和特殊字段两类代表入口，避免只修图例设置

## 11. 验收标准

本轮完成后，需满足以下标准：

1. 在窄容器下，“排列方式”不再出现标签与下拉框拆裂
2. “内部相对坐标”中的 `x/y` 不出现错位、漂移或孤立标签
3. `draw` 表单里同类 `label + select/input` 组合在窄宽度下表现一致
4. 所有相关字段的 label 始终单行省略，不因修复变成多行
5. 正常桌面宽度下的原有布局不出现明显回归

## 12. 实施边界

本设计只为后续实施计划提供明确边界，不在本阶段直接修改业务代码。后续实施计划应围绕以下顺序展开：

1. 先收口共享网格与表单基础层
2. 再接入普通字段组件
3. 最后处理复合字段和特殊 select 组件的例外验证
