# 用户绘图类型每日统计设计

## 背景

前端绘图页的“开始分析”最终会调用后端 `PlotServiceImpl.customDrawPlot`，该方法在参数校验、列名标记校验和防抖限流通过后，把 `UserDrawStreamMessage` 提交到 Redis Stream。用户绘图记录 `app_user_draw` 会定期删除，不能作为长期统计依据。

本次需要在后端记录“用户成功提交过哪些类型的图、提交了多少次”，允许异步统计存在少量误差。后续管理员页面需要按最近 7 天、最近一个月、最近一年、全部时间等时间范围查看最受欢迎的图类型。

## 范围

本次实现后端统计基础能力，不实现管理员前端统计页面。

本次包括：

- 新增一张数据库每日聚合统计表。
- 在绘图任务成功提交 Redis Stream 后异步记录统计。
- 预留后端热门图类型排行查询接口，供后续管理员页面使用。
- 统计逻辑不依赖 `app_user_draw` 的长期保留。

本次不包括：

- 前端管理员统计页面。
- 每次提交一条明细日志。
- Redis 计数后定时落库。

## 统计口径

只有当 `PlotServiceImpl.customDrawPlot` 成功执行 `sendToRedisStream(userDrawId, userId)` 后才计数一次。

以下情况不计数：

- 前端只点击按钮但请求未到达后端。
- 后端参数异常。
- 缺少必选列名标记。
- 防抖或限流拦截。
- Redis Stream 提交失败。
- 用户取消分析。

异步统计写入失败不影响绘图任务提交结果，只记录日志。

## 数据表设计

新增表建议命名为 `app_user_plot_type_daily_stat`。

字段：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `stat_date` | date | 统计日期，按服务器本地日期记录 |
| `user_id` | bigint | 用户 ID |
| `plot_type_id` | bigint | 图类型 ID |
| `count` | bigint | 当天该用户成功提交该图类型的次数 |
| `create_time` | datetime | 创建时间 |
| `update_time` | datetime | 更新时间 |

约束和索引：

- 唯一约束：`uk_stat_date_user_plot_type(stat_date, user_id, plot_type_id)`。
- 查询索引：`idx_stat_date_plot_type(stat_date, plot_type_id)`，用于按时间范围聚合热门图类型。
- 可选索引：`idx_user_plot_type(user_id, plot_type_id)`，用于后续按用户查看个人绘图偏好。

每日聚合表相比明细日志更适合当前需求：支持时间范围查询，数据量可控，也不受 `app_user_draw` 定期删除影响。

## 后端组件

新增持久化组件：

- `UserPlotTypeDailyStat`：JPA 实体，对应新增统计表。
- `UserPlotTypeDailyStatRepository`：提供按唯一键自增、按时间范围聚合排行的查询。

新增服务：

- `UserPlotTypeStatService`：对外暴露统计记录和排行查询能力。
- `AsyncUserPlotTypeStatService`：使用 `@Async` 异步执行统计自增。

新增线程池：

- 在 `AsyncConfig` 中增加轻量级 `userPlotTypeStatTaskExecutor`。
- 线程池规模保持较小，例如核心线程 1、最大线程 2、队列 1000。
- 拒绝策略建议记录警告并丢弃或使用已有工厂支持的策略；统计允许少量误差，不能反向阻塞绘图提交。

## 写入流程

`PlotServiceImpl.customDrawPlot` 保持现有主流程：

1. 校验 `plotType`、`userDraw`、`paramSettings`。
2. 校验必选列名标记。
3. 更新 `UserDraw` 参数。
4. 在 Redisson 锁内调用 `sendToRedisStream(userDrawId, currentUserId)`。
5. Redis Stream 提交成功后，调用异步统计服务：
   - `statDate = LocalDate.now()`。
   - `userId = SecurityUtils.getCurrentUserId()` 或当前方法中已取得的用户 ID。
   - `plotTypeId = plotType.getId()`。
6. 异步服务对 `app_user_plot_type_daily_stat` 执行当天计数 `+1`。

为避免并发提交导致重复插入，统计自增应优先使用数据库原子 upsert：

- MySQL 可使用 `INSERT ... ON DUPLICATE KEY UPDATE count = count + 1, update_time = NOW()`。
- 如果项目数据库方言不支持 upsert，再退回为“先更新，更新行数为 0 时插入，插入唯一冲突后再更新一次”。

## 排行查询接口预留

后续管理员页面需要热门图类型排行。本次预留后端接口，例如：

```text
POST /api/userPlotTypeStat/popular_plot_types
```

请求参数：

```json
{
  "rangeType": "LAST_7_DAYS",
  "limit": 20
}
```

支持范围：

- `LAST_7_DAYS`：最近 7 天，包含今天。
- `LAST_30_DAYS`：最近 30 天，用于“最近一个月”。
- `LAST_365_DAYS`：最近 365 天，用于“最近一年”。
- `ALL`：全部时间。

返回字段建议：

| 字段 | 说明 |
| --- | --- |
| `plotTypeId` | 图类型 ID |
| `plotTypeName` | 图类型名称或文件夹名，按现有可展示字段取值 |
| `groupId` | 图分组 ID |
| `groupName` | 图分组名称 |
| `count` | 时间范围内总提交次数 |

权限建议：

- 查询接口只面向管理员或具备统计查看权限的用户。
- 权限标识后续实现时沿用项目现有 `@PreAuthorize("@el.check('...')")` 规则。

## 错误处理

- 主绘图提交成功后，即使统计异步任务提交或执行失败，也不改变 `customDrawPlot` 的返回。
- 统计服务记录失败日志，日志包含 `statDate`、`userId`、`plotTypeId`。
- 如果 `plotTypeId` 或 `userId` 为空，统计服务直接跳过并记录警告。

## 测试设计

后端单元测试覆盖：

- Redis Stream 提交成功后会触发统计记录。
- Redis Stream 提交失败时不会触发统计记录。
- 参数校验失败或必选列名标记缺失时不会触发统计记录。
- 统计服务对同一天同一用户同一图类型会累加次数。
- 排行查询能按最近 7 天、30 天、365 天和全部时间正确汇总。

数据库层测试或集成测试覆盖：

- 唯一约束避免并发插入重复行。
- upsert 自增 SQL 能正确更新 `count` 和 `update_time`。

## 验收标准

- 新表可独立保存长期统计数据，不依赖 `app_user_draw` 是否被删除。
- 成功提交 Redis Stream 后，数据库中当天对应 `stat_date + user_id + plot_type_id` 的计数增加。
- 统计写入失败不影响绘图任务提交接口返回。
- 后端具备按时间范围汇总热门图类型的查询能力，后续前端管理员页面可以直接接入。
