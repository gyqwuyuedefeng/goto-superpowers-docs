# 多智能体文档审核系统设计

## 背景

当前 `deepseek-review-spec` 和 `deepseek-review-plan` 采用单一 DeepSeek 审核者模式，从固定的五维度（完整性/一致性/清晰度/范围/YAGNI）审核一份文档。用户需要更精准、更全面的审核覆盖：多个不同性格的审核智能体从各自维度独立审核，由 AI aggregator 汇总后交给 controller。

## 设计目标

- 将单一审核者替换为多 reviewer 并行 + aggregator 汇总模式。
- spec review 和 plan review 使用不同的 reviewer 集合。
- 每个 reviewer 有明确的角色定义、审核视角和输出格式。
- aggregator 负责去重、合并、冲突检测，输出结构化汇总报告。
- 保留原有单 reviewer 模式作为向后兼容选项。

## Reviewer 体系

所有 reviewer prompt 遵循统一结构：`common-reviewer-rules.md`（通用规则/严重度/输出格式）+ 角色专属 prompt（角色名、核心问题、关注领域、红线、禁止事项、类别值）。

### Spec Reviewers（6 个）

| Reviewer | 核心问题 | 性格 |
|----------|---------|------|
| `spec-requirement-alignment` | spec 是否准确表达用户真实开发需求？ | 产品经理 + 需求分析师 |
| `spec-context-completeness` | spec 是否给了足够的上下文，避免实现者靠猜？ | 代码库考古者 |
| `spec-scope-boundary` | 边界是否清晰？做什么、不做什么？ | 合同律师 + 边界洁癖 |
| `spec-acceptance-criteria` | spec 是否定义了可验证的"完成"标准？ | 验收负责人 |
| `spec-risk-edge-case` | 是否考虑了异常、边界、失败场景？ | 悲观系统设计师 |
| `spec-yagni-scope-control` | 是否引入了不必要的需求膨胀？ | 极简主义产品 owner |

### Plan Reviewers（7 个）

| Reviewer | 核心问题 | 性格 |
|----------|---------|------|
| `plan-spec-faithfulness` | plan 是否忠实实现 approved spec？ | 严格需求守门人 |
| `plan-context-grounding` | plan 是否真正理解了现有代码库？ | 资深项目维护者 |
| `plan-architecture-elegance` | plan 是否让代码结构更清晰、更易维护？ | 架构洁癖 + 高级工程师 |
| `plan-robustness-bug-hunter` | plan 是否处理了失败、超时、并发冲突？ | 悲观测试工程师 |
| `plan-testability` | plan 的测试方案是否真实可执行？ | 测试负责人 + CI 守门人 |
| `plan-integration-risk` | plan 是否能顺利接进现有系统？ | 发布负责人 |
| `plan-yagni-complexity` | plan 是否过度工程化？ | 极简高级工程师 |

### Aggregator

独立调用，输入为所有 reviewer 报告，输出结构化汇总报告：
- 合并去重
- 严重度归一
- 冲突检测
- 生成 controller 行动清单

## 文件组织

```text
skill/superpowers/reviewers/
  shared/
    common-reviewer-rules.md    # 所有 reviewer 共享的通用规则
  spec/
    requirement-alignment.md

    scope-boundary.md
    acceptance-criteria.md
    risk-edge-case.md
    yagni-scope-control.md

  shared/

```

## 脚本改造

### deepseek-review-spec

- 新增 `--multi` flag，启用多 reviewer 模式。
- 新增 `--reviewers-dir <dir>`，默认 `skill/superpowers/reviewers`。
- `--multi` 时并行派发 6 个 spec reviewer → 聚合报告。
- 不传 `--multi` 时保持原有单 reviewer 行为（向后兼容）。

### deepseek-review-plan

- 同上，但使用 7 个 plan reviewer。
- 额外将 approved spec 作为上下文提供给 reviewer。

## 非目标

- 不创建数据库、调度器、插件系统或 dashboards。
- 不改变 `codex-dp-exec` 的核心逻辑。
- 不删除旧的单一 reviewer prompt 文件。
- 不做 reviewer 的动态注册或热加载（MVP 阶段硬编码列表即可）。

## 验收标准

- `deepseek-review-spec --multi` 可并行运行 6 个 reviewer 并输出 aggregator 汇总报告。
- `deepseek-review-plan --multi` 可并行运行 7 个 reviewer 并输出 aggregator 汇总报告。
- `--single` 模式（默认）行为与改造前一致。
- 任何 reviewer 失败不阻塞其他 reviewer 和 aggregator。
- aggregator 报告包含去重后的结构化 issue 列表。
- 所有脚本通过 `bash -n` 语法检查。
