# goto-web 韦恩图增量保存误报设计

## 1. 背景

生产环境中，绘图页的韦恩图分类（含 `韦恩`、`UpSet`、`花瓣`）在编辑上传表格后执行增量保存时，会稳定出现以下现象：

1. 先显示绿色提示 `保存成功`
2. 随后立即弹出 `增量保存失败，是否尝试全量保存？`

本地测试接口 `POST /api/plotType/save_file_setting_incremental_handsontable` 正常，且其他图类型未出现相同行为。因此本问题不是通用增量保存能力整体失效，而是韦恩图专属保存后处理流程存在异常。

## 2. 已确认事实

### 2.1 用户现象

1. 线上问题稳定复现，且每次都是“绿色成功提示先出现，失败确认框后出现”
2. 只有韦恩图这一组图类型出现问题，其他图类型正常
3. 当前浏览器不允许直接打开开发者工具，因此不能依赖前端现场控制台先做排查

### 2.2 代码定位结论

排查后确认，失败确认框并不是 WebSocket 推送出来的，而是前端 `handsontable.vue` 在增量保存 Promise 的 `.catch()` 中主动弹出：

- 成功提示位置：
  - [goto-web/src/views/components/handsontable.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/handsontable.vue:901)
- 全量保存确认框位置：
  - [goto-web/src/views/components/handsontable.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/handsontable.vue:941)

后端接口本身在保存成功后返回 `ResponseUtils.successMsg("保存成功")`：

- [goto/system/src/main/java/com/freedom/service/impl/PlotTypeServiceImpl.java](/mnt/f/IdeaProjects/goto-software/goto/system/src/main/java/com/freedom/service/impl/PlotTypeServiceImpl.java:469)

这说明当前“保存成功后又弹失败”并不符合接口语义，更像是前端成功回调中的后续逻辑抛出了异常。

## 3. 根因分析

### 3.1 通用成功分支与韦恩图专属分支

在增量保存成功后，`handsontable.vue` 会依次执行：

1. 弹出 `保存成功`
2. 清空变更记录
3. 更新本地数据状态
4. 调用 `this.saveFileSuccess()`

其中 `saveFileSuccess()` 会按 `groupId` 路由到图类型专属回调：

- [goto-web/src/constant/associationParamModify.js](/mnt/f/IdeaProjects/goto-software/goto-web/src/constant/associationParamModify.js:105)
- [goto-web/src/constant/modify/publicFunction.js](/mnt/f/IdeaProjects/goto-software/goto-web/src/constant/modify/publicFunction.js:171)

韦恩图对应 `GROUP_ID_VENN = 6`，会命中：

- [goto-web/src/constant/modify/vennModify.js](/mnt/f/IdeaProjects/goto-software/goto-web/src/constant/modify/vennModify.js:354)

### 3.2 韋恩图专属回调的脆弱点

`vennSaveFileSuccess()` 会继续调用：

1. `valueModify(commonParams)`
2. `plotTypeModify(commonParams)`

其中 `plotTypeModify(commonParams)` 会查找两组可选配置：

1. `视觉属性（Upset韦恩图）`
2. `视觉属性（花瓣图）`

然后直接在 `else` 分支中写入：

1. `flowerGroup.visible = false`
2. `upsetGroup.visible = false`

对应代码：

- [goto-web/src/constant/modify/vennModify.js](/mnt/f/IdeaProjects/goto-software/goto-web/src/constant/modify/vennModify.js:122)

如果生产环境某些韦恩图模板或参数配置缺少这两个分组，则 `flowerGroup` 或 `upsetGroup` 会是 `null`，从而触发运行时异常。

### 3.3 为什么会出现“先成功后失败”

当前前端把“接口成功”与“保存后 UI 联动”放在同一个 Promise 成功链里。

因此会出现以下顺序：

1. 接口返回 `code === 200`
2. 前端先显示 `保存成功`
3. `this.saveFileSuccess()` 内部抛异常
4. 当前 Promise 链被打入外层 `.catch()`
5. 前端误以为“增量保存失败”，弹出“是否尝试全量保存”

因此本问题本质上是：

- 实际保存成功
- 保存后的前端联动逻辑异常
- 前端把联动异常误报成了保存失败

## 4. 目标

### 4.1 功能目标

1. 当增量保存接口成功时，前端必须稳定认定为保存成功
2. 韋恩图缺少可选视觉分组时，不应再触发失败确认框
3. 其他图类型原有保存行为保持不变

### 4.2 健壮性目标

1. 保存成功后的 UI 联动异常不能再伪装成接口失败
2. 前端需要容忍生产模板配置不完整的情况

### 4.3 排查辅助目标

在不影响用户保存结果的前提下，为后续核对生产模板是否缺少韦恩图视觉分组提供足够的调试信息。

## 5. 非目标

本轮不处理以下内容：

1. 不重构增量保存协议
2. 不修改后端 `save_file_setting_incremental_handsontable` 的成功/失败判定逻辑
3. 不调整 WebSocket 通知链路
4. 不强制要求本轮同时补齐生产数据库中的所有韦恩图模板分组
5. 不改其他图类型的参数联动逻辑

## 6. 方案对比

### 6.1 方案 A：仅给韦恩图分组访问加判空

做法：

1. 在 `vennModify.js` 中对 `flowerGroup`、`upsetGroup` 判空
2. 分组不存在时直接跳过可见性设置

优点：

1. 改动最小
2. 能快速止住当前韦恩图误报

缺点：

1. 如果未来别的保存后回调也抛异常，仍会被误判成“增量保存失败”
2. 没有从结构上纠正错误的异常归因

### 6.2 方案 B：只在 `handsontable.vue` 中隔离保存后回调异常

做法：

1. 保持 `vennModify.js` 原样
2. 只在 `res.code === 200` 后，将 `this.saveFileSuccess()` 包入单独 `try/catch`

优点：

1. 可以确保接口成功不再被误判成保存失败
2. 对所有图类型都增强了成功链路的韧性

缺点：

1. 韋恩图专属回调仍会持续抛异常
2. 每次保存都会产生错误日志，且 UI 联动仍然不完整

### 6.3 方案 C：韦恩图判空止血 + 成功链路隔离异常

做法：

1. 在 `vennModify.js` 中为可选分组访问补齐判空保护
2. 在 `handsontable.vue` 中把“接口成功”与“保存后联动”拆开处理
3. 联动回调异常仅记录，不再触发全量保存降级

优点：

1. 直接覆盖当前主因
2. 同时修正错误的异常语义
3. 即使未来再出现类似回调异常，也不会再误导用户

缺点：

1. 改动略多于方案 A
2. 需要补一组更完整的回归验证

### 6.4 最终决策

采用 **方案 C：韦恩图判空止血 + 成功链路隔离异常**。

原因：

1. 当前问题已经明确同时包含“韦恩图配置脆弱”和“前端错误归因”两层问题
2. 只修其中一层都不彻底
3. 方案 C 仍然属于小范围低风险修改，但能同时消除误报与未来同类问题

## 7. 详细设计

### 7.1 `vennModify.js` 的修复设计

文件：

- [goto-web/src/constant/modify/vennModify.js](/mnt/f/IdeaProjects/goto-software/goto-web/src/constant/modify/vennModify.js)

设计原则：

1. `视觉属性（花瓣图）` 与 `视觉属性（Upset韦恩图）` 属于可选分组
2. 若分组存在，则按当前 `plotType` 控制其显隐
3. 若分组不存在，则静默跳过，不抛异常

预期修改点：

1. `flowerGroup.visible = ...` 仅在 `flowerGroup` 存在时执行
2. `upsetGroup.visible = ...` 仅在 `upsetGroup` 存在时执行

这样即使生产配置缺组，韋恩图保存成功后也不会因为访问空对象而失败。

### 7.2 `handsontable.vue` 的修复设计

文件：

- [goto-web/src/views/components/handsontable.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/handsontable.vue)

设计原则：

1. `res.code === 200` 只代表接口保存结果
2. `saveFileSuccess()` 只是保存后的前端联动回调，不应影响保存成功结论

预期执行顺序调整为：

1. 接口返回 `code === 200`
2. 显示 `保存成功`
3. 清空变更记录
4. 更新本地数据状态
5. 单独执行 `this.saveFileSuccess()`，并为其包裹内部 `try/catch`

若 `saveFileSuccess()` 发生异常：

1. 不进入当前外层 Promise 的 `.catch()`
2. 不再弹出 `增量保存失败，是否尝试全量保存？`
3. 记录带上下文的错误日志
4. 可选地给出轻量提示，例如“保存成功，部分界面联动未完成，请刷新页面”

本轮建议默认只记录错误日志，不新增新的用户提示，以避免打断保存主流程。

### 7.3 错误语义规范

修复后，前端错误处理语义统一如下：

1. **接口失败**
   - 条件：接口返回非 `200`，或 HTTP 请求本身异常
   - 行为：允许显示失败提示，允许弹出“是否尝试全量保存”

2. **保存后联动失败**
   - 条件：接口返回 `200`，但 `saveFileSuccess()` 等后处理逻辑异常
   - 行为：保存仍视为成功，只记日志，不触发全量保存降级

3. **模板配置缺失**
   - 条件：某些韦恩图参数分组缺失
   - 行为：静默跳过对应显隐联动

## 9. 验证设计

### 9.1 必测场景

1. 普通图类型增量保存成功
   - 预期：只显示成功提示，不出现失败确认框

2. 韋恩图在分组完整时增量保存成功
   - 预期：只显示成功提示，相关参数联动正常

3. 韋恩图在分组缺失时增量保存成功
   - 预期：只显示成功提示，不出现失败确认框

4. 真正的接口失败场景
   - 预期：仍显示失败提示，并允许用户尝试全量保存

5. 全量保存流程
   - 预期：行为与修复前一致

### 9.2 建议补充验证

1. 保存成功后再次编辑并重复增量保存，确认变更历史清理逻辑未受影响
2. 韋恩图切换 `plotType` 为 `flower`、`Upset`、其他值时，相关视觉分组显隐仍符合预期

## 10. 风险与回滚

### 10.1 风险

1. 若 `saveFileSuccess()` 内部还有其他隐含异常，修复后虽然不再误报为保存失败，但需要通过日志继续观察
2. 若某些联动逻辑依赖异常中断来触发特定效果，可能暴露出新的 UI 状态不一致问题

### 10.2 风险控制

1. 将本轮修改严格限制在韋恩图分组判空与成功链路异常隔离
2. 不改动保存接口协议与后端持久化逻辑
3. 保留错误日志，便于后续继续追踪真实回调异常

### 10.3 回滚策略

如果上线后出现新的联动回归，可先回滚前端本次变更；由于后端未变更，回滚成本较低。

## 11. 实施建议

建议先完成前端修复并上线验证，再决定是否补做生产模板数据同步。

推荐顺序：

1. 实现 `vennModify.js` 判空保护
2. 实现 `handsontable.vue` 成功链路异常隔离
3. 运行前端相关验证
4. 线上复测韋恩图增量保存
5. 并行核查生产韋恩图模板配置

## 12. 结论

本问题的根因不是接口保存失败，而是韋恩图保存成功后的前端专属联动逻辑对生产配置缺失不够容错，导致成功链路内部抛异常，并被错误地展示为“增量保存失败”。

本轮通过“韋恩图分组判空保护 + 保存成功链路异常隔离”即可在小范围改动下解决线上误报，并为后续核查生产模板差异保留空间。
