# UserDraw 3 天未使用直接删除设计

## 背景

当前代码中，`UserDraw` 的清理链路分散在以下旧任务中：

- `LongTimeUserDrawMarkAndNotice`
- `UserDrawTmpFolderRemoveFail`
- `DelayedRemoveUserDrawTask`

现有方案包含“标记删除、邮件通知、延迟队列、补偿清理”等多段流程。当前目标是新增一条更直接的清理方案，供后续替换旧链路使用：

- 仅按 `update_time < NOW() - INTERVAL 3 DAY` 判断是否过期
- 直接删除数据库中的 `UserDraw` 记录
- 同时删除对应临时目录
- 不再依赖 `del_mark`、`email_notice`、延迟队列
- 不在设计中加入 `@Scheduled` 配置，任务由外部调用方触发

## 目标

新增一条独立清理任务，在每次触发时扫描所有 3 天未使用的 `UserDraw`，并对每条数据执行：

1. 删除对应临时目录
2. 删除数据库记录

约束如下：

- 仅按 `update_time` 判断是否过期，口径与旧逻辑保持一致
- 目录不存在时，视为目录删除成功，可以继续删除数据库记录
- 目录删除失败时，数据库不得删除
- 数据库删除失败时，该记录保留，等待下次任务重试
- 失败时必须输出日志，便于排查

## 非目标

本设计不包含以下内容：

- 不保留旧的“标记删除/通知/延迟删除”业务流程
- 不设计邮件通知逻辑
- 不设计延迟队列恢复逻辑
- 不在本次设计中处理旧任务删除的实施细节
- 不加入新的页面、接口或配置项

## 方案决策

采用“任务类 + 删除执行器”的结构。

原因：

- 任务类只负责“扫描哪些数据要删”和“整次任务汇总”
- 删除执行器只负责“安全删除单条 `UserDraw`”
- 后续如果还有手工触发、命令触发或别的清理入口，可以复用单条删除逻辑
- 相比把所有逻辑继续塞进一个任务类中，边界更清晰，测试更容易写

## 设计概览

### 类结构

新增两个类，均保持与现有 `goto/system/src/main/java/com/freedom/quartz/task/` 风格一致：

1. `LongTimeUnusedUserDrawRemoveTask`
2. `UserDrawRemoveExecutor`

### 职责划分

`LongTimeUnusedUserDrawRemoveTask`

- 提供 `run()` 入口
- 查询所有 3 天未使用的 `UserDraw`
- 逐条调用删除执行器
- 统计成功数、失败数、总扫描数
- 输出任务级汇总日志

`UserDrawRemoveExecutor`

- 负责删除单条 `UserDraw`
- 先处理目录，再处理数据库
- 返回单条执行结果，供任务类汇总
- 在失败时输出单条错误日志

## 数据查询设计

### Repository 设计

不复用旧的 `UserDrawSimpleRepository.findUnusedProjects(Integer delMark, Integer day)`。

原因：

- 该方法依赖 `del_mark`
- 新任务不再使用标记删除语义
- 继续复用会让新旧链路耦合，后续删除旧任务时容易产生行为偏差

建议新增独立查询方法，例如：

- `UserDrawSimpleRepository.findUnusedProjects(Integer day)`

查询语义仅保留：

```sql
update_time < NOW() - INTERVAL :day DAY
```

查询结果继续使用 `UserDrawSimple` 即可，字段足够支撑本任务：

- `id`
- `tmp_folder_name`
- 其他日志辅助字段

虽然 repository 方法可以保留 `day` 参数形式，但本次实现范围固定为“3 天未使用直接删除”。任务类中应明确以常量传入 `3`，不要在本次实现中扩展成通用可配置策略。

### 数据库删除

数据库删除直接复用现有：

- `UserDrawRepository.deleteById(id)`

不新增额外删除 SQL。

## 执行流程

### 任务级流程

`LongTimeUnusedUserDrawRemoveTask.run()` 的推荐流程：

1. 输出任务开始日志
2. 查询所有 3 天未使用的 `UserDrawSimple`
3. 逐条执行 `UserDrawRemoveExecutor.execute(userDraw)`
4. 根据返回结果累计成功数和失败数
5. 输出任务结束汇总日志

### 单条删除流程

`UserDrawRemoveExecutor.execute(userDraw)` 的推荐流程：

1. 校验输入参数
2. 根据 `tmpFolderName` 定位对应目录
3. 如果目录不存在，视为目录删除成功
4. 如果目录存在，尝试删除目录
5. 目录步骤成功后，再删除数据库记录
6. 返回执行结果

## 失败语义与重试

### 目录不存在

如果目录已经不存在，不视为失败。

处理方式：

- 记录必要的调试信息或不记录
- 继续删除数据库记录

原因：

- 用户已确认“目录不存在”应视为目录删除成功
- 这可以覆盖历史中间态或人工清理后的场景

### 目录删除失败

如果目录存在但删除失败：

- 本条记录视为失败
- 不得删除数据库记录
- 输出错误日志
- 保留记录，等待下次任务重试

### 数据库删除失败

如果数据库删除失败：

- 本条记录视为失败
- 输出错误日志
- 数据记录保留，等待下次任务重试

### 关于原子性

文件系统删除和数据库删除无法组成真正的跨资源原子事务。

因此本方案接受如下中间态：

- 目录已经删除成功
- 数据库删除失败

该中间态不会阻塞后续收敛，因为下次任务再次扫描到该记录时：

- 目录不存在会被视为目录删除成功
- 任务将继续尝试删除数据库记录

最终可通过重试收敛到“目录和数据库都被清理完成”。

## 日志设计

### 单条失败日志

失败日志至少包含：

- `id`
- `tmpFolderName`
- 失败阶段：目录删除失败 / 数据库删除失败
- 异常信息

### 汇总日志

任务结束时输出：

- 扫描总数
- 删除成功数
- 删除失败数

日志要求统一为：

- 输出任务开始日志
- 输出单条失败日志
- 输出任务结束汇总日志

不要求输出单条成功日志，避免数据量大时日志刷屏。

## 测试设计

至少新增以下测试：

### 1. 主流程测试

新增类似现有任务测试风格的集成测试，覆盖：

- 创建多条过期 `UserDraw`
- 创建对应临时目录
- 触发 `LongTimeUnusedUserDrawRemoveTask.run()`
- 断言目录已删除
- 断言数据库记录已删除

### 2. 目录不存在语义测试

覆盖以下场景：

- 创建过期 `UserDraw`
- 不创建目录，或先删除目录
- 触发任务
- 断言数据库记录仍可被删除

### 3. 执行器失败路径测试

如集成测试不便稳定构造数据库删除失败，可对 `UserDrawRemoveExecutor` 增加更小粒度测试或 mock 测试，验证：

- 目录删除失败时不会继续删数据库
- 数据库删除失败时返回失败结果并记录错误日志

### 4. 未过期数据保护测试

增加一条负向用例，验证：

- `update_time >= NOW() - INTERVAL 3 DAY` 的记录不会被本任务删除

该测试用于保护本次实现最核心的过期判断口径，避免后续实现时误删边界内数据。

## 与旧链路的关系

本设计是新链路设计，不复用以下旧机制：

- `del_mark`
- `email_notice`
- `mark_time`
- `DelayQueueContainer`
- `DelayedRemoveUserDrawTask`

正式切换时，应避免新旧两套删除链路同时生效，否则可能出现同一条数据被重复处理的问题。

## 验收标准

满足以下条件即可认为方案落地正确：

1. 任务能扫描所有 `update_time` 超过 3 天的 `UserDraw`
2. 目录存在且可删除时，目录和数据库记录都能被删除
3. 目录不存在时，数据库记录仍可继续删除
4. 目录删除失败时，数据库记录不会被删除
5. 数据库删除失败时，记录会保留并输出错误日志
6. 任务不依赖 `del_mark`、`email_notice`、延迟队列
7. 存在覆盖主流程和关键异常语义的测试

## 实施边界

本设计完成后，下一步实施计划应聚焦以下改动：

- 新增任务类
- 新增删除执行器
- 新增独立查询方法
- 新增测试

旧任务删除、旧链路迁移、最终切换顺序可在实施计划阶段单独展开。
