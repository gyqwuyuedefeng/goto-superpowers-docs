# 用户绘图类型统计分组名修正设计

## 背景

当前管理员统计页展示的“图类型”来自 `app_plot_type.folder_name`。该字段实际是内部存储键，页面已出现类似 `81624f715aa54917b244725fa40fdabe` 的 UUID 串，不符合业务语义。

用户确认后的口径为：统计中的“画图类型”应以 `app_plot_group.name` 为准；每日统计表需要在新写入时冗余保存该名称，历史数据不回填。

## 目标

1. 在 `app_user_plot_type_daily_stat` 中新增 `group_name` 字段。
2. 新增统计写入时，通过 `app_plot_type.group_id -> app_plot_group.name` 取得名称，并写入 `group_name`。
3. 热门排行接口不再返回 `plotTypeName`，页面展示统一使用 `groupName`。
4. 管理员统计页将“最高频图类型”、柱状图横轴、排行表格中的类型展示全部切换为 `groupName`。
5. 页面删除重复语义的“图分组”列，仅保留“画图类型”。

## 非目标

1. 不回填历史 `app_user_plot_type_daily_stat` 数据。
2. 不修改 `app_plot_type.folder_name` 字段及其现有业务用途。
3. 不重构其他图类型缓存、文件路径或样例资源逻辑。

## 设计

### 数据模型

`app_user_plot_type_daily_stat` 新增：

```sql
`group_name` VARCHAR(255) DEFAULT NULL COMMENT '画图类型名称'
```

该字段是写入时快照字段，来源为当前 `plot_type_id` 对应的 `app_plot_group.name`。

### 写入链路

1. 接收 `plotTypeId`。
2. 根据 `app_plot_type.id` 查询 `group_id`。
3. 根据 `app_plot_group.id` 查询 `name`。
4. 执行每日统计 upsert 时，将 `group_name` 一并写入。
5. 对同一自然日、同一用户、同一图类型的重复写入，继续仅累加 `draw_count`；同时以本次解析出的 `group_name` 覆盖统计行中的 `group_name`，让后续新写入维持正确值。

### 查询链路

热门排行直接从 `app_user_plot_type_daily_stat.group_name` 取值：

- 返回字段保留：`plotTypeId`、`groupId`、`groupName`、`drawCount`
- 删除返回字段：`plotTypeName`
- 其中 `groupId` 仍可通过 `plot_type_id -> app_plot_type.group_id` 获取，便于前端或后续扩展使用。

历史旧行因为不回填，`group_name` 可能为空；接口保持原值，不通过实时 join 回填当前分组名，避免改变历史数据边界。

### 前端展示

管理员统计页统一将 `groupName` 视为“画图类型”：

- 顶部卡片“最高频图类型”显示 `groupName`
- 柱状图横轴显示 `groupName`
- 表格列从“图类型 + 图分组”收敛为单列“画图类型”
- 若 `groupName` 为空，沿用现有空值兜底文案

## 测试策略

1. 仓储/服务测试：
   - 写入统计时，会先按 `plotTypeId` 解析 `groupName`。
   - upsert SQL 包含 `group_name`。
   - 热门排行投影不再依赖 `plotTypeName`。
2. 前端测试：
   - 汇总、图表和表格均以 `groupName` 生成展示数据。
   - 页面源码约束中不再要求 `plotTypeName` 展示。

## 风险与取舍

- 历史旧数据不回填，因此旧区间结果可能出现空 `groupName`。这是用户明确接受的边界。
- `plotTypeName` 移除会同时影响后端 VO、前端 helper 与页面；需要一次性收口，避免接口和页面短期失配。
