# 通用任务真实终态与 R 运行时依赖设计

## 背景

`goto/task` 在处理 Redis Stream 通用任务命令时，当前链路会在任务加入 `CommonTaskThreadPool` 后立即把 `stream_job_execution` 标记为 `SUCCEEDED` 并 ACK 消息。实际 R 任务随后才在 `common_task_pool_*` 线程中执行。2026-06-30 的拼图任务日志显示，账本已成功并 ACK 后，后台 R 调用才失败：

- R 真实错误：`pandoc version 1.12.3 or higher is required and was not found`
- Java 侧随后又把异常/空结果反序列化为 `RInvokeResponse`，产生二次错误
- R 命令中的 `imageList` 出现 `file=""`，说明 `MergeImageTask` 的图片路径在消息反序列化或字段映射链路中可能丢失

这违反了既有长任务稳定性设计中的顺序：业务真实终态落地后再通知和 ACK。

## 目标

1. Redis Stream listener 不阻塞等待长任务执行。
2. `stream_job_execution` 的 `SUCCEEDED` / `FAILED_*` 必须反映后台任务真实执行结果，而不是提交线程池结果。
3. 成功或最终失败通知必须由真实完成处理器发出。
4. ACK 必须在账本终态写入和通知发布之后执行。
5. `goto/task` 服务实例标识支持可配置 serviceId，默认兼容现有 `goto-task`。
6. `MergeImageTask` 的 `imageList.filePath` 不能在 Redis 消息反序列化后丢失。
7. R 运行时依赖缺失必须返回结构化失败，而不是让 Java 侧解析非协议异常字符串。
8. 拼图任务继续使用 `goto/system` 提交时预生成的 `result-manifest.json`，task 成功阶段校验预期输出文件。

## 非目标

1. 不恢复 RabbitMQ。
2. 不让 task 服务直接推送浏览器 WebSocket。
3. 不把所有通用任务改成同步执行。
4. 不在本设计中执行数据库 DDL/DML。
5. 不把 R 依赖检查写成每个 R 函数各自复制的特殊判断。

## 当前事实

- `CommonTaskCommandStreamConsumer` 当前在 listener finally 中根据 `handler.handle(commonTask)` 的结果写账本和 ACK。
- `MergeImageMessageHandler` 的 `handler.handle` 只是构建 R 脚本并把任务提交到 `CommonTaskThreadPool`。
- `CommonTaskThreadPool.add` 返回的是提交线程池结果，不是后台 R 执行结果。
- `MessageContext.ack()` 使用 Redisson `XACK`，自身带幂等保护，跨线程调用在代码结构上可行。
- `OwnerIdProvider` 当前硬编码 ownerId 前缀：`goto-task:{nodeName}:{bootUuid}`。`nodeName` 来自 `TASK_NODE_NAME` 或主机名，`bootUuid` 为进程启动时随机 UUID。
- `spring.application.name` 当前是 `task:${server.port}`，未用于 Job Ledger ownerId。
- WSL 中可用 Rserve 容器为 `goto_rbase_wsl`，端口映射 `6987->6987` 和 `8787->8787`。

## 方案

采用共享线程池完成回调方案。

`CommonTaskCommandStreamConsumer` 只负责创建/读取 Job Ledger、认领租约、转换 `data` 类型、构造完成上下文、提交后台任务。listener 返回时不写成功终态、不发布执行成功通知、不 ACK。

后台任务执行完成后，由共享线程池触发完成事件。`goto/task` 的完成处理器读取提交时保存的 `jobId`、`ownerId`、`fencingToken` 和 ACK 句柄，先确认仍持有执行权，再按真实结果写账本、发布通知、ACK。

## 数据流

1. system 发布 `CommonTask<MergeImageTask>` 到 `stream:common_task_commands`。
2. task listener 收到命令，校验参数。
3. task listener 通过 `JobLeaseService.getOrCreate` 创建或读取账本。
4. task listener 通过 `JobLeaseService.acquire` 把任务置为 `RUNNING` 并获取 fencing token。
5. task listener 把 `CommonTask.data` 从 `Map` 转换为业务类型，例如 `MergeImageTask`。
6. task listener 构造 `CommonTaskCompletionContext`，保存 `jobId`、`ownerId`、`fencingToken`、`messageId`、`AckHandle`、任务提交时间和错误分类所需元数据。
7. task listener 提交任务到 `CommonTaskThreadPool`，提交成功后返回，不 ACK。
8. 后台 `CommonTaskRunnable` 真实执行 stage 和 R 调用。
9. 共享线程池在 runnable 完成、失败或异常时触发完成回调。
10. `CommonTaskRedisCompletionHandler` 读取完成上下文，检查 `isStillOwner(jobId, ownerId, fencingToken)`。
11. 仍持有执行权时，成功路径写 `SUCCEEDED`，失败路径写 `FAILED_FINAL` 或 `FAILED_RETRYABLE`。
12. 最终成功或最终失败发布 Redis Stream 通知。
13. 通知发布后调用 `AckHandle.ack()`。

## 组件设计

### freedom-public/common-thread

新增通用完成回调扩展点，名称可在实现阶段按现有风格确定，例如：

- `ThreadPoolTaskCompletionListener<T>`
- `ThreadPoolTaskCompletionEvent<T>`
- `ThreadPoolTaskCompletionPublisher`

事件内容至少包含：

- 原始任务对象
- 执行结果 `CheckResultMsg`
- 未捕获异常
- 是否取消
- runnable 类名或线程名
- 任务 key

共享层只负责捕获真实执行结果并发布完成事件，不引入 Redis、Job Ledger 或 `goto` 业务类型依赖。

`StageRunnable` 或 `ThreadPool` 必须保证同一后台任务完成事件只触发一次。正常返回、失败返回、抛异常、取消都要有明确结果语义。

### goto/task

新增 `CommonTaskCompletionContext`：

- `jobId`
- `ownerId`
- `fencingToken`
- `messageId`
- `consumerName`
- `idempotencyKey`
- `businessType`
- `businessId`
- `taskType`
- `commonNoticeUUID`
- `AckHandle`
- `submittedAt`
- `retryCount`

新增 `AckHandle`，只暴露幂等 `ack()`，内部包装 `MessageContext`。业务完成处理器不直接依赖完整 `MessageContext`。

新增 `CommonTaskCompletionRegistry`，用稳定任务 key 关联完成上下文。推荐 key 使用 `consumerName + idempotencyKey`，不要只用线程名。完成、最终失败或确认失权后清理 registry，避免内存泄漏。

新增 `CommonTaskRedisCompletionHandler`：

- 读取完成上下文
- 检查租约和 fencing token
- 根据 `CheckResultMsg` 和异常分类结果写账本
- 发布通用任务通知
- 调用 ACK
- 清理完成上下文

`CommonTaskCommandStreamConsumer` 调整为：

- listener 不再在 finally 中基于提交线程池结果写 `SUCCEEDED`
- 提交线程池失败时可直接按错误分类处理
- 认领失败时不 ACK，等待 PEL 恢复
- 已终态任务直接 ACK

### OwnerIdProvider

新增配置项：

```yaml
task:
  instance:
    service-id: goto-task
```

ownerId 格式改为：

```text
{serviceId}:{nodeName}:{bootUuid}
```

默认 `service-id` 为 `goto-task`，兼容现有 ownerId 前缀。`nodeName` 继续优先读取 `TASK_NODE_NAME`，否则使用主机名。`bootUuid` 继续每次进程启动生成，避免重启后误继承旧执行权。

### gotoTools

新增统一 R 运行时依赖检查工具，而不是在每个函数中手写 Pandoc 判断。

内部函数职责：

- 检查 R 包依赖：`requireNamespace(package, quietly = TRUE)`
- 检查外部工具：例如 `rmarkdown::pandoc_available()`、`Sys.which("pdflatex")`
- 返回统一结构化结果，包含 `status`、`msg`、`data`

R 函数只声明依赖，例如：

```r
requireRuntime(packages = c("rmarkdown"), tools = c("pandoc", "pdflatex"))
```

`mergeImages()` 使用统一检查工具声明依赖。依赖缺失时返回标准失败 JSON。依赖满足时继续按 `fileName + "." + imageType` 生成文件。

### goto/system

保持现有提交时预生成 manifest 的模式。`result-manifest.json` 描述预期输出文件，`goto/task` 成功阶段校验文件是否真实存在且非空。

必要时补充测试，确认 manifest 中 Java 本地输出路径与传给 R 的 `outputDirForR + outputBaseName` 对应同一任务目录。

## 错误分类

### 最终失败并 ACK

以下错误标记 `FAILED_FINAL`，发布失败通知并 ACK：

- 参数错误
- `CommonTask.data` 反序列化失败
- `MergeImageTask.imageList` 为空
- 图片 `filePath` 为空
- manifest 预期输出文件不存在或为空
- R 返回标准业务失败
- R 运行时依赖缺失，例如 `pandoc`、`pdflatex`、必要 R 包缺失

这些错误不会通过自动重试自愈。

### 可重试失败或等待恢复

以下错误不应直接最终失败：

- Rserve 短暂连接失败
- Redis 或数据库临时不可用
- 后台线程被中断
- 完成时租约已失权
- task 进程退出导致后台任务无完成事件

能安全记录时写 `FAILED_RETRYABLE`。如果已失权，则旧 owner 不写状态、不通知、不 ACK，等待租约过期和恢复机制接管。

### 未知异常

listener 提交线程池前发生的未知异常默认不 ACK，让消息留在 PEL。后台线程内发生且能拿到完成上下文的未知异常，优先按可重试处理，并把完整错误摘要写入 `errorMessage`。

## 通知语义

- “提交成功”只代表任务已进入后台执行队列，不代表业务成功。
- “执行成功”只能由后台完成处理器在 `markSucceeded` 成功后发布。
- “执行失败”只能由后台完成处理器在 `markFinalFailure` 成功后发布。
- 失权 owner 不发布任何终态通知。
- system 继续消费 `stream:common_task_notifications` 后推送 WebSocket，task 不直接推送浏览器。

## ACK 语义

ACK 顺序必须是：

```text
后台真实完成
→ 校验仍持有 ownerId + fencingToken
→ 写 SUCCEEDED / FAILED_FINAL
→ 发布通知
→ ACK
```

`FAILED_RETRYABLE` 是否 ACK 由错误分类决定。需要重试的消息不 ACK，等待租约过期和恢复链路。不可重试失败必须 ACK，避免同一坏参数无限重放。

## 测试策略

### freedom-public/common-thread

新增或调整单元测试：

- 正常完成触发一次完成回调
- 返回失败触发一次完成回调
- 抛异常触发一次完成回调
- 取消任务不会重复触发完成回调
- 完成事件中包含任务 key、任务对象、结果或异常

### goto/task

新增或调整测试：

- listener acquire 成功后只提交后台任务，不提前 `markSucceeded`
- listener acquire 成功后不提前 ACK
- 后台完成成功时执行 `markSucceeded -> publishNotification -> ack`
- 后台最终失败时执行 `markFinalFailure -> publishNotification -> ack`
- 后台可重试失败时不发布最终成功通知，不错误 ACK
- `isStillOwner=false` 时不写账本、不通知、不 ACK
- `OwnerIdProvider` 默认 serviceId 为 `goto-task`
- 配置 `task.instance.service-id` 后 ownerId 使用配置值
- `MergeImageTask` 从 Redis 消息反序列化后保留 `imageList.filePath`
- 消息里同时存在 `imageList` 和 `imageItems` 时路径不被空值覆盖

### goto/system

- `ToolsServiceImplTest.submitMergeImageTask_preparesResourcesAndPublishesMergeImageTask`
- `MergeImageDownloadSpecifierTest`
- manifest 中预期输出文件路径与 task 输出路径一致

### gotoTools

- 缺少 R 包时返回标准失败 JSON
- 缺少外部工具时返回标准失败 JSON
- `mergeImages()` 调用统一运行时依赖检查工具
- 依赖满足时生成 `fileName.pdf` 和请求的其他格式文件

## 风险与缓解

1. `MessageContext` 跨线程 ACK 失败  
   缓解：通过 `AckHandle` 封装，单元测试覆盖幂等和异常记录。必要时可改为保存 streamKey/group/messageId 后用 Redisson 重新 ACK。

2. 后台任务完成事件丢失  
   缓解：完成回调放在 runnable finally，覆盖异常和取消路径。进程退出场景依赖租约过期和恢复机制。

3. registry 泄漏  
   缓解：完成、最终失败、失权确认后清理；增加超时清理或启动恢复审计。

4. 错误分类过宽导致坏参数反复重试  
   缓解：明确参数、依赖、manifest 校验错误为 `FAILED_FINAL`。

5. 错误分类过窄导致瞬时故障不重试  
   缓解：未知后台异常默认可重试，只有可识别的不可恢复错误最终失败。

6. R 依赖检查引入新返回协议差异  
   缓解：统一使用现有 `returnStatus` / `RInvokeResponse` JSON 协议，并补充 R 与 Java 解析测试。

## 验收标准

1. 通用任务提交到线程池后，listener 不会立即写 `SUCCEEDED` 或 ACK。
2. 后台任务成功后，账本成功、通知成功、ACK 顺序可通过测试或日志验证。
3. 后台任务最终失败后，账本失败、通知失败、ACK 顺序可通过测试或日志验证。
4. 失权 owner 不会发布终态通知或 ACK。
5. `OwnerIdProvider` 支持配置 serviceId，默认行为兼容现状。
6. 拼图任务 Redis 消息反序列化后图片路径不为空。
7. R 依赖缺失时返回结构化失败，Java 不再产生误导性的二次反序列化错误。
8. WSL Docker Rserve 环境可用于手工验证成功和失败路径。
