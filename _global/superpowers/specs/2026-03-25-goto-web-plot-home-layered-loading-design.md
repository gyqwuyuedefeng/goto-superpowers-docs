# goto-web 画图首页分层加载与首屏优先设计

## 1. 背景

当前画图首页位于 `goto-web/src/views/goto/plot/index.vue`，现有加载链路如下：

1. 页面进入后调用 `groupList()` 获取分组和卡片数据
2. 前端读取 IndexedDB 缓存并尝试回填 `imageBase64`
3. 前端通过全局总线发送一次 `commonWebSocketMessageSendToServer`
4. 后端通过 WebSocket 逐张回传缩略图

这个方案能工作，但在体验和稳定性上存在明显问题：

1. 页面缺少完整的加载状态模型，视觉上只有“有图”和“没图”两种结果
2. “未加载完成”和“暂无预览”被混在一起，用户容易误判
3. WebSocket 回图没有明确的优先级和完成标记，首屏是否优先不可控
4. IndexedDB 当前 key 依赖 `i/j`，一旦排序变化或新增卡片，缓存存在串位风险
5. 随着卡片数量从当前 `50-200` 张继续增加，现有全量逐张回传会放大前后端压力和页面抖动

本次设计的目标不是只给当前页面“加一个骨架屏”，而是把画图首页改造成一条可扩展、可收口、可降级的企业级加载链路。

## 2. 已确认约束

本次设计基于以下已确认条件：

1. 当前所有用户看到的首页内容一致，不存在用户级差异化列表
2. 当前卡片数量大约在 `50-200` 张，后续预计还会继续增加
3. 当前阶段仍然要求首页先展示“全部内容的总览”，不改成仅显示部分内容
4. 用户已确认加载目标优先级为：`体验 > 稳定性 > 成本`
5. 用户已确认允许一起改前后端加载链路，只要收益明显
6. 用户已确认首页体验采用折中方案：先显示全部分组和首屏卡片结构，首屏缩略图优先，非首屏延后

## 3. 目标

### 3.1 体验目标

1. 进入页面后尽快看到稳定的分组结构和卡片框架，不出现大面积空白
2. 首屏 `1-2` 屏内的卡片优先显示真实缩略图
3. 非首屏卡片延后加载，但用户仍能立即感知页面结构完整
4. “等待中”、“暂无预览”、“加载失败”三类状态在视觉上明确区分

### 3.2 稳定性目标

1. 缩略图回传顺序可控，优先保障首屏体验
2. WebSocket 批次有开始、增量、结束的清晰边界
3. 单卡片失败或单批次失败不影响整页可用性
4. 缓存命中逻辑不依赖分组位置索引，避免排序变化导致错图

### 3.3 工程目标

1. 保留现有 `groupList + WebSocket + IndexedDB` 三层能力，但重新梳理职责
2. 建立统一的页面级状态和卡片级状态机
3. 为未来“查看全部”、“卡片数量继续上涨”、“更强缓存策略”留下扩展空间

## 4. 非目标

本次设计不包含以下内容：

1. 不重做画图首页整体视觉风格，只优化加载相关交互和状态呈现
2. 不把当前页面直接改造成纯分页或纯虚拟列表页面
3. 不在本轮直接把缩略图完全迁移到 CDN 或独立图片服务
4. 不修改图表绘制页 `plot/draw` 的业务逻辑
5. 不在设计阶段引入与本问题无关的全站状态管理重构

## 5. 方案决策

本轮讨论过三类方案：

1. 仅增加骨架屏，保留现有逐张回图逻辑
2. 分层加载 + 首屏优先
3. 缩略图服务化，WebSocket 仅负责状态通知

最终采用 `方案 2：分层加载 + 首屏优先`。

原因如下：

1. 相比“只加骨架屏”，它不仅改善感知，还能从协议层解决首屏优先和加载收口问题
2. 相比“完全服务化”，它更适合当前阶段，改动可控，但仍具备企业级稳定性
3. 它最符合当前明确的目标：`首屏先完整、首屏图先真、非首屏延后`

## 6. 总体架构

### 6.1 三层加载链路

首页加载拆成三层：

1. `页面结构层`
2. `首屏优先图层`
3. `延后加载层`

### 6.2 页面结构层

`groupList()` 只负责快速返回页面结构所需的轻量数据：

1. 分组信息
2. 卡片基础信息
3. 排序信息
4. 可选的缩略图版本号或更新时间

前端拿到结构数据后，立刻渲染所有分组标题和卡片容器，不等待任何图片加载完成。

### 6.3 首屏优先图层

前端在结构渲染后，识别当前视口内的卡片，并发起一批高优先级缩略图请求。

后端必须优先处理这一批任务，优先回传首屏 `1-2` 屏卡片的缩略图，并在这一批结束后返回明确完成标记。

### 6.4 延后加载层

非首屏卡片不参与首轮资源竞争，而是在以下时机触发后续批次：

1. 首屏批次完成后
2. 浏览器空闲时
3. 用户滚动接近目标区域时

这样可以在不牺牲首页总览感知的前提下，降低同时回图的峰值压力。

## 7. 状态模型

### 7.1 页面级状态

页面级状态用于控制整体 UI 反馈：

- `booting`：结构数据尚未返回
- `struct_ready`：分组和卡片框架已渲染
- `first_screen_done`：首屏关键卡片已完成加载或已明确收口
- `all_done`：当前轮次全部批次结束
- `fatal_error`：结构接口失败，页面不可继续

### 7.2 卡片级状态

卡片级状态用于控制单张卡片的视觉反馈：

- `idle`：尚未进入当前加载轮次
- `cached`：本地缓存命中
- `queued`：已加入加载队列，等待处理
- `loading`：后端处理中或图片回传中
- `ready`：真实缩略图可展示
- `empty`：后端明确返回无预览图
- `failed`：该卡片加载失败
- `timeout`：该卡片在预设时间内未收到结果

### 7.3 状态使用原则

1. `queued/loading` 阶段展示骨架，不展示“暂无预览”
2. 只有后端明确返回空结果时才进入 `empty`
3. `failed/timeout` 必须与 `empty` 在视觉上区分
4. 页面是否结束 loading 由批次完成标记驱动，而不是由“是否还有空卡片”猜测

## 8. 前后端职责拆分

### 8.1 HTTP 结构接口职责

HTTP 接口负责“快返回结构”，不负责大体积图片数据。

建议返回内容包括：

1. 分组列表
2. 卡片基础元数据
3. 稳定业务标识，例如 `groupId`、`typeId`
4. 可选的图片版本信息，例如 `imageVersion` 或 `updatedAt`

### 8.2 WebSocket 职责

WebSocket 负责“带优先级的缩略图增量流”，而不是无序全量散射回图。

WebSocket 需要支持：

1. 高优先级首屏批次
2. 低优先级延后批次
3. 批次内逐张回传
4. 批次完成标记
5. 单卡片失败或空结果标记

### 8.3 IndexedDB 职责

IndexedDB 负责本地命中和版本校验。

缓存策略要求：

1. 缓存 key 不再使用 `i/j`
2. 缓存 key 使用稳定业务标识，例如 `groupId + typeId + imageVersion`
3. 命中旧版本缓存时必须能识别并刷新
4. 页面刷新后可优先显示已命中的真图，减少首屏等待

## 9. WebSocket 协议调整建议

### 9.1 请求模型

前端发起缩略图请求时，不再只发送一个模糊的 `group_list_image_base64` 事件。

建议请求体至少包含：

- `requestId`
- `priority`
- `rangeType`
- `cardKeys`
- `groupVersion` 或等效版本标识

字段语义如下：

- `requestId`：用于标识本次批次
- `priority`：例如 `high`、`low`
- `rangeType`：例如 `first_screen`、`viewport`、`deferred`
- `cardKeys`：本次需要回图的稳定卡片标识集合
- `groupVersion`：用于前后端确认结构版本是否一致

### 9.2 回传模型

后端逐张回传时，建议每条消息至少包含：

- `requestId`
- `cardKey`
- `status`
- `image`
- `version`

其中：

- `status` 取值建议为 `ready / empty / failed`
- `image` 仅在 `ready` 时存在
- `version` 用于缓存校验

另外，协议必须支持独立的批次完成消息，例如：

- `requestId`
- `type: batch_done`
- `rangeType`
- `summary`

独立完成消息优先于把 `isLast` 挂在最后一张卡片上，因为当某一批次没有任何成功图片、只有空结果或失败结果时，仍然需要稳定收口。

### 9.3 完成标记要求

完成标记是本轮设计的关键能力，必须满足：

1. 前端可以明确知道首屏批次结束
2. 前端可以明确知道延后批次结束
3. 即使部分卡片失败，批次仍然可以正常收口
4. 即使某一批次没有成功图片返回，也能稳定发出结束信号

没有完成标记时，前端无法正确结束骨架、显示失败态、推进下一轮批次。

## 10. 前端页面表现

### 10.1 首屏加载规则

首页首屏表现按以下顺序执行：

1. 进入页面后先请求结构数据
2. 结构返回后立即渲染所有分组标题和卡片框架
3. 视口内卡片进入高优先级加载
4. 非首屏卡片先维持轻骨架占位
5. 首屏批次收口后，再继续非首屏批次

### 10.2 骨架屏使用原则

骨架屏适合这个页面，但只作为完整状态体系的一部分。

具体原则如下：

1. 页面 `booting` 阶段使用分组级骨架和固定卡片骨架
2. 卡片 `queued/loading` 阶段使用图片骨架
3. 已知空结果不再显示骨架，而显示明确的“暂无预览”
4. 已知失败或超时显示失败态和重试入口

### 10.3 图片展示策略

1. 缓存命中时直接显示真实图片
2. WebSocket 返回真实图片后，用轻量淡入替换骨架
3. 动画控制在 `120-180ms`，避免闪烁和抖动
4. 不使用整页级 loading 蒙层，避免阻断浏览

### 10.4 进度反馈

页面可以提供轻量级进度提示，例如：

1. 首屏预览已加载数量
2. 当前批次是否完成

进度提示只能辅助理解，不应替代骨架和卡片状态本身。

## 11. 延后加载策略

### 11.1 首轮范围

首轮只覆盖首屏 `1-2` 屏内的卡片，而不是一次性抢占全部卡片资源。

### 11.2 后续触发条件

非首屏批次在以下条件下触发：

1. 首屏批次完成
2. 用户滚动接近目标卡片
3. 浏览器空闲

### 11.3 未来扩展

本设计要为未来两类扩展留接口：

1. “查看全部”模式下的进一步分批加载
2. 卡片数量持续增长后的更强懒加载策略

本轮不直接实现这两项，但协议和状态模型不能堵死后续演进路径。

## 12. 错误处理与降级

### 12.1 可恢复错误

以下情况视为可恢复错误：

1. 单卡片失败
2. 单卡片超时
3. 单卡片无预览
4. 某一低优先级批次部分失败

处理原则：

1. 不阻断整页浏览
2. 页面结构继续可用
3. 卡片进入 `empty / failed / timeout` 对应状态
4. 允许后续轻量重试

### 12.2 不可恢复错误

以下情况视为不可恢复错误：

1. `groupList` 结构接口失败
2. 当前页面核心结构数据无法初始化

处理原则：

1. 展示整页错误态
2. 提供明确重试入口
3. 不把全页停留在无限骨架或无限 loading 中

### 12.3 WebSocket 不可用时的降级

如果 WebSocket 整体不可用：

1. 页面仍应优先显示结构层
2. 已命中的缓存图继续展示
3. 未命中的卡片给出明确提示，不允许长期无反馈
4. 页面不得因为某一批次未完成而永远卡在 loading

## 13. 测试与验证要求

至少覆盖以下验证场景：

1. 无缓存进入页面时，是否先稳定展示结构和骨架
2. 有缓存进入页面时，缓存图是否优先展示
3. 排序变化或新增卡片后，缓存是否仍能正确命中
4. 首屏卡片是否优先于非首屏卡片加载
5. 后端返回 `ready / empty / failed / isLast` 时，前端状态是否正确收口
6. WebSocket 延迟、中断、部分失败时，页面是否仍保持可用
7. 单卡片失败是否不会影响整页状态推进

## 14. 上线顺序

建议分三步上线，降低联调风险：

### 14.1 第一步：前端状态与表现层

先在前端建立统一状态模型，并改造以下能力：

1. 页面级状态
2. 卡片级状态
3. 骨架、空态、失败态的视觉区分
4. 稳定缓存 key 的本地结构

这一阶段即使后端协议未改完，也能显著改善感知体验。

### 14.2 第二步：协议与批次收口

联调后端完成以下改造：

1. 高低优先级批次
2. 批次完成标记
3. 单卡片状态回传
4. 版本校验字段

这一阶段解决首屏优先和状态收口问题。

### 14.3 第三步：延后加载与后续优化

最后接入：

1. 滚动接近触发
2. 浏览器空闲时预取
3. 更细的性能统计与调优

## 15. 实施边界与直接影响文件

本设计预期主要影响以下区域：

1. `goto-web/src/views/goto/plot/index.vue`
2. `goto-web/src/api/plot.js`
3. `goto-web/src/utils/IndexedDBUtils.js`
4. `goto-web/src/views/components/websocket/index.vue`
5. 后端画图首页结构接口与缩略图 WebSocket 消息处理逻辑

其中重点变更点为：

1. 首页状态建模
2. WebSocket 请求与回传协议
3. 本地缓存 key 与版本策略
4. 首屏优先与非首屏延后策略

## 16. 最终结论

本轮不采用“只加骨架屏”的最小方案，也不直接切到“缩略图服务化”的长期大改。

最终采用的方案是：

`结构先出、首屏优先、非首屏延后、协议可收口、缓存可校验。`

这套方案能够在当前 `50-200` 张卡片规模下显著提升首页体验，并为后续卡片数量上涨和“查看全部”等能力预留稳定扩展路径。
