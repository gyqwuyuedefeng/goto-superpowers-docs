# goto-web `goto` 目录 Dialog 分级响应式统一设计

## 1. 背景

`goto-web/src/views/goto` 下的大多数业务弹窗已经复用统一包装组件 [dialog.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/dialog.vue)，但当前实现仍然以业务页面传入的固定 `width` / `height` 像素值为主。

现状问题主要集中在三点：

1. 同类弹窗在不同页面的尺寸策略并不一致，视觉基线不稳定
2. 组件虽然做了居中处理，但缺少基于视口的统一宽高约束，小屏或窗口缩小时容易不协调
3. 内容较长的弹窗虽然部分场景可滚动，但没有形成统一的 header / body / footer 布局规范

本轮目标不是重写所有弹窗实现，也不是把整个前端项目所有弹窗体系都重构一遍，而是聚焦 `goto` 目录中已经走 `Dialog` 包装组件这一条链路，建立统一、可维护、企业级的响应式弹窗尺寸规范。

## 2. 已确认约束

本次设计基于以下已确认约束：

1. 统一范围只覆盖 `goto-web/src/views/goto` 下已经使用 `Dialog` 包装组件的弹窗
2. 不纳入直接书写 `el-dialog` 的页面，本轮不处理那部分体系
3. 对外不再继续传递 `width`、`height`
4. 组件对外统一改为 `size="sm|md|lg"` 模式
5. 最终宽高完全由 `Dialog` 组件根据 `size + viewport` 计算
6. 现有弹窗的历史 `width`、`height` 只用于迁移时辅助判断归类到 `sm` / `md` / `lg`
7. 弹窗必须保持水平、垂直居中显示
8. 弹窗必须能根据视口自动调整宽高，不允许整体溢出视口

## 3. 目标

### 3.1 体验目标

- `goto` 目录内使用统一 `Dialog` 的弹窗具备一致的尺寸语言
- 弹窗在桌面端始终稳定居中，不因内容高度差异出现位置漂移
- 在窗口缩小、小屏设备或较低可视高度场景下，弹窗能自动收敛到安全尺寸
- 长内容弹窗只让内容区滚动，不让整个弹窗脱离视口

### 3.2 工程目标

- 把业务页面里的像素尺寸语义回收到共享组件
- 建立 `sm` / `md` / `lg` 的明确分级，替代页面级硬编码 `width` / `height`
- 降低后续维护和新增弹窗时的尺寸决策成本
- 把改造边界控制在 `Dialog` 组件及其 `goto` 目录调用点内

### 3.3 维护目标

- 后续新增弹窗时，业务只需要选择 `size`，不再手动决定像素宽高
- 如果未来需要统一调整弹窗比例，只修改一个组件和一套尺寸规则
- 让尺寸体系具有“按语义使用”的团队约束，而不是继续让页面各自演化

## 4. 非目标

本轮不做以下事情：

1. 不改造 `goto-web` 中未使用统一 `Dialog` 的 `el-dialog`
2. 不重写弹窗业务内容结构
3. 不修改确认、取消、打开、关闭等业务行为链路
4. 不引入自动内容识别或根据 DOM 复杂度智能猜尺寸
5. 不在本轮设计阶段处理全项目所有弹窗统一

## 5. 当前实现与问题归纳

当前统一包装组件 [dialog.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/dialog.vue) 具备以下特点：

1. 通过 `width + 'px'` 直接把宽度传给 `el-dialog`
2. 通过打开时操作内部 DOM 的方式设置固定高度
3. 使用额外 class 控制水平和垂直居中
4. 已经把 `.el-dialog` 设为纵向 flex，`body` 区域可滚动

现有实现的问题不在“是否有统一组件”，而在“统一规则仍然是像素驱动”：

- 业务调用点持续传递不同的 `width` / `height`
- 组件内部缺少清晰的尺寸分级语义
- 对小屏和视口变化缺少统一收敛规则
- 业务页面无法只从用途判断尺寸，而必须反复试像素值

## 6. 方案对比与决策

### 6.1 备选方案

#### 方案 A：组件内部自动推断尺寸等级

根据插槽内容、表格数量、表单长度等自动判断 `sm` / `md` / `lg`。

优点：

- 调用侧改动少
- 旧页面表面上迁移成本最低

缺点：

- 规则隐式，难以预测
- 容易误判，后续维护成本高
- 不符合企业级共享组件“显式语义优先”的原则

#### 方案 B：组件显式接受 `size`

业务调用侧明确传 `size="sm|md|lg"`，组件内部统一映射到响应式宽高规则。

优点：

- 规则清晰
- 团队协作成本低
- 后续扩展最稳定

缺点：

- 现有调用点需要迁移归类

#### 方案 C：保留 `width` / `height` 与 `size` 双轨并存

旧页面继续传像素，新页面逐步传 `size`。

优点：

- 过渡阻力最小

缺点：

- 组件边界长期不干净
- 团队会继续传像素，统一目标被稀释
- 后续维护会长期背双语义负担

### 6.2 最终决策

采用 **方案 B：组件显式接受 `size`**。

设计理由：

1. 这是最符合企业级后台组件治理的方式
2. 尺寸语义从业务页面剥离后，后续统一调整成本最低
3. 团队新增弹窗时只需按业务用途选择尺寸等级，而不是反复调像素
4. 可以通过一次迁移把 `goto` 目录下已有弹窗逐步纳入稳定规则

## 7. 统一尺寸模型

### 7.1 对外接口

`Dialog` 组件对外尺寸接口统一为：

- `size="sm|md|lg"`，默认 `md`
- `fullscreen` 保留，优先级高于 `size`

本轮设计下：

- 不再要求业务传 `width`
- 不再要求业务传 `height`
- 现有历史像素值仅用于迁移时辅助归类

### 7.2 尺寸语义

#### `sm`

适用场景：

- 确认类弹窗
- 短表单
- 轻量选择器
- 简单补录

#### `md`

适用场景：

- 常规设置
- 普通业务表单
- 常见配置面板
- 中等复杂度的选择器

#### `lg`

适用场景：

- 表格型弹窗
- 历史记录
- 模板管理
- 复杂配置
- 内容较长、信息密度较高的业务弹窗

### 7.3 视口比例规则

建议默认规则如下：

- `sm`：桌面宽约 `42vw`，高上限 `58vh`
- `md`：桌面宽约 `60vw`，高上限 `72vh`
- `lg`：桌面宽约 `78vw`，高上限 `82vh`
- 小屏统一收敛：宽约 `92vw`，高上限 `88vh`

这组规则表达的是“默认桌面分级 + 小屏安全收敛”：

1. 桌面端不同用途的弹窗保留层级差异
2. 小屏时优先保证可用性，而不是坚持桌面档位比例
3. 所有档位都不能突破视口安全边界

### 7.4 精确实施规则

为避免不同实现者理解偏差，本轮规划阶段使用以下精确规则：

#### 桌面分级规则

当视口宽度 `>= 768px` 时：

- `sm`：宽度取 `42vw`，`max-width: 720px`，`max-height: 58vh`
- `md`：宽度取 `60vw`，`max-width: 1040px`，`max-height: 72vh`
- `lg`：宽度取 `78vw`，`max-width: 1440px`，`max-height: 82vh`

#### 小屏收敛规则

当视口宽度 `< 768px` 时，无论 `sm` / `md` / `lg`，统一切换为：

- 宽度：`92vw`
- 最大高度：`88vh`

#### 高度兜底规则

当视口高度 `< 700px` 时，即使视口宽度仍属于桌面端，也以“可用性优先”收紧最大高度：

- `sm`：`max-height: min(58vh, calc(100vh - 32px))`
- `md`：`max-height: min(72vh, calc(100vh - 32px))`
- `lg`：`max-height: min(82vh, calc(100vh - 32px))`

以上规则的目的不是让所有设备尺寸绝对一致，而是确保：

1. 有明确断点，便于实现和验收
2. 小屏直接进入统一安全模式
3. 即使在矮视口桌面窗口里，弹窗也不会因为 `vh` 过大而挤出安全区域

## 8. 布局与交互规范

### 8.1 居中规则

统一要求：

- 弹窗始终水平居中
- 在常规视口空间足够时，弹窗垂直居中
- 不再依赖业务内容高度去决定初始定位

当视口高度不足时，优先级顺序明确如下：

1. footer 必须可见
2. 整个弹窗必须处于视口安全区域内
3. 在满足前两条的前提下再尽量保持垂直居中

也就是说：

- 正常情况下使用垂直居中
- 当垂直居中会导致底部溢出时，允许退出严格垂直居中
- 退出垂直居中后，弹窗以上下安全边距方式保持在视口内，不允许 footer 被裁切

### 8.2 滚动规则

统一要求：

- header 固定
- footer 固定
- body 区域负责滚动

这样可以保证：

1. 用户在长内容弹窗中始终看到标题和操作按钮
2. 复杂表单或表格不会把整个弹窗推离视口
3. 不同弹窗间的滚动体验一致

### 8.3 视口变化规则

当以下情况发生时，弹窗尺寸需要重新按视口规则计算：

- 初次打开
- 浏览器窗口 resize
- 小屏与大屏宽度切换

要求结果是：

- 弹窗宽高及时收敛
- 不出现横向超界
- 不出现底部不可达或 footer 被裁切
- 视口足够时恢复到垂直居中
- 视口不足时自动切换到“安全边距优先”布局

### 8.4 安全边距规则

当弹窗无法在当前视口中保持严格垂直居中时，统一切换到安全边距模式：

- 顶部安全边距：`16px`
- 底部安全边距：`16px`
- 弹窗总高度上限：`calc(100vh - 32px)`

安全边距模式下：

1. 弹窗仍保持水平居中
2. 弹窗顶部与底部都必须保留最小 `16px` 空间
3. 若内容仍超出，则继续压缩 body 可滚动区域，而不是牺牲 footer 可见性

进入安全边距模式的判定规则统一为：

- 当按垂直居中放置后，弹窗底部会超过 `100vh - 16px`，或弹窗顶部会小于 `16px` 时，立即退出严格垂直居中并切换到安全边距模式

## 9. 迁移原则

### 9.1 迁移范围

仅迁移 `goto-web/src/views/goto` 下使用 `Dialog` 包装组件的页面。

### 9.2 迁移方式

迁移时删除调用侧的 `width` / `height` 传递，改为显式 `size`。

迁移归类遵循用途优先，而不是机械像素映射：

- 确认、短输入、轻量选择归 `sm`
- 常规设置、普通配置归 `md`
- 表格、历史记录、模板管理、复杂配置归 `lg`

### 9.3 历史像素值的使用方式

历史 `width` / `height` 只作为辅助参考：

- 用来快速识别原先偏小、偏中、偏大的弹窗
- 不作为新的尺寸公式输入
- 不保留为对外长期兼容接口

原因是：

1. 同样的像素值可能对应完全不同的业务复杂度
2. 机械映射会把旧的不合理尺寸直接继承到新体系中
3. 迁移目标是“按用途统一”，不是“按旧像素复刻”

## 10. 组件职责与调用侧职责

### 10.1 `Dialog` 组件职责

- 接收 `size`
- 根据 `size + viewport` 计算最终宽度和最大高度
- 管理统一的居中显示
- 管理 header / body / footer 布局
- 管理 body 滚动边界
- 在视口变化时重新应用尺寸规则

### 10.2 业务调用侧职责

- 根据业务用途传递 `size`
- 不再传入固定 `width` / `height`
- 不在页面局部再次定义独立尺寸体系

## 11. 验收标准

本轮设计完成后的验收重点如下：

1. `goto` 目录内通过 `Dialog` 打开的同类弹窗具有一致尺寸语言
2. 桌面端弹窗打开后稳定处于水平、垂直居中位置
3. 浏览器缩放或小屏场景下，弹窗宽高会自动收敛，不溢出视口
4. 长内容场景下只有 body 滚动，header / footer 保持可见
5. `lg` 类型弹窗在常见笔记本分辨率下仍可正常使用
6. 现有打开、关闭、确认、取消以及 `open` / `opened` / `close` 事件行为不受影响

建议至少覆盖以下验证样本：

- 一个 `sm` 弹窗
- 一个 `md` 弹窗
- 一个 `lg` 弹窗
- 一个带长表单或长列表的弹窗
- 一个窗口 resize 后重新适配的场景

## 12. 风险与注意事项

### 12.1 内容布局历史债务暴露

部分业务内容过去依赖固定像素高度。切换到统一 `size` 后，可能暴露出内部布局本身未做自适应的问题。

这类问题不属于尺寸体系失败，而是业务内容区布局需要顺手校正。

### 12.2 尺寸归类偏差

少数弹窗原先尺寸设置不合理，单纯参考旧像素值可能会误判档位。

因此迁移时必须以业务用途为主，而不是完全按旧值自动映射。

### 12.3 组件内部 DOM 操作兼容性

当前 `Dialog` 已直接操作 Element UI 内部 DOM。改造时需要确保样式应用时机稳定，避免出现打开瞬间闪动、尺寸跳变或 class 未生效的问题。

## 13. 实施边界结论

本轮方案的核心结论如下：

1. 统一入口保持为 [dialog.vue](/mnt/f/IdeaProjects/goto-software/goto-web/src/views/components/dialog.vue)
2. 对外尺寸接口收敛为 `size="sm|md|lg"`
3. 宽高完全按视口比例规则计算，不再由业务传入像素值主导
4. 只覆盖 `goto-web/src/views/goto` 下已使用 `Dialog` 的弹窗
5. 迁移时按业务用途把历史弹窗归类到 `sm` / `md` / `lg`

这份设计已经足够进入实现规划阶段，不再继续扩展到 `goto-web` 全项目其他弹窗体系。
