# 个人工程规范体系设计

## 背景

当前已经有一版初步的 Python 开发规范，内容覆盖了依赖管理、FastAPI、异步任务、LangGraph、日志与 SSE 等方向，说明目标已经不是“写代码风格建议”，而是建立一套在新项目启动时可直接套用的默认工程标准。

但这版草稿仍有几个明显问题：

- 通用 Python 规则与特定技术栈规则混在一起，容易让简单脚本也被套上 Agent 或微服务约束。
- 规范更像偏好清单，还不是可供 AI 直接执行的工程规则。
- 缺少企业级基础治理内容，例如配置管理、错误处理、测试分层、安全基线、CI 门禁。
- 没有明确的“默认栈、例外条件、适用场景”边界，后续仍会反复讨论技术选型。

本次设计的目标不是只润色原有条款，而是把它升级为一套长期可维护、结构化、可让 AI 直接遵循的个人工程规范体系。

## 目标

- 为个人新项目提供默认工程标准，尽量消除重复的基础技术选型讨论。
- 将规范组织为“总纲 + 模板 + 跨栈约束 + 启动清单”的结构，避免不同场景相互污染。
- 让规范同时服务于人工与 AI：人用于快速决策，AI 用于生成结构、代码、测试与交付物。
- 对 Python、前端、Java 分别形成独立章节，提升长期维护性。
- 在 Python 规范中，将通用工程底线与 API/AI Agent/脚本三类模板明确拆开。

## 非目标

- 本次不直接产出所有分册的最终成文版本。
- 本次不替代具体项目的业务设计文档。
- 本次不强行统一所有项目的数据库、中间件或部署方式，只定义个人默认标准与偏离条件。
- 本次不把规范写成团队流程制度，重点是个人长期复用与 AI 执行。

## 设计原则

### 1. 总纲与技术栈分离

通用规范只定义长期稳定的工程底线，不直接绑定特定框架。具体框架、Agent 编排、异步任务、微服务接入等规则必须下沉到对应模板或跨栈规范中。

### 2. 模板优先

新项目启动时，不再从零开始讨论技术选型，而是先判断项目属于哪种模板，再继承对应的默认栈和执行方式。

### 3. 默认优先，例外显式

规范不是百科全书，而是默认工程标准。允许偏离默认栈，但必须有明确理由、影响说明和替代方案。

### 4. AI 可执行

规范必须能被 AI 直接使用，因此每条关键规则都应具备清晰的约束等级、适用范围、执行标准和例外条件。

### 5. 长期维护友好

规范应支持按主题拆分，便于未来独立维护 Python、前端、Java 和跨栈协作部分，而不是把所有规则压在单一长文档里。

## 总体结构

建议将规范拆成如下文档体系：

- `00-governance.md`
- `10-python-base.md`
- `11-python-api-template.md`
- `12-python-ai-agent-template.md`
- `13-python-script-template.md`
- `20-frontend-base.md`
- `21-frontend-vue-template.md`
- `30-java-base.md`
- `31-java-spring-template.md`
- `40-cross-stack-contract.md`
- `90-project-bootstrap-checklist.md`

### 各文档职责

`00-governance.md` 负责定义所有项目统一继承的工程底线，包括目录、配置、日志、测试、安全、可观测性、AI 协作约束等。

`10-python-base.md` 负责定义所有 Python 项目都应遵守的基础规范，不直接强绑 FastAPI、LangGraph、TaskIQ 等特定技术。

`11` 到 `13` 三份模板文档负责定义不同 Python 场景下的默认技术栈与实现方式。

`20`、`21`、`30`、`31` 负责前端与 Java 的独立规范，避免所有约束都塞入 Python 文档。

`40-cross-stack-contract.md` 负责 Python、Java、前端三者之间的契约、事件格式、trace 透传、任务状态流转等跨栈规则。

`90-project-bootstrap-checklist.md` 负责新项目启动时的执行清单，让 AI 或人工都能按固定步骤完成初始化。

## 使用方式

这套规范的使用流程应固定为：

1. 启动新项目时，先识别项目类型。
2. 从预设模板中选择最贴近的默认模板。
3. 继承总纲规则与模板规则。
4. 仅在出现明确例外条件时偏离默认技术栈。
5. 由 AI 或人工按启动清单生成项目骨架、依赖、配置、测试和最小运行说明。

建议在总纲中明确一条最高优先级原则：

`除非项目需求明确冲突，否则 AI 与人工都应优先遵守本规范中的默认模板与默认技术栈，不重复进行基础技术选型讨论。`

## Python 规范拆分方案

### 1. Python 通用规范

Python 通用规范只覆盖所有 Python 项目都适用的内容，包括：

- Python 版本策略
- 依赖管理与锁文件
- 项目结构与模块边界
- 类型系统与静态检查
- 配置与密钥管理
- 数据建模与序列化基础约束
- 异步与并发原则
- 日志、监控与追踪基线
- 错误处理与可靠性
- 测试策略
- 安全基线
- AI 协作执行要求

这一层可以保留 `Poetry`、`Pydantic v2`、类型注解、结构化日志、`pytest`、`ruff`、`mypy` 等长期稳定标准。

### 2. Python API 服务模板

适用于提供 HTTP API 的标准后端服务，默认栈建议固定为：

- Python `3.11`
- `Poetry`
- `FastAPI`
- `Pydantic v2`
- `pydantic-settings`
- `SQLAlchemy 2.x`
- `Alembic`
- `httpx`
- `Redis`
- `pytest + pytest-asyncio + pytest-cov`
- `ruff + mypy`
- `uvicorn`
- `Docker`

该模板中定义：

- `FastAPI lifespan` 初始化和销毁全局资源
- 使用 `Depends + yield` 管理请求级资源
- OpenAPI 与响应模型规则
- 长耗时任务接口的返回策略
- SSE 作为默认实时状态推送方式

### 3. Python AI Agent 服务模板

适用于多步 LLM 工作流、工具调用、Agent 编排、人工介入类项目。该模板继承 Python API 服务模板，并增加：

- `LangGraph` 作为默认 Agent 编排框架
- `TaskIQ` 作为默认异步任务框架
- `Langfuse` 作为 LLM tracing 标准
- `Redis` 或 `Redis Stream` 作为状态与事件中枢
- `SSE` 作为前端进度推送方式
- `human-in-the-loop` 作为显式工作流节点
- 基于状态持久化的重试与恢复策略

该模板中定义：

- 使用 `Supervisor Pattern` 组织多智能体工作流
- 按节点而不是按超长函数组织 Agent 行为
- 对重试次数、人工介入、错误回放、任务状态流转设统一约束

### 4. Python 脚本与数据任务模板

适用于 CLI、ETL、批处理、分析脚本、研究型工具。默认栈建议固定为：

- Python `3.11`
- `Poetry`
- `pydantic-settings`
- `Pydantic v2`
- 结构化日志
- `Typer`
- `httpx`
- 按需使用 `pandas`
- `pytest`
- `ruff + mypy`

该模板强调：

- 即使是脚本，也不得长期依赖全局环境、裸配置或 `print()` 驱动调试。
- 一旦出现长期运行、多人协作、数据库接入、对外服务等特征，应升级为正式服务模板。
- 一旦引入多步 LLM 编排，应升级为 AI Agent 模板。

## 现有草稿的技术归位

当前草稿中出现的技术与规则应按下列方式重组：

- `Poetry`、类型注解、`Pydantic v2`、异步原则、结构化日志 -> `10-python-base.md`
- `FastAPI lifespan`、`Depends + yield`、OpenAPI、SSE 接口 -> `11-python-api-template.md`
- `TaskIQ`、`LangGraph`、`Supervisor Pattern`、`human-in-the-loop`、`Langfuse`、重试 Prompt 调整 -> `12-python-ai-agent-template.md`
- `Nacos`、Redis Stream 事件命名、Python/Java/Vue 状态闭环、trace 透传 -> `40-cross-stack-contract.md`

这里的核心原则是：

`通用规范只定义工程底线；特定框架、Agent 编排与跨语言协作规则必须归入对应模板或跨栈规范。`

## Python 工业级规范框架

建议将 `10-python-base.md` 固定为以下 12 个模块：

1. 项目定位与适用范围
2. Python 版本与依赖管理
3. 目录结构与模块边界
4. 类型系统与静态检查
5. 配置与密钥管理
6. 异步、并发与任务执行
7. 数据建模与持久化边界
8. API 与接口契约基础规则
9. 错误处理与可靠性
10. 日志、监控与追踪
11. 测试策略
12. 安全基线与 AI 协作执行要求

这比原始的 7 个章节更完整，也更接近企业级治理框架。

## 条款风格与可执行性

为了让规范可供 AI 直接执行，所有关键规则建议采用统一体例。

### 1. 规则等级

每条规则都应标注以下等级之一：

- `MUST`
- `SHOULD`
- `MAY`

### 2. 条款模板

建议每条关键规则固定包含以下字段：

- `规则`
- `原因`
- `执行标准`
- `例外`
- `AI 执行提示`

### 3. 示例

示例条款可写为：

- `规则`：Python 服务项目 `MUST` 使用 `src` 布局。
- `原因`：避免导入路径污染，保持开发、测试与部署环境一致。
- `执行标准`：业务代码位于 `src/<package_name>/`，测试位于 `tests/`。
- `例外`：一次性临时脚本可不采用 `src` 布局，但不得演化为长期项目。
- `AI 执行提示`：创建新 Python 服务项目时，默认生成 `src/`、`tests/`、`pyproject.toml`、`.env.example`、`README.md`。

## AI 执行规则

建议在总纲或 Python 基础规范中加入独立章节，明确 AI 的执行边界：

- AI 在新建项目时 `MUST` 先识别项目模板，再生成结构。
- AI `MUST` 默认遵循规范中的默认栈，不重复发散式讨论基础选型。
- AI 在新增代码时 `MUST` 同步补齐类型、测试、配置样例与最小运行说明。
- AI `MUST NOT` 在同一项目内混入多套风格或多套基础设施方案。
- AI 在偏离默认栈时 `MUST` 说明原因、影响与替代方案。
- AI `SHOULD` 优先复用既有抽象与模板，不重复造轮子。

## 跨栈规范边界

由于后续希望将规范拆成 Python、前端、Java 三类，因此需要在设计阶段明确跨栈边界。

跨栈规范建议只处理以下内容：

- 统一错误码
- trace_id/request_id 透传
- Python/Java/Vue 间的接口契约
- SSE 事件格式
- Redis Stream 事件命名与状态流转
- 任务状态码语义
- 服务注册与发现约束，例如 Nacos 接入方式

跨栈规范不负责单一语言内部的编码细节。

## 推荐落地顺序

建议后续按以下顺序将规范正式成文化：

1. 先完成 `00-governance.md`
2. 再完成 `10-python-base.md`
3. 再写 `11-python-api-template.md`、`12-python-ai-agent-template.md`、`13-python-script-template.md`
4. 然后补 `20/21` 与 `30/31`
5. 最后收束到 `40-cross-stack-contract.md` 与 `90-project-bootstrap-checklist.md`

这样可以先确保最常用的 Python 规范和启动模板可用，再逐步扩展到其他技术栈。

## 风险与注意事项

- 如果把所有技术选型继续堆在单一文档中，规范会重新退化成“偏好清单”。
- 如果没有 `MUST/SHOULD/MAY` 分级，AI 很难判断哪些是硬约束，哪些是建议。
- 如果不区分“总纲、模板、跨栈”，后续任何新框架接入都会污染基础规范。
- 如果缺少 AI 执行章节，这份规范对 AI 的实际约束力会明显不足。

## 结论

本次设计建议将现有初版 Python 规范升级为一套“总纲 + 模板 + 跨栈契约 + 启动清单”的个人工程规范体系。

其中 Python 部分不再采用单篇大而全写法，而是拆为：

- 通用基础规范
- API 服务模板
- AI Agent 服务模板
- 脚本与数据任务模板

这样既能保留稳定的默认技术栈，也能避免简单项目被重型架构污染，并且更适合 AI 在新项目启动、代码生成、测试补齐和结构设计时直接执行。
