# goto 过期路径配置硬清理设计

## 摘要

在 `2026-04-20-goto-多拓扑路径与配置重构设计.md` 与对应实施计划完成后，`goto` 项目中仍保留一批仅用于迁移过渡的旧路径配置。这些配置项已经在 `RProperties` 中被 `@Deprecated` 标记，但仍然残留在公共 YAML 配置、少量生产代码、测试代码和说明文档中。

本次设计的目标不是继续兼容，而是执行一次“硬清理”：

- 先将受影响配置文件备份到 `goto/.backup/`
- 再删除 `RProperties` 中与多拓扑路径重构相关、且已标记 `@Deprecated` 的字段
- 同步删除 YAML 中对应配置项
- 一并修复所有仍在引用这些旧字段的生产代码、测试和当前仍有维护价值的文档
- 以“项目可编译、定向测试通过、变更范围可解释”为完成标准

## 背景

4 月 20 日的多拓扑路径重构已经引入了以下新模型：

- `RuntimeTopology`
- `LogicalPath`
- `PathTarget`
- `PathResolver`

这些抽象已经承担了 Java 本地路径与 R 远端路径的统一解析职责。理论上，以下旧字段已不应继续作为事实来源存在：

- `publicRootForR`
- `chartRootForR`
- `dataProcessRoot`
- `dataProcessRootForR`
- `chartUserForR`
- `codeRoot`
- `workingDir`
- `tempPathRootForR`
- `userNamespaceRoot`
- `excelConvertOutputDirForR`
- `imageConvertOutputDirForR`

当前如果继续保留它们，会带来三个问题：

1. 配置事实源不唯一，容易让后续维护者误判“旧字段仍然有效”
2. 测试中继续以这些字段作为“误导值”构造上下文，会掩盖真实依赖关系
3. `RProperties` 与 YAML 之间继续保留无效绑定，会让迁移状态长期停留在“半完成”

## 目标

- 删除 `RProperties` 中与多拓扑路径重构相关、且已标记 `@Deprecated` 的字段。
- 删除公共 YAML 中对应的旧配置项。
- 清理所有仍直接读取这些字段的生产代码。
- 修正所有仍依赖这些字段的测试和测试注释。
- 保留一份仓库内可追溯的配置备份，便于审计和临时回滚。
- 不影响未标记废弃的旧配置项。

## 非目标

- 不清理未标记 `@Deprecated` 的旧配置项，如 `chartRoot`、`tempPathRoot`、`excelConvertOutputDir`、`imageConvertOutputDir`。
- 不批量改写所有历史文档，仅更新当前仍用于开发、运维或回归验证的文档。
- 不重新设计多拓扑路径模型，本次只执行既定设计下的收口清理。
- 不额外扩展新的路径抽象或新增运行模式。

## 当前范围确认

### 1. 删除对象仅限已标记过期字段

本次清理对象必须同时满足两个条件：

- 与多拓扑路径重构直接相关
- 已在 `goto/common/src/main/java/com/freedom/properties/RProperties.java` 中标记 `@Deprecated`

### 2. 当前主要配置落点

根据现状扫描，这批旧配置主要集中于：

- `goto/common/src/main/resources/application-common-home.yml`
- `goto/common/src/main/resources/application-common-public.yml`

`application-common-wsl.yml` 已基本转为新模型，不作为本次主要删除对象来源。

### 3. 当前代码影响面

受影响点预计包括：

- `RProperties.java` 本身
- 少量仍直接读取这些旧字段的生产代码
- 若干通过“给旧字段设置误导值”来验证新逻辑优先级的测试
- 仍面向当前开发/运维流程的设计文档、回归文档和测试说明

## 方案比较

### 方案 A：只删 YAML，不删 Java 字段

优点：

- 改动小
- 编译风险低

缺点：

- `RProperties` 仍保留无意义字段
- 后续代码仍可能继续读取这些字段
- 无法实现真正的“硬清理”

### 方案 B：删 YAML 和 Java 字段，但不处理所有引用

优点：

- 清理力度比方案 A 大

缺点：

- 容易在编译或运行期暴露零散问题
- 会留下不一致测试和注释
- 不能作为完整收口方案

### 方案 C：完整硬清理，同时保留仓库内备份

优点：

- 配置、代码、测试、当前文档一致
- 后续维护者不会再把过期字段当成可用入口
- 备份文件留在仓库内，便于对照

缺点：

- 改动面更大，需要认真核对引用点

本次采用方案 C。

## 设计

## 一、备份策略

在删除前，先将原始配置文件复制到仓库内固定目录：

- `goto/.backup/`

备份规则：

- 保留原文件名语义
- 文件名增加日期和用途标识，避免覆盖历史备份
- 只备份本次会被修改的配置文件，不扩大范围

建议备份对象：

- `application-common-home.yml`
- `application-common-public.yml`

备份的作用是：

- 审计本次清理前的完整状态
- 在极端情况下手工比对或局部回退
- 避免依赖 Git 历史以外的额外恢复手段

## 二、配置清理策略

从公共 YAML 中删除以下旧配置项：

- `publicRootForR`
- `chartRootForR`
- `dataProcessRoot`
- `dataProcessRootForR`
- `chartUserForR`
- `codeRoot`
- `workingDir`
- `tempPathRootForR`
- `userNamespaceRoot`
- `excelConvertOutputDirForR`
- `imageConvertOutputDirForR`

删除原则：

- 只删已确认属于过期迁移配置的项
- 不因为“看起来也旧”而顺手删除未标记废弃的配置
- 删除后如果某个表达式引用链断裂，必须同步收敛到新模型或仍保留的非废弃字段

## 三、Java 硬清理策略

从 `RProperties` 中删除上述 `@Deprecated` 字段，并同步处理以下两类代码：

### 1. 生产代码

如果仍有生产代码读取这些字段，统一改为：

- 路径类读取：`PathResolver + LogicalPath + PathTarget`
- 默认目录类读取：迁移到当前仍有效的新入口，或收敛到单一兼容入口

重点不是“让字段删掉后还能凑合运行”，而是让调用方明确只依赖新模型。

### 2. 测试代码

当前部分测试通过给旧字段设置“误导值”来验证新逻辑优先级。硬清理后，这类测试要改成：

- 通过 `PathResolver` mock 提供真实路径
- 通过仍保留的非废弃字段构造必要上下文
- 如果测试目的只是“证明不再依赖旧字段”，则调整断言和注释，改为证明“路径完全来自 PathResolver”

## 四、文档清理策略

不追求清洗所有历史文档，但需要更新当前仍会被使用的文档，包括：

- 当前设计/实施计划中仍把这些字段视为兼容保留项的段落
- 当前回归清单或测试说明中提到这些旧字段作为输入条件的部分

处理原则：

- 保留历史背景，但改成过去时
- 不再把这些字段描述为“现存兼容入口”

## 五、完成标准

本次硬清理完成必须同时满足：

1. `goto/.backup/` 中已生成本次修改前的配置备份
2. `RProperties` 中相关 `@Deprecated` 字段已删除
3. 公共 YAML 中对应配置项已删除
4. 所有生产代码编译通过
5. 所有受影响测试已调整并通过
6. `git diff` 可清晰解释为“备份 + 过期配置硬清理”，不夹带无关改动

## 受影响验证建议

建议至少执行以下验证：

- `common` 模块定向测试
- `system` 模块定向测试
- 受影响的路径解析、下载、工具、清理任务相关测试
- 变更检测，确认只影响预期范围

## 风险与应对

### 风险 1：仍有隐藏引用未被发现

应对：

- 先全仓库检索字段名
- 删除字段后以编译失败作为第二层兜底

### 风险 2：测试语义和实现一起失真

应对：

- 对每个受影响测试重新明确它真正验证的行为
- 不把“删除旧字段”机械替换成“删除几行 mock”

### 风险 3：回归时难以确认删除前配置内容

应对：

- 在 `goto/.backup/` 中保留原始配置快照

## 结论

本次工作应作为多拓扑路径重构的收口步骤执行，而不是继续拖延兼容期。采用“仓库内备份 + 删除过期 YAML + 删除 `RProperties` 过期字段 + 清理所有实际引用”的硬清理方案，能够把当前项目从“旧新模型并存”推进到“旧模型正式退出”，且风险在可控范围内。
