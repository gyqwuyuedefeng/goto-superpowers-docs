# 企业信息本地查询工具设计

## 1. 背景

当前有一批存放在目录 `/mnt/f/tmp/c` 下的企业信息 JSON 文件，需要交付一套可直接给最终用户使用的本地查询软件。目标用户运行软件后，无需单独安装 Python 或数据库，即可在本机完成数据导入、查询筛选与结果导出。

经抽样确认，数据源具备以下特征：

1. 目录下当前约有 `2943` 个 `.json` 文件。
2. 抽样文件内容为企业对象数组，单文件样本长度为 `1000`。
3. 目标字段在样本中大多存在，但 `email`、`mainBusiness`、`phone` 存在少量缺失。
4. 业务上只需要保留以下字段：
   - `companyName`
   - `city`
   - `area`
   - `businessScope`
   - `email`
   - `industryNea`
   - `legalRep`
   - `mainBusiness`
   - `phone`
   - `regAddress`

本工具定位为单机本地查询工具，不是多用户系统，也不是长期运行的服务端平台。首版重点是稳定导入、快速查询、可重复更新和便捷导出。

## 2. 已确认需求

本设计基于以下已确认口径：

1. 软件定位为单机本地查询工具。
2. 首版技术方案采用 `Python + FastAPI + SQLite + PyInstaller`。
3. 数据入库时仅保留 `phone` 或 `email` 至少有一个的记录。
4. 若某条记录 `phone` 和 `email` 都为空，则直接跳过。
5. 查询能力需包含：
   - 一个总搜索框进行关键词检索
   - `city`、`area`、`industryNea` 三个筛选条件
6. 导出能力需同时支持：
   - 导出当前筛选结果
   - 导出勾选的部分记录
7. 软件交付时需同时支持以下导入方式：
   - 直接使用已存在的本地数据库查询
   - 用户重新选择目录做“清空后重建导入”
   - 用户重新选择目录做“增量更新导入”
8. 增量更新的唯一键定义为：
   - `city + area + companyName`
9. 重新导入时允许：
   - 先清空再重建
   - 直接导入并按唯一键更新已有记录

## 3. 目标

### 3.1 功能目标

1. 用户双击可执行文件后即可打开本地查询界面。
2. 用户可从指定 JSON 目录导入数据。
3. 导入时仅保留目标字段，并过滤掉无电话且无邮箱的记录。
4. 用户可按关键词和地区条件查询企业信息。
5. 用户可导出当前筛选结果或手动勾选结果。
6. 用户后续可对同一批或新一批 JSON 数据执行重建导入或增量更新导入。

### 3.2 性能与可用性目标

1. 在几百万级记录规模下，搜索仍保持可接受的本地响应速度。
2. 导入过程对用户可见，避免界面“无反馈卡死”。
3. 查询、导出、重复导入均在单机环境中稳定运行。

### 3.3 交付目标

1. 最终产物可打包为 Windows 下可直接运行的 `.exe`。
2. 最终用户无需自行安装 Python、SQLite 或其他开发依赖。
3. 软件应具备基本日志、任务状态和错误提示能力，便于现场使用与问题排查。

## 4. 非目标

本轮不包含以下内容：

1. 不做公网部署或多用户权限体系。
2. 不接入在线数据库或远程同步。
3. 不接入 AI 检索、RAG、向量库或语义搜索。
4. 不做复杂的数据清洗规则，例如电话规范化、邮箱校验纠错、企业重名高级合并。
5. 不实现复杂权限、审计、操作日志后台。
6. 不优先支持 Excel 原生 `xlsx` 导出作为唯一格式，首版导出格式以 `CSV` 为主。

## 5. 方案对比与决策

### 5.1 方案 A：FastAPI + SQLite + FTS5 + 单页前端 + PyInstaller

做法：

1. 用 Python 脚本遍历 JSON 文件并导入 SQLite。
2. 用 SQLite 普通表保存结构化字段。
3. 用 SQLite `FTS5` 建立全文索引，支撑关键词搜索。
4. 用 FastAPI 暴露本地查询、导入和导出接口。
5. 用简单单页前端承载查询表格和导入交互。
6. 用 PyInstaller 打包为 Windows 可执行文件。

优点：

1. 开发效率高，和现有需求匹配度最高。
2. 对 JSON 处理、批量导入、本地 API 和打包交付都比较成熟。
3. 后续加筛选、增量更新、导出都容易扩展。
4. `FTS5` 能显著改善几百万级文本检索体验。

缺点：

1. 首次导入时间可能较长，需要专门设计后台任务与进度反馈。
2. 本地服务启动后仍依赖浏览器作为界面载体。

### 5.2 方案 B：FastAPI + SQLite 普通索引 + LIKE 查询

优点：

1. 实现最简单。
2. 数据库结构更直观。

缺点：

1. 几百万级数据下，多字段模糊搜索性能不稳。
2. `businessScope`、`mainBusiness`、`regAddress` 等长文本字段查询体验会明显下滑。

### 5.3 方案 C：PySide6 桌面端 + SQLite

优点：

1. 更接近传统桌面软件交付形态。
2. 无需依赖浏览器承载界面。

缺点：

1. UI 开发与打包复杂度明显更高。
2. 对当前需求而言收益不明显，开发周期更长。

### 5.4 最终决策

采用 **方案 A：FastAPI + SQLite + FTS5 + 单页前端 + PyInstaller**。

原因：

1. 该方案与用户期望的技术栈一致。
2. 在开发成本、交付速度和后续扩展性之间平衡最好。
3. `FTS5` 能避免首版退化为低性能的 `LIKE` 模糊搜索工具。

## 6. 总体架构

软件拆分为以下 4 个模块：

### 6.1 数据导入模块

职责：

1. 扫描用户选定目录下的全部 `.json` 文件。
2. 逐文件读取企业数组数据。
3. 抽取目标字段并过滤无效记录。
4. 执行“清空后重建导入”或“增量更新导入”。
5. 维护导入任务进度和统计结果。

### 6.2 本地数据库模块

职责：

1. 存储筛选查询所需的结构化企业数据。
2. 维护唯一键索引，支撑增量更新。
3. 维护全文索引，支撑关键词检索。
4. 保存导入任务状态、应用元信息与必要日志。

### 6.3 本地服务模块

职责：

1. 提供查询、导入、进度查询、导出等本地 HTTP 接口。
2. 负责应用启动、数据库初始化、静态页面分发。
3. 管理导入后台任务，避免主线程阻塞。

### 6.4 单页前端模块

职责：

1. 提供搜索框、筛选项、结果表格和导出入口。
2. 展示导入状态、统计信息和错误提示。
3. 支持勾选导出与当前筛选结果导出。

## 7. 数据模型设计

建议采用“主表 + 全文索引表 + 导入任务表”的结构。

### 7.1 主表 `companies`

字段建议如下：

1. `id`：自增主键
2. `unique_key`：由 `city + area + companyName` 规范化后生成
3. `company_name`
4. `city`
5. `area`
6. `business_scope`
7. `email`
8. `industry_nea`
9. `legal_rep`
10. `main_business`
11. `phone`
12. `reg_address`
13. `source_file`：来源 JSON 文件名
14. `created_at`
15. `updated_at`

约束与索引：

1. 在 `unique_key` 上建立唯一索引。
2. 在 `city`、`area`、`industry_nea` 上建立普通索引，优化筛选。

### 7.2 全文索引表 `companies_fts`

使用 SQLite `FTS5`，纳入以下字段：

1. `company_name`
2. `legal_rep`
3. `phone`
4. `email`
5. `business_scope`
6. `main_business`
7. `reg_address`

设计目标：

1. 一个搜索框覆盖企业名、法人、联系方式、经营范围、主营业务、注册地址等常见文本查询场景。
2. 将结构化筛选与全文搜索分离，避免所有字段都走慢查询。

### 7.3 导入任务表 `import_jobs`

字段建议如下：

1. `job_id`
2. `mode`
3. `status`
4. `source_dir`
5. `total_files`
6. `processed_files`
7. `inserted_count`
8. `updated_count`
9. `skipped_count`
10. `failed_files`
11. `error_message`
12. `started_at`
13. `finished_at`

用途：

1. 前端轮询导入进度。
2. 导入结束后展示成功、更新、跳过和失败统计。
3. 支撑失败排查，而不是只返回“导入失败”。

## 8. 导入与更新规则

### 8.1 文件扫描规则

1. 扫描用户选定目录下全部 `.json` 文件。
2. 默认递归遍历子目录，避免目录结构变化导致漏导。
3. 若目录下没有 JSON 文件，明确提示用户。

### 8.2 记录抽取规则

每条企业记录仅抽取以下字段：

1. `companyName`
2. `city`
3. `area`
4. `businessScope`
5. `email`
6. `industryNea`
7. `legalRep`
8. `mainBusiness`
9. `phone`
10. `regAddress`

### 8.3 跳过规则

以下记录直接跳过，不入库：

1. `phone` 和 `email` 同时为空。
2. `city`、`area`、`companyName` 任一为空，无法稳定生成唯一键。

说明：

1. 首版不尝试为唯一键缺失记录设计备用合并策略。
2. 这是为了保证“增量更新”语义可靠，避免重复或错误覆盖。

### 8.4 值规范化规则

首版只做轻量规范化：

1. 字符串去首尾空白。
2. 空字符串视为空值。
3. 唯一键生成前对 `city`、`area`、`companyName` 执行去空白与基础规范化。

首版不做：

1. 电话格式统一化。
2. 邮箱合法性强校验。
3. 地址结构拆分。

### 8.5 清空后重建导入

执行流程：

1. 清空 `companies` 表。
2. 清空并重建 `companies_fts` 索引。
3. 重新扫描目录并导入全部合法记录。
4. 更新任务统计与最后导入时间。

适用场景：

1. 用户确认需要用选定目录完全覆盖本地数据。
2. 原有数据库疑似不一致或希望确保全量同步。

### 8.6 增量更新导入

执行流程：

1. 扫描目录并读取记录。
2. 基于 `city + area + companyName` 生成唯一键。
3. 若唯一键已存在，则更新业务字段与 `updated_at`。
4. 若唯一键不存在，则插入新记录。
5. 每批事务提交后同步维护 FTS 索引。

更新字段范围：

1. `company_name`
2. `city`
3. `area`
4. `business_scope`
5. `email`
6. `industry_nea`
7. `legal_rep`
8. `main_business`
9. `phone`
10. `reg_address`
11. `source_file`
12. `updated_at`

### 8.7 导入执行策略

为避免大批量数据导入时内存膨胀或前端无反馈，建议：

1. 按文件流式处理，不把全量数据一次性读入内存。
2. 按固定批次执行数据库写入与事务提交。
3. 后台线程或后台任务执行导入，主线程只返回任务编号。
4. 前端定时查询任务进度。

## 9. 查询与搜索设计

### 9.1 查询入口

首页提供一个总搜索框，覆盖以下字段的关键词检索：

1. `companyName`
2. `legalRep`
3. `phone`
4. `email`
5. `businessScope`
6. `mainBusiness`
7. `regAddress`

### 9.2 筛选项

首页额外提供以下筛选：

1. `city`
2. `area`
3. `industryNea`

### 9.3 查询逻辑

1. 关键词为空时，仅按筛选条件查询结构化数据。
2. 关键词不为空时，先走 `FTS5` 检索候选结果，再叠加 `city`、`area`、`industryNea` 筛选。
3. 查询结果默认分页返回，避免一次性加载过大结果集。

### 9.4 排序策略

首版采用稳定、简单的排序策略：

1. 有关键词时按全文匹配优先。
2. 同等匹配下按企业名称排序。

首版不做：

1. 复杂加权排序。
2. 拼音首字母排序优化。
3. 自定义多字段排序器。

## 10. 导出设计

### 10.1 导出类型

需同时支持两种导出：

1. 导出当前筛选结果
2. 导出勾选结果

### 10.2 导出格式

首版导出格式使用 `CSV`。

原因：

1. 实现稳定。
2. 依赖简单。
3. Excel 可直接打开。
4. 对几百万数据的流式写出支持更直接。

### 10.3 导出规则

1. 导出当前筛选结果时，由后端按当前关键词与筛选条件重新查询全量命中集后输出 CSV。
2. 导出勾选结果时，由前端提交勾选记录 ID 列表，后端按 ID 导出。
3. 导出文件名应包含时间戳，避免覆盖历史导出结果。
4. 导出字段顺序与表格展示顺序保持一致。

### 10.4 大结果集处理

1. 当筛选结果较大时，由后端生成文件并返回下载。
2. 前端在发起导出前给出“数据量较大可能需要较长时间”的提示。
3. 首版允许大结果导出，但不额外实现复杂断点续导或导出任务中心。

## 12. 接口设计

建议提供以下本地接口：

### 12.1 `GET /api/search`

用途：

1. 分页查询企业数据。

核心参数：

1. `keyword`
2. `city`
3. `area`
4. `industryNea`
5. `page`
6. `pageSize`

### 12.2 `POST /api/import/start`

用途：

1. 启动导入任务。

核心参数：

1. `sourceDir`
2. `mode`

其中 `mode` 为：

1. `rebuild`
2. `incremental`

### 12.3 `GET /api/import/status`

用途：

1. 查询导入任务进度与统计结果。

核心参数：

1. `jobId`

### 12.4 `GET /api/stats`

用途：

1. 获取首页概览信息，例如总记录数、最后导入时间和最近任务状态。

### 12.5 `POST /api/export`

用途：

1. 导出当前筛选结果或勾选结果。

核心参数：

1. `mode`
2. `keyword`
3. `city`
4. `area`
5. `industryNea`
6. `ids`

其中 `mode` 为：

1. `current-filter`
2. `selected-rows`

## 13. 异常处理

### 13.1 导入前异常

1. 所选目录不存在或不可访问时，明确提示。
2. 所选目录下无 JSON 文件时，明确提示“未发现 JSON 文件”。

### 13.2 导入中异常

1. 个别 JSON 文件损坏或编码异常时，记录失败文件并继续处理后续文件。
2. 若已有导入任务在执行，则拒绝启动新任务并提示“当前已有导入任务执行中”。
3. 若数据库写入异常，则终止当前任务并记录错误信息。

### 13.3 查询异常

1. 数据库未初始化或当前无数据时，页面提示先导入数据。
2. 查询参数非法时，返回明确错误而不是空白页面。

### 13.4 导出异常

1. 无匹配结果时，提示当前没有可导出的数据。
2. 勾选导出但未选择记录时，提示先勾选记录。

## 14. 打包与运行方式

### 14.1 交付内容

最终交付建议包括：

1. Windows 可执行文件 `企业信息查询工具.exe`
2. 应用工作目录
3. SQLite 数据库文件
4. 导出目录
5. 简短使用说明

### 14.2 启动行为

用户双击 `.exe` 后，程序执行以下动作：

1. 初始化应用工作目录与数据库。
2. 启动本地 FastAPI 服务。
3. 自动打开浏览器进入查询页面。
4. 若浏览器未自动打开，则明确显示访问地址。

### 14.3 运行目录建议

建议把以下文件都放在应用工作目录中：

1. `app.db`
2. `exports/`
3. `logs/`
4. `config.json`

这样便于：

1. 用户保留本地导入结果
2. 多次启动复用数据库
3. 排查导入失败或导出问题

## 15. 测试策略

### 15.1 ETL 单元测试

1. JSON 记录字段抽取是否正确。
2. `phone` 和 `email` 同时为空时是否跳过。
3. `city + area + companyName` 唯一键生成是否稳定。
4. 空字符串规范化是否符合预期。

### 15.2 数据库集成测试

验证内容：

1. 建表与索引初始化。
2. 唯一键冲突下的增量更新行为。
3. `FTS5` 全文搜索结果正确性。
4. `city`、`area`、`industryNea` 组合筛选正确性。
5. 分页查询正确性。

### 15.3 接口测试

验证内容：

## 16. 验收标准

以下条件全部满足时，可视为首版交付达标：

3. 以 `city + area + companyName` 作为唯一键完成增量更新。
4. 支持关键词搜索。
5. 支持 `city`、`area`、`industryNea` 筛选。
6. 支持导出当前筛选结果。
7. 支持导出勾选结果。
8. 支持“清空后重建导入”和“增量更新导入”。
9. 能打包为用户可直接运行的 Windows 程序。

## 17. 实施前备注

1. 当前数据量从样本推断可能达到数百万级，因此实现阶段必须优先保证导入分批写入与查询索引设计，不可把本工具按“小规模脚本”处理。
2. 首版应优先追求稳定导入、稳定搜索和稳定导出，不应过早引入不必要的高级能力。
3. 在进入实现计划前，仍需进一步明确项目目录落点、代码组织方式、前端技术选择和 Windows 打包细节，但这些属于实施计划阶段处理的问题，不影响当前设计定稿。
