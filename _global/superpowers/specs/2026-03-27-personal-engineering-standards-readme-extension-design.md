# personal-engineering-standards README 与扩展规范设计

## 1. 背景

当前 `personal-engineering-standards/` 已经形成一组正式规范分册：

- `00-governance.md`
- `10-python-base.md`
- `11-python-api-template.md`
- `12-python-ai-agent-template.md`
- `13-python-script-template.md`
- `20-frontend-base.md`
- `21-frontend-vue-template.md`
- `30-java-base.md`
- `31-java-spring-template.md`
- `40-cross-stack-contract.md`
- `90-project-bootstrap-checklist.md`

但目录本身还缺少两个关键入口层文档：

- 缺少一个项目级 `README.md`，无法让使用者快速理解“这套规范是什么、当前范围是什么、先看什么文档、怎么开始使用”
- 缺少一个单独的“后续扩展方式规范”文档，无法约束未来如何新增模板、补充 Vue 3、何时启用 Java、如何修改编号和边界

本次只补齐这两个文档，不调整现有 11 份正式规范正文结构。

## 2. 目标

### 2.1 `README.md`

让首次进入该目录的人可以在 1 分钟内回答以下问题：

- 这套规范是什么
- 当前正式生效范围是什么
- 哪些内容已经启用，哪些只是预留
- 11 份文档分别负责什么
- 使用顺序是什么
- 如果要用这套规范启动新项目，应从哪里开始

### 2.2 `EXTENDING.md`

让未来维护者或 AI 在扩展这套规范时可以回答以下问题：

- 新内容应该放到哪一层：`00 / base / template / 40 / 90`
- 什么时候应该新增模板，什么时候只记例外
- 如何补充 Vue 3
- 什么时候才能正式纳入 Java
- 如何分配编号与命名
- 修改默认栈时要同步检查哪些文档

## 3. 范围

### 3.1 本轮范围

目标目录固定为仓库根目录下的 `personal-engineering-standards/`。

新增两个文件：

- `personal-engineering-standards/README.md`
- `personal-engineering-standards/EXTENDING.md`

### 3.2 明确不在本轮范围

- 不重写现有 11 份规范正文
- 不新增 Vue 3 模板正文
- 不正式启用 Java 模板
- 不调整现有编号体系

## 4. 关键设计决策

### 4.1 采用 `README.md + EXTENDING.md`

不把扩展规则塞进 `README.md` 的原因：

- `README.md` 应保持为入口文档，面向使用者
- 扩展规范更像维护者规则，应单独成文
- 单独文件更利于未来 AI 在“要扩展规范本身”时直接引用

### 4.2 不把扩展规范写成编号正文

不采用 `99-extension-governance.md` 之类编号分册的原因：

- 会与面向项目使用的正式规范分册混淆
- 容易让 AI 把它当作项目规范正文的一部分，而不是维护手册
- `EXTENDING.md` 这个命名更直接表达“这是维护和扩展这套规范的方法”

### 4.3 统一链接规范

两份新文档中的链接规则固定为：

- 在 `README.md` 与 `EXTENDING.md` 正文中，指向本目录中的其他规范文件时，一律使用 `./<file>.md`，这是唯一默认写法
- 只有在描述“该目录在仓库中的位置”时，才使用 `personal-engineering-standards/<file>.md`
- `MUST NOT` 使用机器绝对路径

### 4.4 状态标记定义

文档地图中的状态标记含义固定为：

- `ACTIVE`：当前已纳入默认工程规范执行范围，可被 AI 与人工直接作为默认标准使用
- `RESERVED`：当前仅保留编号与预留位，不属于当前默认工程规范执行范围；可以被引用说明其“未启用”状态，但不得作为默认模板执行

默认栈的单一事实源固定为：

- 治理层定义，见 `./00-governance.md`
- 具体技术栈定义，见对应的 `base/template` 文档
- `README.md` 只做导航与总结，不重新发明新的默认栈定义

## 5. 文件职责

### 5.1 `README.md`

应包含以下内容：

1. 项目简介
2. 当前正式生效范围
3. 当前未启用但预留的部分
4. 文档地图
5. 推荐阅读顺序
6. 新项目使用方式
7. 与 `EXTENDING.md` 的关系说明

最小结构固定为以下标题，标题文字必须保持一致：

1. `# personal-engineering-standards`
2. `## 1. 项目定位`
3. `## 2. 当前生效范围`
4. `## 3. 文档地图`
5. `## 4. 推荐阅读顺序`
6. `## 5. 如何使用这套规范`
7. `## 6. 后续扩展入口`

写作要求：

- 以入口说明为主，不重复展开各分册正文
- 明确写出“当前默认前端 = Vue 2，Vue 3 后补；Java 暂不启用”
- 文档地图中的每一项统一写成：`文件名 + 一行职责 + 状态标记（ACTIVE / RESERVED）`
- 文档中的路径引用使用仓库相对路径，如 `./00-governance.md`，不使用机器绝对路径
- 全文使用中文为主，与现有规范正文保持一致
- 让 AI 能把它作为目录级导航文档使用

状态标记固定如下：

- `30-java-base.md`：`RESERVED`
- `31-java-spring-template.md`：`RESERVED`
- 其余当前 9 份正式分册：`ACTIVE`

文档地图条目格式进一步固定为：

- 一条只写一个文件
- 采用单行表达
- 职责描述控制在一句话内，不展开多级解释
- 不在职责描述中重复写技术细节与例外规则

示例：

- `./00-governance.md`：整套规范的上位治理文档，定义继承关系、规则等级与例外管理。`ACTIVE`
- `./30-java-base.md`：Java 方向的预留编号，当前版本暂不启用。`RESERVED`

### 5.2 `EXTENDING.md`

应包含以下内容：

1. 本文定位
2. 何时允许扩展或修改规范
3. 内容归位规则
4. 编号与命名规则
5. 新增模板的判断标准
6. Vue 3 补充方式
7. Java 启用方式
8. 修改默认栈时的同步检查清单
9. AI 在扩展规范时的执行要求

最小结构固定为以下标题，标题文字必须保持一致：

1. `# EXTENDING`
2. `## 1. 本文定位`
3. `## 2. 何时允许扩展或修改`
4. `## 3. 内容归位规则`
5. `## 4. 编号与命名规则`
6. `## 5. 新增模板的判断标准`
7. `## 6. Vue 3 的补充方式`
8. `## 7. Java 的启用方式`
9. `## 8. 修改默认栈时的同步检查清单`
10. `## 9. AI 扩展执行要求`

写作要求：

- 以“如何扩展这套规范本身”为主题，不写项目启动步骤
- 与 `00-governance.md` 保持一致，但不重复抄写整套治理正文
- 更偏维护规则和扩展流程，而不是重新定义规范体系
- 文中的路径引用使用仓库相对路径，不使用机器绝对路径
- 全文使用中文为主，与现有规范正文保持一致
- 本文只解释扩展机制，不新增正式规范条款
- `Vue 3 的补充方式` 与 `Java 的启用方式` 只写“何时允许补充、补充前必须满足什么门槛、需要同步哪些文档”，不写模板正文与项目初始化步骤

各章节至少要回答以下问题：

- `## 1. 本文定位`
  - 本文服务谁
  - 本文与 `README.md`、`00-governance.md` 的关系是什么
- `## 2. 何时允许扩展或修改`
  - 哪些情况才值得改规范本身
  - 哪些情况只应记录项目例外，不应修改默认规范
- `## 3. 内容归位规则`
  - 新内容应落到 `00 / base / template / 40 / 90 / README / EXTENDING` 的哪一层
  - 判断一条规则是否放错位置的标准是什么
- `## 4. 编号与命名规则`
  - 新分册如何编号
  - 何时可以复用现有编号，何时必须新增编号
  - 文件名如何保持与现有体系一致
- `## 5. 新增模板的判断标准`
  - 什么情况下允许新增模板
  - 什么情况下应回收到现有 base 或 template
- `## 6. Vue 3 的补充方式`
  - 只有在什么条件下才能正式补 Vue 3
  - 补充前必须同步检查哪些现有文档
  - 本节只写准入条件与同步检查，不写 Vue 3 模板正文
- `## 7. Java 的启用方式`
  - 只有在什么条件下才能把 `30/31` 从 `RESERVED` 转为 `ACTIVE`
  - 启用前必须同步检查哪些现有文档
  - 本节只写准入条件与同步检查，不写 Java / Spring 模板正文
- `## 8. 修改默认栈时的同步检查清单`
  - 修改默认技术栈时，至少要回看哪些文件
  - 哪些变化只改单文件，哪些变化必须联动多文件
- `## 9. AI 扩展执行要求`
  - AI 在扩展这套规范本身时必须先做什么
  - AI 不能做什么

## 6. 验收清单

当 `README.md` 与 `EXTENDING.md` 同时满足以下条件时，视为符合本 spec：

- `README.md` 能在不阅读正文细节的情况下解释这套规范当前是什么、当前启用什么、从哪里开始使用
- `README.md` 的文档地图对 11 个分册全部给出单行职责与状态标记
- `EXTENDING.md` 能约束未来新增模板、补 Vue 3、启用 Java、修改默认栈时的归位与同步检查方式
- 两份文档都只做导航与扩展机制说明，不新增正式规范条款
- 两份文档都使用中文与相对路径，不出现机器绝对路径

## 7. 内容边界

### 6.1 `README.md` 不应包含

- 大段重复 `00-governance.md` 的条款正文
- 各模板的详细技术规范
- 未来扩展的完整操作细则
- 新的正式规范要求

### 6.2 `EXTENDING.md` 不应包含

- 项目使用者的快速上手说明
- `90-project-bootstrap-checklist.md` 中已有的启动步骤
- 某一技术模板的完整正文规则复制
- 新的正式规范要求

## 8. 预期结果

完成后，这个目录会形成三层结构：

- 分册正文：11 份规范文件
- 入口层：`README.md`
- 维护扩展层：`EXTENDING.md`

这样既能服务“使用这套规范的人”，也能服务“未来继续扩展这套规范的人或 AI”。
