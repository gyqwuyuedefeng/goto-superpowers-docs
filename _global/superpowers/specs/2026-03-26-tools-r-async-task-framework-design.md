# 工具页 R 异步任务框架设计

## 背景

当前工具页里至少有 3 条与文件生成/下载相关的工具链：

- `goto-web/src/views/goto/toolsBox/type/excelFormatConvert.vue`
- `goto-web/src/views/goto/toolsBox/type/imageConvert.vue`
- `goto-web/src/views/goto/toolsBox/type/tool.vue`

其中它们的执行方式和下载方式并不统一：

- Excel 格式转换已经接入 `CommonTaskThreadPool + R source/param` 机制。
- 图片格式转换当前仍通过 shell 脚本 `fileConvert.sh` 进行转换。
- 合并图片下载当前仍是“点击下载时同步生成文件并立即返回”。

这会带来几个问题：

- 工具页用户体验不一致，有的先执行任务再下载，有的点击即同步生成。
- 后端 R 工具接入方式不一致，难以维护，也不利于后续扩展新工具。
- 下载结果的元数据存储、任务状态回传、错误处理缺乏统一约定。

本次设计目标不是只修复一个工具，而是建立一套可以持续承载后续工具接入的“R 异步任务流程骨架”。

## 目标

- 将图片格式转换改造为基于 `CommonTaskThreadPool + R source/param` 的统一异步任务。
- 将合并图片下载改造为“提交异步任务 -> WebSocket 通知完成 -> 再下载结果”的统一交互。
- 保持 Excel 格式转换现有能力不倒退，并将其纳入统一流程约定。
- 为后续新增工具预留统一接入规范，避免继续复制分散实现。

## 非目标

- 本次不把所有工具强行合并为一个“大一统”任务数据模型。
- 本次不重写现有 WebSocket 通知协议，只复用现有 `noticeType/status` 语义。
- 本次不做无关的 UI 重构，只改与任务提交、状态展示、结果下载相关的交互。

## 范围

本次纳入统一设计的工具包括：

- Excel 格式转换
- 图片格式转换
- 合并图片导出

其中重点改造对象为：

- 图片格式转换：由 shell 脚本切换为统一 R 任务机制
- 合并图片导出：由同步下载切换为异步任务生成

## 总体设计

### 设计原则

统一的是流程骨架，不是业务模型。

也就是说，后续每个工具仍然保留自己的：

- 请求 DTO
- 任务数据模型
- 任务 handler
- 结果 stage
- 下载 specifier

但它们都必须复用同一套流程约定：

1. 前端提交任务，立即获得 `taskId` 和 `processing` 状态
2. 服务层完成参数校验、输入落盘、输出目录规划、任务投递
3. handler 只负责声明 R `source` 与函数调用参数
4. 统一线程池执行 R 任务
5. stage 在成功后校验结果文件并写入 Redis
6. 前端通过 WebSocket 接收状态变更
7. 用户在任务成功后再点击下载

### 不采用“大一统工具任务模型”的原因

虽然当前只有 3 个工具，但它们的输入形态已经不同：

- Excel 格式转换是表格数据重塑
- 图片格式转换是单文件转单文件
- 合并图片导出是画布元素合成并输出多格式产物

如果现在强行统一成一个 `ToolTask` 上帝对象，字段会快速膨胀，后续维护风险更高。因此本次选择“统一流程约定 + 独立业务模型”的方式。

## 架构与职责划分

### 1. 服务层职责

每个工具对应的 service 负责：

- 校验提交参数
- 为任务创建 `taskId`
- 将前端输入转换为 R 可消费的输入资源
- 规划输入/输出目录
- 构造 `CommonTask<T>`
- 通过既有异步投递机制提交任务
- 立即返回 `taskId/status=processing`

服务层不负责：

- 直接执行 R
- 在提交阶段写结果 Redis
- 在提交阶段生成最终下载文件

### 2. Handler 职责

每个工具的 handler 统一转向声明式模式：

- 设置 R 所需 `source`
- 构造函数调用字符串 `param`
- 将任务交给 `CommonTaskThreadPool`

handler 不再负责：

- 自己管理外部 shell 执行
- 自己写 Redis 结果
- 自己维护同步下载逻辑

### 3. Stage 职责

每个工具对应自己的 stage，职责固定为：

- 在任务成功后校验输出文件存在且非空
- 提取可下载所需元数据
- 将标准结果结构写入 Redis

可抽一个通用父类或通用辅助方法，统一处理结果校验和 Redis 存储的公共逻辑。

### 4. 下载职责

下载链统一约定为：

- 前端仅持有 `taskId`
- 下载时通过 `businessType + taskId` 获取下载信息
- 下载指定器只负责读取已生成文件，不再承担同步生成职责

这意味着“生成”与“下载”彻底解耦。

## 图片格式转换设计

### 当前问题

当前图片转换链路中：

- service 提交 `ImageConvertTask`
- handler 落盘输入文件
- handler 调 shell 脚本 `fileConvert.sh`
- handler 自己写 Redis

这导致图片转换没有复用现有统一 R 任务机制。

### 新设计

图片格式转换仍保留独立的任务类型 `IMAGE_CONVERT`，但执行方式改为：

- service 负责接收 `inputFormat/outputFormat/fileName/fileData/commonNoticeUUID`
- service 将 Base64 输入落到任务目录中的临时输入文件
- service 计算 R 可访问的输入路径与输出路径
- service 组装 `ImageConvertTask`
- handler 仅设置：
  - `source("/home/goto/host/Software/goto/Code/Tools/fileConvert.R")`
  - `fileConvert(inputFile, outputFile, dpi = 300)`
- handler 通过 `CommonTaskThreadPool` 提交统一执行
- `ImageConvertTaskStage` 在成功后将结果元数据写入 Redis

### 参数约定

本次采用最小接口：

`fileConvert(inputFile, outputFile, dpi = 300)`

其中：

- `inputFile`：R 可访问的输入文件路径
- `outputFile`：R 可访问的输出文件路径
- `dpi`：固定为 `300`

本次不增加前端可配置 dpi。

### 前端交互

图片转换页面保持现有交互心智：

- 选择格式
- 上传文件
- 提交转换
- 等待 WebSocket 通知
- 成功后点击下载

因此前端主要变更应集中在：

- 与后端新任务元数据结构对齐
- 确保成功后按 `taskId` 下载

## 合并图片异步化设计

### 当前问题

当前合并图片导出链路中：

- 前端在 `tool.vue` 点击下载
- 同步进入下载链
- `MergeImageDownload` 在下载时临时创建资源、调用 R、压缩文件、返回下载

这导致：

- 用户等待时间集中在下载点击后
- 前端无法明确区分“生成中”和“下载中”
- 后端生成与下载强耦合

### 新设计

新增独立任务类型，例如 `MERGE_IMAGE`，采用独立 task、handler、stage。

流程改为：

- 前端提交 `typeList/imageList/fileName/commonNoticeUUID`
- service 校验参数，并整理图片输入资源
- service 规划任务目录与最终输出目录
- service 构造 `CommonTask<MergeImageTask>` 并异步投递
- handler 负责构造 R 所需的 `imageList` 参数和合并函数调用
- stage 成功后写 Redis 结果
- 前端收到成功通知后，再调用统一下载流程下载已生成文件

### 现有逻辑复用策略

`MergeImageDownload` 中已有的核心业务逻辑应被拆分复用，包括：

- 坐标从左上角转左下角
- 计算全局边界
- 计算归一化 `lb/rt`
- 处理上传来源与分析结果来源的文件路径
- 拼装 R 中的 `imageList <- list(...)`

这些逻辑应迁移到新的异步任务生成链中，避免继续放在同步下载类里。

### 下载链调整

改造后，`MergeImageDownload` 不应继续承担“生成文件”职责。

它应被简化为以下两种之一：

- 仅作为下载读取器，通过 `taskId` 找到已生成产物并下载
- 被新的 merge-image 下载 specifier 替代

总原则是：下载阶段只读结果，不再触发实际生成。

### 前端交互

`tool.vue` 的“导出图像”流程调整为：

1. 用户打开下载设置
2. 点击确认后不是立即下载，而是提交合并任务
3. 页面进入处理中状态
4. WebSocket 通知成功后显示“下载结果”
5. 用户点击下载结果获取文件

这样与图片转换页面保持一致。

## 扩展性设计

### 统一流程骨架

为支持未来继续新增工具，本次需建立统一接入规范：

1. 新工具必须有独立 service 提交入口
2. 新工具必须有独立 task data
3. 新工具必须通过 handler 声明 `source + param`
4. 新工具必须有 stage 负责结果校验和 Redis 存储
5. 新工具必须通过 `businessType + taskId` 下载

### 可复用封装点

建议抽取以下可复用能力：

- 工具任务结果的统一 Redis 结构
- 工具任务输出目录规划工具
- stage 公共结果校验与写 Redis 辅助逻辑
- 前端工具页异步任务状态管理公共逻辑
- 新下载业务类型接入规范

### Redis 结果结构建议

不同工具可保留自己的额外字段，但公共字段至少统一包含：

- `taskId`
- `businessType`
- `fileName`
- `outputPath`
- `createTime`

按需附加：

- `inputFormat`
- `outputFormat`
- `typeList`
- `scenario`

### 目录规划建议

所有工具统一按 `taskId` 建立输出目录，避免散落临时文件。

建议约定：

- 输入临时目录：用于接收上传文件或中间资源
- 输出结果目录：用于最终可下载产物
- 一个任务一个目录

这样便于：

- 清理
- 排错
- 按任务追踪产物

## 状态回传设计

本次不新增新的 WebSocket 协议，而是复用现有状态语义。

前端统一处理：

- 提交失败
- 排队中
- 执行中
- 执行成功
- 执行失败
- 已拒绝
- 已取消

这样图片转换与合并图片可以共享一套前端状态机思路。

## 错误处理

错误处理分三层：

### 1. 提交前校验

由 service 完成：

- 必填参数是否缺失
- 格式是否合法
- 文件名是否合法
- 合并图片时图片数量是否足够

### 2. 执行前校验

由 handler 或其前置准备逻辑完成：

- 输入文件是否存在
- 目录是否创建成功
- R source 文件是否可访问
- R 调用参数是否完整

### 3. 执行后校验

由 stage 完成：

- 输出文件是否存在
- 输出文件是否为空
- Redis 是否写入成功

如果执行成功但结果文件不存在，必须按失败态回传，不能让前端拿到不可下载的成功结果。

## 测试策略

至少覆盖以下测试范围：

### 图片格式转换

- 提交后能够正确构造任务
- handler 能正确生成 `fileConvert(inputFile, outputFile, dpi = 300)`
- 成功后 Redis 元数据可被下载链正确读取
- 失败时前端能收到失败状态

### 合并图片任务

- 上传图片和分析结果图片都能正确进入任务输入
- 坐标计算与归一化结果正确
- R 参数中的 `imageList` 拼装正确
- 成功后结果能被下载
- 失败路径能正确中止并回传错误

### 框架层

- 通用 stage/Redis 写入辅助逻辑工作正常
- 统一目录规划符合预期
- 新工具可以按规范增量接入

## 风险与约束

### 风险

- 合并图片现有逻辑较多，迁移时容易遗漏同步下载时代的隐式行为
- 图片转换从 shell 切到 R 后，输出格式能力完全依赖 `fileConvert.R`
- 如果 Redis 结果结构变更不受控，下载链可能出现兼容问题

### 约束

- 必须兼容现有 WebSocket 通知语义
- 必须兼容现有基于 `taskId` 的下载方式
- 必须保证旧工具页的基本使用路径不被破坏

## 实施建议

建议按以下顺序落地：

1. 先抽公共约定和可复用结果写入逻辑
2. 改造图片格式转换为统一 R 任务
3. 改造合并图片为独立异步任务
4. 最后统一前端工具页的任务状态与下载交互

这样可以把风险拆开，并保留逐步验证空间。

## 预期结果

完成后，工具页相关能力将形成统一模式：

- Excel 格式转换：统一异步 R 任务
- 图片格式转换：统一异步 R 任务
- 合并图片导出：统一异步 R 任务

并且后续新增工具只需按相同骨架接入，而不需要再发明新的执行与下载链路。
