# DeepSeek 文档审核循环设计

## 背景

当前 `superpowers:brainstorming` 在写完 spec 后只做 controller 自检，然后要求用户查看确认。`superpowers:writing-plans` 和 `hybrid-writing-plans` 在写完实施计划后也主要依赖自检和用户/后续执行阶段的判断。这个流程能保证人工门禁，但 spec 或计划中的缺口会更晚暴露，尤其是 Hybrid 工作流里 DeepSeek 实施器不能依赖当前对话历史，文档质量直接决定后续执行质量。

现有仓库已经具备可复用基础：

- `/home/gyq/.local/bin/codex-dp-exec` 可以把完整 prompt 交给 DeepSeek 侧 Codex 执行，并持久化 run 状态、日志和最终报告。
- `skills/brainstorming/spec-document-reviewer-prompt.md` 已有 spec 文档审核标准。
- `skills/writing-plans/plan-document-reviewer-prompt.md` 已有实施计划审核标准。
- `hybrid-subagent-driven-development` 已经证明 controller + DeepSeek implementer 的分工可行。

本设计把 DeepSeek 用作独立文档 reviewer：写完 spec 或实施计划后自动调用 DeepSeek 审核，controller 读取反馈并判断是否属实。属实则修改文档后再次审核，不属实则记录忽略理由并再次审核，直到审核通过。

## 目标

- 将 `codex-dp-exec` 源文件纳入 `/mnt/g/Obsidian/skill/superpowers` 版本控制，并保留 `/home/gyq/.local/bin/codex-dp-exec` 兼容入口。
- 新增 `deepseek-review-spec` 和 `deepseek-review-plan` 两个脚本，复用 `codex-dp-exec` 调用 DeepSeek 审核文档。
- 修改 `brainstorming` 的 spec 写入后流程：controller 自检后自动运行 DeepSeek spec 审核循环，通过后再进入用户 review gate。
- 修改普通 `writing-plans` 和 `hybrid-writing-plans` 的计划写入后流程：controller 自检后自动运行 DeepSeek plan 审核循环，通过后再进入执行方式选择。
- 确保 DeepSeek 每次审核都获得完整上下文，不能只看到单个文档片段导致误判。
- 确保 controller 不盲从 DeepSeek 反馈，必须判断反馈是否属实后再修改。

## 非目标

- 不让 DeepSeek 替代用户最终确认。spec 通过 DeepSeek 审核后，仍保留用户 review gate。
- 不让 DeepSeek 修改 spec、计划或代码。DeepSeek reviewer 只读上下文并输出审核报告。
- 不新增替代 `codex-dp-exec` 的执行器。两个审核脚本只是薄封装。
- 不改变 `subagent-driven-development` 原版代码执行流程。
- 不把所有文档审核问题都视为阻塞。脚本和 skill 只要求阻塞真实会影响规划或实施的问题。

### 审核脚本职责

`deepseek-review-spec` 输入：

- `--spec <file>`：要审核的 spec。
- `--project-root <dir>`：当前项目根目录。
- `--context <file-or-dir>` 可重复：补充上下文，例如 AGENTS.md、相关 skill、用户规则、已有设计约束。
- `--run-root <dir>` 可选：默认复用 `codex-dp-exec` 的 run root 发现规则。
- `--controller-id <id>` 可选：传给 `codex-dp-exec`。
- `--output <file>` 可选：审核报告输出路径。

`deepseek-review-plan` 输入：

- `--plan <file>`：要审核的实施计划。
- `--spec <file>`：对应已批准 spec。
- `--project-root <dir>`：当前项目根目录。
- `--context <file-or-dir>` 可重复：补充上下文。
- `--run-root <dir>`、`--controller-id <id>`、`--output <file>` 可选。

两个脚本都负责生成临时 prompt 文件，调用：

```bash
/home/gyq/.local/bin/codex-dp-exec \
  --cd <project-root> \
  --prompt-file <generated-prompt> \
  --task-id <stable-review-task-id> \
  --controller-id <controller-id> \
  --output-last-message <review-report>
```

审核脚本不修改文档、不执行 git commit、不运行项目测试。它们只读取输入文件并产生报告。

## Spec 审核循环

`brainstorming` 写完 spec 后的新流程：

1. Controller 执行现有 spec self-review，修复明显占位、矛盾、歧义和范围问题。
2. Controller 调用 `deepseek-review-spec`，传入 spec、项目规则、`brainstorming/SKILL.md`、`spec-document-reviewer-prompt.md` 和必要上下文。
3. DeepSeek 输出结构化审核报告，状态为 `Approved` 或 `Issues Found`。
4. 如果 `Approved`，进入用户 review gate。
5. 如果 `Issues Found`，controller 逐条判断：
   - 属实且会影响计划或实施：修改 spec。
   - 不属实或只是风格建议：不修改，但在本轮 controller 记录中说明忽略理由。
6. 修改或判断完成后再次调用 `deepseek-review-spec`。
7. 循环直到 DeepSeek 审核状态为 `Approved`。

用户 review gate 保持不变。用户要求修改 spec 后，controller 修改文档并重新执行 DeepSeek spec 审核循环。

## Plan 审核循环

`writing-plans` 和 `hybrid-writing-plans` 写完计划后的新流程：

1. Controller 执行现有 plan self-review，检查 spec 覆盖、占位、类型一致性、Hybrid readiness 等。
2. Controller 调用 `deepseek-review-plan`，传入 plan、对应 spec、项目规则、计划 skill、`plan-document-reviewer-prompt.md` 和必要上下文。
3. DeepSeek 输出结构化审核报告，状态为 `Approved` 或 `Issues Found`。
4. 如果 `Approved`，进入现有执行方式选择或 handoff。
5. 如果 `Issues Found`，controller 逐条判断：
   - 属实且会导致 implementer 构建错误、卡住、偏离 spec 或缺少验证：修改计划。
   - 不属实或只是偏好建议：不修改，记录忽略理由。
6. 修改或判断完成后再次调用 `deepseek-review-plan`。
7. 循环直到 DeepSeek 审核状态为 `Approved`。

Hybrid 计划的审核重点额外包括：每个 DeepSeek task 是否自包含、是否有稳定 `task_id`、worktree、可编辑范围、禁止范围、测试命令、报告格式、恢复规则和 controller 审核门禁。

## Prompt 上下文要求

DeepSeek 审核 prompt 必须包含：

- 当前任务角色：只读 reviewer，不修改文件。
- 目标文档路径和完整内容。
- 对应 spec 或 plan 的完整内容。
- 项目 AGENTS.md 或等效规则摘要，尤其是文档路径、中文命名、Maven、数据库、Git、worktree 规则。
- 当前涉及的 superpowers skill 规则。
- 对应 reviewer prompt 的审核标准。
- 输出格式要求。
- 校准要求：只阻塞会真实影响规划或实施的问题；风格、措辞和非必要增强放到 advisory。

输出格式固定为：

```markdown
## DeepSeek Spec Review

**Status:** Approved | Issues Found

**Blocking Issues:**
- [位置]: [问题] - [为什么会影响后续计划或实施]

**Advisory Recommendations:**
- [建议]

**Context Gaps:**
- [如果因为上下文不足无法判断，明确说明缺什么]
```

计划审核使用 `## DeepSeek Plan Review`，其余结构相同。

## Controller 判断规则

Controller 必须把 DeepSeek 反馈当作审核意见，而不是命令。

- 对每条 blocking issue，先定位文档内容和相关上下文。
- 如果问题属实，修改文档并在下一轮审核中验证。
- 如果问题不属实，保留文档不变，并在本轮工作记录中说明原因。
- 如果 DeepSeek 因上下文不足误判，下一轮 prompt 必须补充缺失上下文。
- 如果同一问题三轮仍无法收敛，controller 停止循环并向用户说明阻塞点。

## 错误处理

- `codex-dp-exec doctor` 失败：提示修复工具环境，不进入审核循环。
- DeepSeek run 失败且无报告：controller 读取 `status`、`stdout`、`stderr`，修复调用问题后重试一次；重复失败则向用户报告工具阻塞。
- DeepSeek 返回 `Context Gaps`：controller 补充上下文后重跑。
- 生成的 prompt、report、run 状态默认不提交，除非用户明确要求保留审核证据。

## 风险与约束

- DeepSeek 审核会增加时间和 token 成本，但能提前发现 spec 和计划的上下文缺口。
- 如果 prompt 上下文不足，DeepSeek 可能误报；因此脚本必须给足上下文，controller 必须做属实性判断。
- `superpowers` 上游对偏个人工作流的核心变更接受度低。本设计面向本地版本控制和个人工作流，不假定提交上游 PR。
- 审核循环可能不收敛，因此设置三轮同类问题的停止条件，避免无限循环。
