# goto 多拓扑路径与配置重构设计

## 摘要

当前 `goto` 项目的路径与连接配置同时承载了部署环境、Java 所在系统、R 所在系统、共享存储路径视图、网络连通方式等多种语义，导致 `application-common-home.yml`、`application-common-public.yml`、`application-common-wsl.yml` 中出现大量重复配置，并在业务代码中形成 `xxx` / `xxxForR`、`separator`、手工拼路径等分散逻辑。

本设计将配置模型升级为“运行拓扑 + 路径视图解析”：

- 保留现有 profile 文件名和使用习惯：`application-home.yml`、`application-public.yml`、`application-wsl.yml`
- 将 profile 的职责收缩为“声明当前拓扑 + 提供少量默认值”
- 在代码中新增统一的拓扑解析与路径解析层
- 将 Java 本地路径与 R 远端路径的差异从 YAML 中的重复字段，收敛为统一的运行时解析行为
- 逐步废弃 `separator`、`xxxForR` 以及大量可推导的派生路径配置

## 目标

- 消除现有三套公共配置文件中的大面积重复项。
- 显式表达真实运行场景，而不是使用含糊的“home/public/wsl”语义承载全部差异。
- 将“Java 读写路径”和“传给 R 的路径”统一建模，避免业务代码手工分支。
- 保留现有 profile 名称，降低启动习惯和外部脚本改造成本。
- 支持以下三个长期稳定场景：
  - 生产环境：Java 与 R 分别运行在 Linux Docker 中，通过映射目录共享文件。
  - 本地开发：Java 在 Windows IDEA 中启动，R 在 WSL 中的 Docker 中运行。
  - WSL 全链路测试：Java 在 WSL 中通过 `java -jar` 启动，R 在 WSL 中的 Docker 中运行。
- 为后续新增运行场景保留扩展点，但不引入过度自动推断。

## 非目标

- 本次设计不改变现有外部 profile 文件名。
- 本次设计不追求“完全自动识别环境，无需配置”的黑盒行为。
- 本次设计不包含业务功能重构，只聚焦配置组织与路径解析模型。
- 本次设计不要求一步删除所有旧字段，而是允许新旧模型短期并存。

## 当前问题分析

### 1. 配置语义混杂

当前公共配置文件中混在一起的实际维度至少包括：

- 部署环境：本地、生产、WSL 临时运行
- Java 进程所在系统：Windows、WSL/Linux、Docker Linux
- R 进程所在系统：WSL Docker、Linux Docker
- 文件根目录的可见路径：Java 侧视图、R 侧视图
- 网络连通方式：`localhost`、Docker 网桥地址、WSL/宿主机地址

这些维度没有被显式拆开，导致一个配置文件需要同时回答多类问题，配置重复不可避免。

### 2. 双视角路径已存在但未完成抽象

当前配置已经隐式体现了“同一资源有 Java 视图与 R 视图”这一事实，例如：

- `publicRoot` 与 `publicRootForR`
- `dataProcessRoot` 与 `dataProcessRootForR`
- `chartRoot` 与 `chartRootForR`
- `tempPathRoot` 与 `tempPathRootForR`

这说明问题本质不是“缺 profile”，而是“缺少统一的路径视图模型”。

### 3. 业务代码中存在分散的路径拼接规则

现有代码中同时存在以下模式：

- 使用 `separator` 手工拼接路径
- 使用 `File.separator` 拼接本地路径
- 手工选择 `xxx` 还是 `xxxForR`
- 对尾部斜杠进行重复补齐

这会带来以下风险：

- 规则分散，新增功能时容易继续复制错误模式
- Windows/WSL/Linux 混合场景下，路径格式容易出错
- 很多字段只是由根路径推导而来，却需要在 YAML 中重复维护

## 稳定运行拓扑

本设计将项目长期需要支持的运行关系显式定义为三个稳定拓扑：

### 1. 生产拓扑

- Java：Linux Docker
- R：Linux Docker
- 文件：通过 Docker 映射目录共享
- 特征：Java 与 R 通常看到同一套 Linux 风格路径

### 2. 本地 Windows 开发拓扑

- Java：Windows IDEA 直接启动
- R：WSL 中的 Docker
- 文件：Java 侧与 R 侧看到的路径格式不同
- 特征：同一业务资源需要同时支持 Windows 路径视图与 Linux 路径视图

### 3. 本地 WSL 全链路测试拓扑

- Java：WSL 中通过 `java -jar` 启动
- R：WSL 中的 Docker
- 文件：Java 与 R 侧都使用 Linux 风格路径
- 特征：与生产场景接近，但连接地址与默认根路径可能不同

## 方案比较

### 方案 A：主要靠 profile 切换

每个稳定场景维护一套 profile，路径、连接信息与派生目录都写入 YAML。

优点：

- 启动直观
- 易于理解

缺点：

- 配置重复仍会持续增加
- 派生目录仍然需要成片复制

### 方案 B：主要靠环境变量注入

YAML 只保留骨架，绝大部分路径与地址由环境变量注入。

优点：

- 适合 Docker 与部署平台
- 机器相关信息不容易写死进仓库

缺点：

- 本地排查不直观
- IDE、脚本、Docker、CI 之间容易形成多套环境变量维护点

### 方案 C：自动识别运行环境

程序在运行时自动识别 Windows、WSL、Docker 等环境并推导路径规则。

优点：

- 使用时最省事

缺点：

- Windows + WSL + Docker 混合场景下误判风险高
- 排查时隐式逻辑过多

### 方案 D：少量 profile 表达拓扑，细节通过环境变量覆盖，并由统一解析层生成最终路径

这是本设计的最终选择。

优点：

- 显式、稳定、可控
- 配置重复显著下降
- 既保留 profile 的可读性，又保留环境变量覆盖能力
- 适合后续逐步替换旧字段

缺点：

- 需要引入新的配置抽象与解析层
- 迁移期会有新旧模型并存的复杂度

## 一、保留现有 profile 文件名，但重定义其职责

以下文件名保持不变：

- `application-home.yml`
- `application-public.yml`
- `application-wsl.yml`

它们未来的职责不再是“复制一整份完整环境配置”，而是：

- 声明当前运行拓扑
- 提供该拓扑下少量默认路径根与连接参数
- 允许通过环境变量覆盖机器相关差异

建议语义如下：

- `application-home.yml`：本地 Windows 启动 Java，R 在 WSL Docker 中运行
- `application-public.yml`：生产 Linux Docker 到 Docker 共享挂载场景
- `application-wsl.yml`：WSL 中启动 Java，R 在 WSL Docker 中运行

## 二、配置只保留“原始事实”，不再长期保存大量派生结果

未来 YAML 应主要保存以下三类信息。

### 1. 运行拓扑声明

用于表达当前启用的是哪种稳定拓扑。

示意：

```yml
goto:
  runtime:
    topology: local-win-java-wsl-r
```

### 2. 根路径视图

用于描述同一逻辑资源在不同运行边界下看到的根路径。

示意：

```yml
goto:
  storage:
    public:
      java-root: F:/IdeaProjects/goto-software/goto-deploy-backend/public
      r-root: /root/software/public
    task:
      java-root: G:/VMSharedFolder/goto/goto-task
      r-root: /tmp/goto-task
```

如果某个拓扑下 Java 与 R 看到同一根路径，可允许使用共享根的等价写法，由程序展开。

### 3. 外部连接信息

包括但不限于：

- `rserve.host`
- `rserve.port`
- 数据库地址
- Redis 地址
- RabbitMQ 地址

这些字段继续支持环境变量覆盖，避免将机器依赖信息硬编码进仓库。

## 三、由代码统一推导派生路径

以下类型的路径不再建议长期保留为独立配置项：

- `chartRoot`
- `chartRootForR`
- `dataProcessRoot`
- `dataProcessRootForR`
- `tempPathRoot`
- `tempPathRootForR`
- `excelConvertOutputDir`
- `excelConvertOutputDirForR`
- `imageConvertOutputDir`
- `imageConvertOutputDirForR`
- `fileConvertScript`

这些字段中的大多数都可以从少量根路径与固定目录规则推导出来，应该由统一解析层在运行时生成。

## 四、引入统一的拓扑与路径解析抽象

建议新增以下核心抽象。

### 1. `RuntimeTopology`

表示当前运行拓扑，例如：

- `PROD_DOCKER_SHARED`
- `LOCAL_WIN_JAVA_TO_WSL_DOCKER_R`
- `LOCAL_WSL_JAVA_TO_DOCKER_R`

该枚举或模型负责回答：

- Java 当前运行在什么系统
- R 当前运行在什么系统
- Java 与 R 是否看到同一套路径
- 是否需要跨系统路径转换

### 2. `PathTarget`

表示路径是给谁使用的，最少包含：

- `JAVA_LOCAL`
- `R_REMOTE`

业务代码以后不再显式选择 `xxx` 还是 `xxxForR`，而是说明“我要哪一个逻辑资源在谁的视图下的路径”。

### 3. `LogicalPath` 或 `StorageKey`

表示业务上的逻辑资源，例如：

- `PUBLIC_ROOT`
- `TASK_ROOT`
- `CHART_ROOT`
- `DATA_PROCESS_ROOT`
- `TEMP_ROOT`
- `USER_NAMESPACE_ROOT`
- `EXCEL_CONVERT_OUTPUT`
- `IMAGE_CONVERT_OUTPUT`

### 4. `PathResolver`

统一负责：

- 根据当前 `RuntimeTopology` 选择路径视图
- 根据 `LogicalPath` 生成目录或文件路径
- 统一处理 Windows 与 Linux 路径拼接
- 统一处理尾部斜杠与子目录拼接
- 为 Java 本地读写生成 `JAVA_LOCAL` 路径
- 为传递给 R 的参数生成 `R_REMOTE` 路径

## 五、拆分现有配置模型职责

当前 `RProperties` 同时承担了以下职责：

- 目录结构定义
- 路径派生结果承载
- R 连接信息承载

建议逐步拆分为更清晰的模型，例如：

- `StorageProperties`：根路径、目录规则、逻辑资源配置
- `RuntimeProperties`：当前拓扑声明
- `RserveConnectionProperties`：R 服务连接信息

旧的 `RProperties` 在迁移期可暂时保留，但不应继续扩张。

## 六、明确废弃方向

以下模式应被视为待收口的历史设计：

- 配置项 `separator`
- 所有仅用于表达“给 R 看”的成对路径字段
- 所有直接由根路径拼接得出的派生目录字段
- 业务代码中手工判断路径分隔符和尾部斜杠的逻辑

迁移目标是：

- 禁止新增新的 `xxxForR`
- 禁止新增新的 `separator` 依赖
- 禁止业务代码继续直接拼接跨平台路径

## 迁移策略

建议按四步推进。

### 第一步：引入新模型，不立即删除旧字段

新增：

- `RuntimeTopology`
- `PathTarget`
- `LogicalPath`
- `PathResolver`
- 新的运行拓扑与存储配置模型

同时保留现有 `RProperties`，让系统先具备新旧两套入口。

目的：

- 降低首次重构风险
- 允许分模块渐进迁移

### 第二步：优先收口高风险路径拼接点

优先改造以下位置：

- 使用 `separator` 的代码
- 上传、下载、临时目录、数据处理目录拼接代码
- 传递给 R 的参数拼接代码
- 需要同时生成 Java 路径与 R 路径的业务入口

目的：

- 先控制最容易出错的跨平台路径逻辑
- 尽快让路径规则从业务代码中退出

### 第三步：瘦身 YAML，逐步移除派生字段

当 `PathResolver` 稳定后，再逐步清理配置文件：

- 删除 `separator`
- 删除 `xxxForR`
- 删除大量派生目录字段
- 将子目录规则内聚到解析层

目的：

- 减少配置维护面
- 防止不同 YAML 中再次出现派生字段复制

### 第四步：补齐回归保护

迁移完成前应补齐以下验证。

#### 1. 配置绑定测试

验证 `home / public / wsl` 三个 profile 下：

- 当前拓扑声明是否正确
- 根路径是否正确加载
- R 连接信息是否正确加载

#### 2. 路径解析测试

验证同一 `LogicalPath` 在不同拓扑下：

- `JAVA_LOCAL` 返回值是否正确
- `R_REMOTE` 返回值是否正确
- 是否符合 Windows/WSL/Linux 预期格式

#### 3. 关键链路集成测试

至少覆盖以下场景：

- Windows Java 场景下传给 R 的路径是否为 Linux 风格
- WSL Java 场景下 Java 与 R 是否得到同一路径
- 生产拓扑下是否不会发生无意义的路径转换

## 风险与权衡

### 1. 迁移期新旧模型并存

短期内可能出现两套属性模型同时存在的情况，但这是可接受代价。该策略优先保证平滑迁移，而不是一次性替换所有代码。

### 2. 旧代码漏改风险

最大的风险不在配置文件，而在于仍有业务代码继续手工拼接路径。为降低风险，应通过检索和测试尽可能将这些逻辑收口到统一解析层。

### 3. 不采用全自动识别

本设计有意放弃“程序自动猜测当前场景”的激进方案，换取可读性和可排障性。对于当前 Windows + WSL + Docker 的混合场景，这一取舍是合理的。

## 验收标准

满足以下条件即可认为重构设计达标：

- `application-home.yml`、`application-public.yml`、`application-wsl.yml` 文件名保持不变。
- 配置中能够显式表达当前运行拓扑。
- Java 路径与 R 路径通过统一解析层生成，而不是依赖成片 `xxxForR` 字段。
- `separator` 不再作为新的路径拼接方案继续扩散。
- 关键跨平台链路具备路径解析测试与基础集成验证。

## 决策结论

本次设计确定采用以下原则：

- 选择“少量 profile 表达拓扑，细节由环境变量覆盖，最终路径由统一解析层生成”的混合方案。
- 保留现有 `application-home.yml`、`application-public.yml`、`application-wsl.yml` 文件名与使用习惯。
- 将配置从“保存大量结果路径”转为“保存少量原始事实”。
- 将路径差异的处理职责从业务代码迁移到统一的拓扑与路径解析层。
