# 小程序会话守卫与自动登录设计文档

**日期：** 2026-03-20
**项目：** `goto-wechat`
**目标：** 为微信小程序补齐自身登录态管理，在不新增后端 refresh 接口的前提下，实现首页自动校验、静默自动登录、失效恢复和扫码登录后的统一收口。
**架构：** `pages/index/index` 作为会话守卫入口，`pages/login/login` 继续只负责扫码登录链路。

---

## 1. 背景与现状

### 1.1 已确认的现有链路

- 小程序扫码登录页 [`goto-wechat/pages/login/login.js`](../../../goto-wechat/pages/login/login.js) 会在进入后建立 WebSocket，并调用 `/auth/wechat_scan_login` 完成登录。
- 扫码登录成功后，小程序会把返回的 token 写入本地，再跳转到首页 [`goto-wechat/pages/index/index.js`](../../../goto-wechat/pages/index/index.js)。
- 小程序请求封装 [`goto-wechat/utils/request.js`](../../../goto-wechat/utils/request.js) 已经会自动携带 `Authorization` 请求头。
- 后端 `freedom-public` 中的 [`AuthorizationController.java`](../../../freedom-public/eladmin-system/src/main/java/com/freedom/modules/system/rest/AuthorizationController.java) 已提供：
  - `/auth/wechat_scan_login`
  - `/mini-program/login`
  - `/auth/info`
- 后端鉴权成功后会调用 `checkRenewal(token)`，即只要带 token 的受保护请求持续发生，Redis 在线会话会自动续期。

### 1.2 当前缺口

- 小程序没有统一的“会话守卫入口”。
- 启动首页时不会主动验证本地 token 是否仍然有效。
- token 失效后，前端当前只会提示并删除 token，没有统一的静默重登流程。
- `pages/login/login` 与普通首页启动的职责未完全分离，后续很容易把两套登录流混在一起。
- [`goto-wechat/app.js`](../../../goto-wechat/app.js) 中存在 `/pages/assistant/index` 跳转路径，与当前 `app.json` 页面配置不一致，需要收口到首页。
- 当前 token 的存储格式未规范化；若直接持久化后端返回的 `Bearer xxx`，请求层可能拼出 `Bearer Bearer xxx`。

### 1.3 设计目标

1. 小程序正常启动时，自动判断当前会话是否有效。
2. 本地无 token 或 token 已失效时，自动执行 `wx.login -> /mini-program/login` 静默登录。
3. 扫码登录成功后仍然跳到首页，但首页只做统一校验，不重复发起第二套登录流程。
4. 不新增 refresh-token 接口，不做前端定时续期。
5. 将登录态逻辑集中管理，避免分散在页面、请求封装、应用启动代码中。

### 1.4 非目标

- 不改造扫码登录页为通用登录页。
- 不新增后端 refresh / renew API。
- 不引入前端定时器主动续期。
- 不在本次设计中扩展用户信息缓存、权限模型或复杂全局状态管理。

---

## 2. 方案选择

本次采用方案 A：`pages/index/index` 作为小程序自身会话守卫入口。

不采用其他方案的原因：

- 不采用 `app.js` 全局守卫方案：当前 `app.js` 已掺杂二维码启动逻辑，继续增加鉴权编排会加重耦合。
- 不采用 refresh-token 方案：后端当前模型已经能在受保护请求中自动续 Redis 在线会话，额外引入 refresh 接口收益不足。

---

## 3. 架构设计

### 3.1 模块职责

#### `pages/login/login`

- 只负责微信扫码进入小程序后的登录链路。
- 规范入口固定为 `pages/login/login?scene=<qrUuid>`；页面内部统一将 `scene` 解析并归一成 `qrUuid`。
- 登录成功后把后端返回的认证结果交给 `session.js` 落库，再跳转到首页。
- 不负责静默登录，不负责全局续期，不负责首页守卫。

#### `pages/index/index`

- 作为首页展示入口。
- 作为统一会话守卫入口。
- 页面进入时先执行 `ensureSession()`，只有状态确认后才开放首页功能。

#### `utils/session.js`

新增会话管理模块，统一处理：

- token 读写
- 认证结果落库
- 会话清理
- 首页会话校验
- 静默自动登录
- 登录中的并发锁
- 401 失效恢复入口
- 首页失败态的一次性传递与消费

对外公开的方法约定为：

- `getToken()`
- `saveAuthResult(authInfo)`
- `clearSession()`
- `ensureSession()`
- `silentLogin()`
- `handleAuthFailure()`
- `consumeEntryState()`
- `setEntryState()`

#### `utils/request.js`

- 继续负责通用请求封装和自动带 token。
- 不再把 401 处理停留在“toast + remove token”。
- `401/403` 统一视为当前小程序会话失效，并委托给 `session.js` 处理恢复。
- `session.js` 内部发起的 `/auth/info`、`/mini-program/login` 请求必须显式绕过 401 自动恢复拦截，避免递归触发。
- 请求头中的 `Authorization` 统一由 `request.js` 负责拼接为 `Bearer <rawToken>`；`session.js` 持久化的始终是裸 token。

#### `utils/api.js`

认证接口由 `authApi` 统一暴露：

- `getInfo()` -> `/auth/info`
- `miniProgramLogin(code)` -> `/mini-program/login`
- 保留已有 `wechatScanLogin(code, qrUuid)` -> `/auth/wechat_scan_login`

#### `app.js`

- 只保留小程序启动参数处理和必要全局初始化。
- 不再保留 app 级 WebSocket 登录或扫码登录分支。
- 扫码进入的唯一业务入口为 `pages/login/login`，扫码成功后的页面跳转统一收口到 `/pages/index/index`。

### 3.2 首页状态模型

首页增加明确状态，避免 UI 和逻辑互相穿插：

- `checking`：正在校验本地 token
- `silentLoggingIn`：正在进行静默登录
- `ready`：鉴权完成，可正常使用页面
- `failed`：本次自动恢复失败，展示重试入口

状态规则：

- `ensureSession()` 仅在首页 `onLoad` 时自动触发一次。
- 若 `session.js` 中存在待消费的首页失败态，首页优先展示 `failed`，本次 `onLoad` 不再自动发起 `ensureSession()`。
- 只有用户点击“重试登录”后，首页才清除失败态并重新执行 `ensureSession()`。

## 4. 详细数据流

### 4.1 正常启动且本地 token 有效

1. 小程序进入 `pages/index/index`
2. 首页调用 `ensureSession()`
3. `ensureSession()` 发现本地存在 token
4. 调用 `/auth/info` 做主动校验
5. 校验通过，首页状态变为 `ready`
6. 后续业务请求正常发送

说明：

- 这次 `/auth/info` 同时也是一次续期触发点，因为后端鉴权成功后会调用 `checkRenewal(token)`。

### 4.2 启动时本地无 token

1. 首页调用 `ensureSession()`
2. 发现本地没有 token
3. 进入 `silentLogin()`
4. 执行 `wx.login`
5. 使用返回 code 调用 `/mini-program/login`
6. 登录成功后由 `session.js` 写入新 token
7. 再次调用 `/auth/info` 统一确认当前会话
8. 成功后首页状态变为 `ready`

### 4.3 启动时本地 token 失效

1. 首页调用 `/auth/info`
2. 后端返回 `401/403`
3. 前端清除本地 token
4. 进入 `silentLogin()`
5. 成功后重新调用 `/auth/info`
6. 校验通过后进入 `ready`

### 4.4 扫码登录成功后进入首页

1. 微信扫码进入小程序时，微信小程序码固定落到 `pages/login/login`
2. 小程序码参数使用 `scene=<qrUuid>`，由 `login.js` 归一解析为 `qrUuid`
3. 页面通过 `/auth/wechat_scan_login` 登录成功
4. 登录页调用 `session.js.saveAuthResult(authInfo)`，由 `session.js` 完成裸 token 落库
5. 页面跳转 `pages/index/index`
6. 首页执行 `ensureSession()`
7. 因 token 已存在，只会调用 `/auth/info` 校验
8. 校验通过后进入 `ready`

说明：

- 该流程不会再次执行 `wx.login -> /mini-program/login`，因此与扫码登录链路不冲突。
- `app.js` 不再参与扫码登录业务编排，避免和 `login.js` 形成双入口。

### 4.5 已进入业务页面后遇到 `401/403`

1. 某业务请求返回 `401` 或 `403`
2. `request.js` 将该事件交给 `session.js`
3. `session.js` 判断当前是否已有登录恢复流程在执行
4. 若无，则发起一次统一恢复；若已有恢复流程，则等待同一个恢复 Promise 完成
5. `request.js` 必须等待该恢复 Promise 完成后，再决定原始请求的返回结果
6. 若恢复成功，原始业务请求以“会话已恢复，请手动重试”的错误返回，不做自动重放
7. 页面保持在当前页，由用户手动重试刚才的操作
8. 若恢复失败，统一 `reLaunch` 回 `pages/index/index`，由首页展示 `failed` 状态和“重试登录”入口

## 5. 错误处理与并发控制

### 5.1 `/auth/info` 的错误策略

- `401/403`
  - 视为会话无效
  - 清理本地 token
  - 进入 `silentLogin()`
- 网络异常、超时、`5xx`
  - 不清理 token
  - 首页进入 `failed` 状态
  - 允许用户手动重试

这样可以避免临时网络抖动把本地有效 token 错误清除。

### 5.2 `silentLogin()` 的错误策略

- `wx.login` 失败：视为微信侧登录能力失败
- `/mini-program/login` 失败：视为后端静默登录失败

两种情况都采用同一策略：

- 单次自动尝试
- 不做无限重试
- 失败后由 `session.js` 设置一次性首页失败态
- 再统一 `reLaunch` 到首页，由首页展示“重试登录”按钮

### 5.3 登录中锁

`session.js` 必须维护单例级“登录中锁”，避免出现以下问题：

- 首页启动时正在静默登录，其他请求又因 401 再次发起登录
- 多个并发请求各自拿到不同 token，相互覆盖本地存储
- 页面状态与本地 token 不一致

实现要求：

- 同一时刻只允许一个 `ensureSession()` / `silentLogin()` 流程在执行
- 后续请求如果发现已有恢复流程，应等待同一个 Promise，而不是重新发起登录
- 原始业务请求必须等待恢复 Promise 结束后再返回结果
- 登录恢复成功后，不自动重放之前失败的业务请求；失败请求以“会话已恢复，请手动重试”的错误返回，由页面侧按需手动重试
- 登录恢复最终失败时，由 `session.js` 负责统一跳转回首页 `failed` 状态，而不是让各个业务页自行决定去向

### 5.4 会话内部请求的拦截豁免

`session.js` 发起的以下请求属于会话内部请求：

- `/auth/info`
- `/mini-program/login`

这些请求必须带一个显式标记，例如 `skipAuthRecovery: true`，用于告诉 `request.js`：

- 这些请求仍然正常携带 token（若有）
- 这些请求遇到 `401` 时，不再走“401 -> 委托 `session.js` -> 再发会话请求”的统一恢复链路
- 这些请求遇到 `403` 时，也同样不再走统一恢复链路
- 它们的失败由 `session.js` 自己直接判断并决定下一步动作

这样可以避免 `session.js` 在恢复过程中再次被 `request.js` 递归拉起，导致死循环或状态错乱。

### 5.6 Token 存储格式约束

`session.js` 统一负责 token 的规范化与持久化，格式约束如下：

- 后端若返回 `Bearer xxx`，`session.js` 落库前必须去掉前缀，只存 `xxx`
- 后端若返回裸 token，则按原样存储
- `wx.setStorageSync` 中保存的始终是裸 token
- `request.js` 在构建请求头时，统一拼接 `Authorization: Bearer <rawToken>`

这样可以避免请求头出现 `Bearer Bearer xxx` 的重复前缀问题。

### 5.5 首页失败态传递机制

恢复失败后，不能只依赖页面内存状态，因为 `reLaunch` 会创建新的首页实例。

本方案要求：

- `session.js` 维护一个一次性“首页入口状态”容器，例如：
  - `mode: normal`
  - `mode: failed`
- 当恢复失败时，`session.js` 先写入 `mode: failed`，再执行 `reLaunch('/pages/index/index')`
- 新首页实例在 `onLoad` 时先读取并消费这个状态：
  - 若为 `failed`，则直接渲染失败页，不自动执行 `ensureSession()`
  - 若为 `normal` 或为空，才进入常规自动校验流程
- 用户点击“重试登录”后，由首页调用 `session.js` 清除该一次性状态，并重新执行 `ensureSession()`

这样可以保证：

- 恢复失败后不会因为回到首页而立即再次自动登录
- 首页失败态是显式、可消费、可清除的
- 页面跳转与会话状态机保持一致

## 7. 验收标准

### 7.1 功能验收

1. 本地无 token 时，直接打开小程序，应自动执行静默登录并进入首页。
2. 本地有有效 token 时，打开小程序应只触发 `/auth/info` 校验，不重复调用 `/mini-program/login`。
3. 本地 token 失效时，打开小程序应先校验失败，再静默登录成功，最终进入首页。
4. 扫码登录成功后跳转首页，首页只执行统一校验，不重复发起第二套登录。
5. 业务页请求返回 401 时，不应只停留在 toast 提示和删 token，而应统一进入恢复流程。
6. 业务页请求返回 403 时，也按同一套会话失效恢复流程处理。
7. 业务页触发 `401/403` 后，原始请求会等待恢复流程结束，但不自动重放；如果恢复成功，用户停留在当前页并手动重试当前操作。
8. 自动恢复失败时，小程序应统一回到首页可重试状态，而不是死循环、空白页或半登录状态。

### 7.2 技术验收

1. 所有 token 读写都收口到 `session.js`。
2. 扫码登录页不直接操作 `wx.setStorageSync("token")`，而是通过 `session.js` 统一落库。
3. `request.js` 中 `401/403` 处理不再直接结束，而是委托会话管理模块，并等待恢复 Promise 结束后再返回错误。
4. 首页只有在 `ready` 状态下才允许用户继续正常操作。
5. 恢复失败后回到首页时，不会在 `onLoad` 立即再次自动登录，而是停留在 `failed` 状态等待用户手动重试。
6. `app.js` 中不存在跳转到无效页面 `/pages/assistant/index` 的逻辑残留，也不存在第二套 app 级扫码登录编排逻辑。
7. 小程序本地持久化的 token 为裸 token，请求头由 `request.js` 统一补 `Bearer ` 前缀。

## 8. 风险与后续计划

### 8.1 已知风险

- 若后端 `/mini-program/login` 与 `/auth/info` 的用户上下文不完全一致，静默登录后再次调用 `/auth/info` 可能暴露数据结构差异。
- 业务页面在恢复前若继续发起请求，可能会出现一次性失败提示，因此请求层与首页守卫的调用顺序需要明确。

### 8.2 后续计划边界

本设计完成后，下一步应进入 implementation plan 阶段，细化：

- `session.js` 的接口定义
- 首页状态切换与占位 UI
- `request.js` 的 401 委托机制
- 扫码登录页与首页跳转的收口修改
- 手工验证路径与回归清单

---

## 9. 结论

本方案通过“首页守卫 + 静默登录 + 统一 401 恢复”补齐小程序自身会话管理能力，同时保持扫码登录页职责单一，并继续复用后端现有的 token 校验与续期机制。方案范围清晰、依赖现有后端能力、实现成本可控，适合直接进入实施计划阶段。
