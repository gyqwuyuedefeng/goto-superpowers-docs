# Rserve 临时目录与通用任务重试闭环修复设计

## 背景

2026-07-21，`goto/task` 的 Excel 宽转长任务连续失败。R 调用链中的首个业务异常为：

```text
Error in wb$saveWorkbook() :
  Failed to create temporary directory
  '/tmp/RtmpiXaBPk/workbookTemp_14dd930610b5c'

In dir.create(path = tmpDir, recursive = FALSE) :
  No such file or directory
```

Java 随后又把异常文本当作 `RInvokeResponse` JSON 解析，产生伴生的 `JsonParseException`。任务账本最终进入 `FAILED_RETRYABLE`，消息不通知、不 ACK。

本设计处理 Rserve 临时目录失效、跨容器进程关闭、R 异常解析和通用任务有限重试闭环。经确认，本次不修改 `gotoTools`、`gotoBase` 和 `rsession`。

## 现场证据与根因

### Excel 保存调用链

当前调用链为：

```text
gotoTools::reshapeData()
→ wide2LongData()
→ gotoBase::saveTable()
→ openxlsx::write.xlsx()
→ wb$saveWorkbook()
→ dir.create(tempfile("workbookTemp_"), recursive = FALSE)
```

故障发生时，`openxlsx` 要创建的子目录位于 `/tmp/RtmpiXaBPk` 下。错误是 `No such file or directory`，说明父级 R 会话临时目录在当时已经不存在，而不是输出 Excel 目录错误。

### Rserve 启动配置未生效

现有启动脚本把所有命令写在同一行，并使用 `&` 分隔：

```bash
rstudio-server start \
  & rm -rf /root/software/rserve/working_root \
  & mkdir -p /root/software/rserve/working_root \
  & export TMPDIR=/tmp/r_temp TEMP=/tmp/r_temp TMP=/tmp/r_temp \
  & R CMD Rserve --RS-conf /home/goto/host/rserve.conf --no-save \
  & tail -f /dev/null
```

由此产生三个问题：

1. `export` 在后台子 Shell 中执行，不会传递给随后启动的 Rserve。
2. `/tmp/r_temp` 从未创建。
3. 工作目录删除和创建并发执行，存在时序竞争。

线上只读检查已经确认：

- Rserve 主进程环境中没有 `TMPDIR`、`TMP`、`TEMP`。
- `/tmp/r_temp` 不存在。
- Rserve 实际使用 `/tmp/RtmpiXaBPk`。
- `/tmp` 尚有约 4.2 GB 可用空间，inode 使用率约 19%，不是磁盘或 inode 耗尽。
- `goto-rbase` 自 2026-06-18 启动后未重启。

### Rserve fork 临时目录风险

Unix 下 Rserve 为连接 fork 子进程。fork 出来的 R 进程继承并共享 R 启动时确定的会话临时目录。某个子进程退出或外部清理动作删除该目录后，仍在运行或后续 fork 的进程可能继续持有旧路径。

R 官方提供 `tempdir(check = TRUE)`，用于检查并在失效时重建会话临时目录。仅设置运行中的 `Sys.setenv(TMPDIR=...)` 不会改变当前 R 会话的 `tempdir()`；环境变量必须在 Rserve 主进程启动前生效。

故障发生两天后，`/tmp/RtmpiXaBPk` 已重新存在，但容器没有重启。无法追溯它由哪个进程或操作重新创建。这个未知项不影响根因判断：日志证明故障时父目录缺失，线上环境证明预期的 `TMPDIR` 配置从未生效。

### 跨容器 kill 无效

`ROperationUtils.closeRserveSession()` 根据 Rserve 工作目录 `conn<PID>` 拼出：

```text
kill -9 <PID>
```

该命令在 `goto-task` 容器执行，而目标 Rserve 子进程位于 `goto-rbase` 容器。线上 PID namespace 分别为：

```text
goto-task:  pid:[4026532381]
goto-rbase: pid:[4026532418]
```

两个容器没有使用 host PID 模式，因此 `goto-task` 无法通过该 PID 杀掉 `goto-rbase` 中的 Rserve 子进程；如果 `goto-task` 内恰好存在同号进程，还存在误杀风险。

线上还发现多个存活数周的 Rserve 子进程。它们可能来自不同调用方，本设计先补齐连接生命周期观测和任务侧幂等关闭，不在来源未确认前批量杀进程。

### R 异常被错误解析为 JSON

`ROperationUtils.evalReturn()` 捕获 `Rsession.RException` 后，把 `e.getMessage()` 传入正常响应反序列化方法。`Failed to create temporary directory ...` 不是 JSON，因此产生第二个 `JsonParseException`。

这个异常是日志污染和错误分类干扰，不是 Excel 保存失败的原因。

### 当前可重试状态无法形成闭环

现有异步完成链路存在以下不一致：

1. 后台失败后把账本从 `RUNNING` 更新为 `FAILED_RETRYABLE` 并增加 `retry_count`。
2. 任务认领 SQL 只允许 `PENDING` 或租约过期的 `RUNNING` 进入 `RUNNING`，没有包含 `FAILED_RETRYABLE`。
3. `CommonTaskCompletionContext.retryCount` 每次固定写成 `0`，没有使用数据库账本值。
4. `@StreamListener(maxRetries = 3)` 只覆盖 listener 同步执行阶段，无法限制已经提交线程池后的异步 R 失败。

因此“标记可重试、不 ACK”可能让消息长期停留在 PEL，而不是完成有限重试。

## 目标

1. 确保 Rserve 启动前获得真实存在、权限正确的专用临时目录。
2. 每次业务 R 调用前检查并尝试一次临时目录自愈。
3. 删除 `goto-task` 跨容器按 PID 杀 Rserve 子进程的行为。
4. 保留原始 R 错误，消除异常文本的二次 JSON 解析。
5. 让 `FAILED_RETRYABLE` 能够重新认领，并受持久化最大重试次数约束。
6. 确保可重试任务最终成功或明确进入 `FAILED_FINAL`，不会无限留在 PEL。
7. 保持现有 Excel 输出内容和表头样式不变。

## 非目标

1. 不修改 `gotoTools`。
2. 不修改 `gotoBase`。
3. 不修改 `rsession`。
4. 不把 `openxlsx` 切换成 `writexl`。
5. 不在本设计中批量清理线上 Rserve 子进程。
6. 不通过手工数据库 DML 修复现有任务。
7. 不重构所有 R 调用接口或 Redis Stream 框架。

## 方案比较

### 方案一：平台根因修复

修正 Rserve 启动环境，在通用 R 连接层增加临时目录检查、自愈、错误分类和幂等关闭，同时补齐通用任务有限重试。

优点：

- 覆盖所有依赖 R 临时目录的任务。
- 不改变 Excel 业务输出。
- 同时解决伴生异常和任务无法收敛的问题。

缺点：

- 涉及 Rserve 镜像、部署脚本、共享 R 调用库和 `goto/task`。
- 需要真实 Rserve 集成验证。

本设计采用此方案。

### 方案二：仅切换 Excel 写出库

把 `reshapeData()` 恢复为 `writexl::write_xlsx()`，绕过本次 `openxlsx` 的 `workbookTemp_*` 路径。

优点是改动小、恢复快；缺点是可能失去表头样式，而且没有修复 Rserve 临时目录、错误解析、跨容器 kill 和任务重试问题。

本设计不采用。

### 方案三：仅运维重启或定期建目录

重启 `goto-rbase` 会生成新的会话临时目录，可作为紧急恢复手段。定期创建目录只能修复当前路径，无法保证 Rserve 主进程环境和任务状态闭环。

该方案只能用于止血，不作为最终修复。

## 组件责任

### goto-rbase-build

负责镜像内 Rserve 的启动和运行目录准备：

1. 使用严格 Shell 模式，启动步骤串行执行。
2. 创建 `/tmp/r_temp`。
3. 把共享基目录权限设置为带 sticky bit 的 `1777`；具体 `Rtmp*` 子目录仍由 R 管理。
4. 创建 `/root/software/rserve/working_root`。
5. 在启动 Rserve 前导出 `TMPDIR`、`TMP`、`TEMP`。
6. 目录创建、权限设置或可写检查失败时终止启动，不能带着无效配置继续运行。

### goto-deploy-backend

同步保存与镜像一致的 Rserve 启动脚本和部署说明，避免构建仓库与部署仓库配置漂移。

当前 `docker-compose.yml` 没有给 `goto-rbase` 挂载外部 entrypoint，因此生产生效事实源是新构建的 `goto-rbase` 镜像；部署仓库中的脚本必须明确其用途，不能让运维误以为修改宿主机文件即可影响现有容器。

### freedom-public/common-thread-r

负责通用 Rserve 连接生命周期：

1. 建立连接。
2. 在执行业务命令前执行临时目录健康检查。
3. 失败时关闭当前连接并重新建立一次连接。
4. 第二次仍失败则返回稳定错误类别和原始错误。
5. 任务结束时只调用客户端连接关闭，不再执行跨容器 Shell kill。
6. 关闭入口必须幂等。
7. 记录连接建立、临时目录检查、业务执行和连接关闭的结构化日志。

### goto/common

负责 Job Ledger 状态和持久化重试事实：

1. `stream_job_execution.retry_count` 是重试次数唯一事实源。
2. 认领操作允许满足重试条件的 `FAILED_RETRYABLE` 转为 `RUNNING`。
3. 状态更新继续校验 `owner_id`、`fencing_token` 和当前状态。
4. 最大重试次数配置化，默认值与通用任务监听器的重试语义保持一致。

本设计不要求新增数据库字段，优先复用现有 `retry_count`、`completed_at`、`status_version` 和 fencing token。

### goto/task

负责异步任务失败决策、通知和 ACK：

1. 从 Job Ledger 获取当前 `retry_count`，不再把完成上下文重试次数固定为零。
2. Rserve 连接异常和临时目录异常可重试。
3. 确定性业务错误直接最终失败。
4. 达到最大重试次数后写 `FAILED_FINAL`、发布失败通知并 ACK。
5. 未达到上限时写 `FAILED_RETRYABLE`，不发布终态通知、不 ACK，等待 PEL 恢复链路重新获取。
6. 失权 owner 继续遵循不写状态、不通知、不 ACK。

### gotoTools、gotoBase、rsession

本次不修改：

- `gotoTools::reshapeData()` 保持原有数据转换行为。
- `gotoBase::saveTable()` 继续使用 `openxlsx`，保留表头样式。
- `rsession` 继续作为 Java 与 Rserve 的传输适配层。

只有后续独立测试证明 `R.close()` 自身存在传输层缺陷时，才另行设计 `rsession` 修复。

## 任务数据流

```text
Redis Stream 收到任务
→ 创建或读取 Job Ledger
→ 校验重试次数和任务状态
→ 认领执行权并进入 RUNNING
→ 提交后台线程池
→ 建立 Rserve 连接
→ 检查 tempdir(check = TRUE)
→ 创建并删除可写探针
→ 执行 gotoTools::reshapeData()
→ openxlsx 写出 Excel
→ 返回结构化成功或稳定错误类别
→ 幂等 R.close()
→ 写入任务终态
→ 发布通知
→ ACK
```

临时目录检查第一次失败时，在同一次任务内执行：

```text
关闭当前连接
→ 新建一次连接
→ 再次执行临时目录检查
```

该次连接内自愈不增加 Job Ledger 的 `retry_count`。

## 临时目录健康检查

连接建立后执行的检查必须验证：

1. `tempdir(check = TRUE)` 返回非空路径。
2. 目录真实存在。
3. 当前 Rserve 子进程对目录具有写权限。
4. 能在目录内创建并删除一个唯一探针文件。

检查日志至少包含：

- Rserve 地址
- `tempdir()` 路径
- 目录是否存在
- 是否可写
- 当前连接工作目录 `conn<PID>`
- 自愈尝试次数

日志不得包含认证信息或业务文件内容。

## 错误模型

通用 R 调用返回稳定错误类别，并保留原始错误文本：

- `R_CONNECTION_UNAVAILABLE`：连接失败、连接重置或连接超时。
- `R_TEMP_DIR_UNAVAILABLE`：临时目录检查或探针失败。
- `R_EXECUTION_ERROR`：R 正常执行后返回业务或运行错误。
- `R_RESPONSE_INVALID`：R 正常返回，但响应不符合协议。

错误类别用于重试决策，原始错误文本用于日志和最终失败通知。

只有正常返回的 R 字符串才进入 `RInvokeResponse` JSON 反序列化。捕获 `Rsession.RException` 后直接构造失败结果，不再调用正常响应反序列化方法。

## 重试与状态流转

### 允许重试

- `R_CONNECTION_UNAVAILABLE`
- `R_TEMP_DIR_UNAVAILABLE`
- 数据库、Redis 等短暂基础设施异常
- 明确的任务执行中断

### 直接最终失败

- 参数非法
- 输入数据或文件格式错误
- 必需文件不存在
- R 运行时依赖明确缺失
- R 返回确定性业务失败

### 未知异常

未知异常可以按可重试处理，但必须受持久化最大重试次数约束，不能无限留在 PEL。

### 状态机

未达到最大重试次数：

```text
RUNNING
→ FAILED_RETRYABLE（retry_count + 1）
→ PEL 恢复
→ FAILED_RETRYABLE 重新认领为 RUNNING
```

达到最大重试次数：

```text
RUNNING
→ FAILED_FINAL
→ 发布失败通知
→ ACK
```

成功路径保持：

```text
RUNNING
→ SUCCEEDED
→ 发布成功通知
→ ACK
```

最大重试次数使用配置项，默认允许 3 次重试。首次执行和连接内的一次自愈不计入这 3 次异步重试。

## Rserve 子进程治理

本次删除 task 容器内的跨容器 kill，但不直接把当前长期子进程归因于某一个调用方。

上线后增加以下观测：

- 每次 Rserve 连接建立和关闭的关联标识。
- task 侧连接建立总数、关闭总数和关闭异常数。
- Rserve 子进程数量与最长存活时间。
- Job Ledger 中 `RUNNING`、`FAILED_RETRYABLE` 的数量与停留时间。

若连接关闭日志完整但 Rserve 子进程仍持续增长，再单独采集连接来源和 Rserve 服务端日志，形成独立修复，不在本次实现中批量 kill。

## 测试策略

### 启动脚本测试

1. `bash -n` 校验语法。
2. ShellCheck 校验后台命令、变量和错误处理。
3. 在隔离容器中验证目录准备和启动顺序。
4. 确认 Rserve 主进程环境包含正确的 `TMPDIR`、`TMP`、`TEMP`。
5. 确认 `tempdir()` 位于 `/tmp/r_temp/Rtmp*`，且目录存在、可写。

### freedom-public 单元测试

1. 临时目录正常时直接执行业务调用。
2. 临时目录缺失时，`tempdir(check = TRUE)` 能完成自愈。
3. 目录不可写时返回 `R_TEMP_DIR_UNAVAILABLE`。
4. 第一次检查失败时只重建一次连接。
5. 第二次失败时不继续循环重连。
6. `Rsession.RException` 不进入 JSON 反序列化。
7. 关闭连接不执行 `SystemUtils.exec("kill -9 ...")`。
8. 多次关闭同一连接保持幂等。

### goto/common 与 goto/task 测试

1. `FAILED_RETRYABLE` 在未超限时可以重新认领为 `RUNNING`。
2. 重试次数读取自数据库账本。
3. 未达到上限时不发布终态通知、不 ACK。
4. 达到上限后写 `FAILED_FINAL`、通知并 ACK。
5. 成功路径保持 `markSucceeded → publishNotification → ack`。
6. 业务不可重试失败保持 `markFinalFailure → publishNotification → ack`。
7. `isStillOwner=false` 时不写状态、不通知、不 ACK。
8. 同一个幂等键不会产生并行执行记录。
9. `@StreamListener` 同步重试次数和异步账本重试次数不会互相覆盖或重复累计。

### 真实 Rserve 集成测试

测试只在隔离容器执行：

1. 连续建立、关闭多轮 Rserve 连接。
2. 删除测试会话的 `Rtmp*`，验证下一次连接检查能够自愈。
3. 使用 `openxlsx` 写出 Excel，并验证文件可正常读取。
4. 调用现有 `gotoTools::reshapeData()` 执行宽转长。
5. 重复执行原始任务场景，确认不再出现 `workbookTemp_*` 错误。
6. 对比测试前后的 Rserve 子进程数量，确认 task 创建的连接关闭后没有持续增长。

## 回滚策略

1. Rserve 启动脚本修复与现有 R 包兼容，可优先保留。
2. 如果共享 R 调用层出现回归，回滚 `freedom-public` 和 `goto/task` 到上一版本。
3. 回滚不会修改 `gotoTools`、`gotoBase`、`rsession` 或 Excel 文件格式。
4. 回滚期间若再次发生临时目录故障，可通过重启 `goto-rbase` 临时恢复，但必须保留告警并尽快重新部署根因修复。

## 验收标准

1. Rserve 主进程环境包含：

   ```text
   TMPDIR=/tmp/r_temp
   TMP=/tmp/r_temp
   TEMP=/tmp/r_temp
   ```

2. R 的 `tempdir()` 位于 `/tmp/r_temp/Rtmp*`。
3. 原始 `w2l` Excel 转换成功，输出内容和表头样式不变。
4. 日志中不再出现 `workbookTemp_*` 创建失败。
5. R 错误不再伴随 `JsonParseException`。
6. `goto-task` 不再输出跨容器 `kill -9 <Rserve PID>`。
7. 可重试任务最终成功或在达到上限后明确失败并 ACK。
8. `FAILED_RETRYABLE` 不会无限停留在 PEL。
9. 连续任务执行不会导致 task 创建的 Rserve 子进程持续增长。
10. `gotoTools`、`gotoBase`、`rsession` 没有代码变更。
