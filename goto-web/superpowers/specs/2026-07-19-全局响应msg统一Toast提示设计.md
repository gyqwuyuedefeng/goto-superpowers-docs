# goto-web 全局响应 msg 统一 Toast 提示设计

## 背景

`goto-web` 通过 `src/utils/request.js` 创建统一 Axios 实例。当前响应拦截器在请求成功时只处理下载响应并返回 `response.data`，不会展示统一响应中的 `msg`；只有 HTTP 异常分支会直接调用 `Vue.$toast.error(...)`。

前端已有 `src/utils/vueToastificationNotice.js`，并在 `src/main.js` 中以 `this.ToastificationNotice` 的形式挂载到 Vue 原型。页面因此存在较多局部成功或错误提示，但这些提示依赖调用方逐个编写，容易遗漏。

后端 `CommonResponse` 的统一结构为：

```text
{
  code: number,
  msg: string | null,
  data: unknown
}
```

其中 `ResponseUtils` 的约定是：

- `code === 200`：业务成功；
- `code === 0`：业务失败；
- `success()`、`successData(...)` 不设置 `msg`；
- `successMsg(...)`、`success(data, msg)`、`failureMsg(...)` 会设置 `msg`；
- 业务失败经常仍使用 HTTP 200 返回，因此也会进入 Axios 的成功响应拦截器。

项目还存在 `code === 2000` 等旧协议响应，不能按 `CommonResponse` 的 200/0 规则推断提示类型。

## 目标

1. 所有经过统一 Axios 实例的 `CommonResponse`，只要 `msg` 是非空字符串，就自动展示 Toast。
2. `code === 200` 使用成功 Toast，`code === 0` 使用错误 Toast。
3. 普通 `success()` 或其他没有 `msg` 的响应不自动提示。
4. 复用现有 `ToastificationNotice` 的样式和交互配置。
5. 移除同一次接口响应产生的页面重复提示，让后端 `msg` 成为该响应的唯一提示内容。
6. 不改变 Axios Promise 的返回数据、下载处理、鉴权跳转和异常处理行为。

## 非目标

1. 不为没有 `msg` 的成功响应补默认“操作成功”。
2. 不自动处理 `code === 2000` 或其他非 200/0 协议。
3. 不修改后端 `CommonResponse`、`ResponseUtils` 或现有接口返回文案。
4. 不统一 WebSocket 通知、表单校验、前端条件提示或非 Axios 通知。
5. 不新增按请求关闭全局提示的配置开关。
6. 不在本次工作中重构 HTTP 401、403、超时或服务器异常处理。

## 架构设计

### ToastificationNotice

扩展 `src/utils/vueToastificationNotice.js`，新增不依赖 Vue 组件实例的统一响应提示入口，建议接口为：

```js
notifyResponse(toast, responseData)
```

参数职责：

- `toast`：全局 `Vue.$toast` 实例；
- `responseData`：Axios 成功响应中的 `response.data`。

该入口完成以下判断：

1. `responseData` 不是普通对象时直接返回。
2. `msg` 不是字符串，或 `msg.trim()` 为空时直接返回。
3. `code === 200` 时展示成功 Toast。
4. `code === 0` 时展示错误 Toast。
5. 其他 `code` 不提示。

实际展示内容使用去除首尾空白后的 `msg`。

为避免出现两套 Toast 配置，将现有格式拼接、位置、时长和交互参数下沉到一个可接收 toast 实例的内部方法。原有组件 API 保持不变：

```js
notifySuccess(context, title, message)
notifyError(context, title, message)
notifyWarning(context, title, message)
notifyInfo(context, title, message)
notifySimple(context, title, message, messageType)
```

这些方法继续通过 `context.$toast` 工作，不要求修改现有页面调用。

### Axios 响应拦截器

`src/utils/request.js` 的成功响应处理顺序调整为：

```text
收到 Axios response
  → 执行现有 ifDownload(response)
  → ToastificationNotice.notifyResponse(Vue.$toast, response.data)
  → 返回 response.data
```

`notifyResponse` 只观察响应数据，不修改对象。因此所有 API 调用方仍取得原有 `response.data`，对象引用也保持不变。

下载响应经 `ifDownload` 处理后可能变为 `{ blob, fileName }`，由于没有符合规则的 `code/msg`，不会产生 Toast。其他 Blob、字符串、数组或空响应同样不会提示。

## 数据流与行为矩阵

| 响应数据 | 行为 |
| --- | --- |
| `{ code: 200, msg: "保存成功" }` | 展示一次成功 Toast“保存成功” |
| `{ code: 0, msg: "保存失败" }` | 展示一次错误 Toast“保存失败” |
| `{ code: 200, msg: null }` | 不提示 |
| `{ code: 200, msg: "   " }` | 不提示 |
| `{ code: 2000, msg: "上传成功" }` | 不自动提示 |
| `{ code: 500, msg: "异常" }` | 不自动提示 |
| Blob、数组、字符串、`null` | 不提示 |

`code` 使用严格数值比较。字符串 `"200"` 或 `"0"` 不视为统一协议，避免把结构不一致的返回误判为 `CommonResponse`。

## 重复提示清理规则

前端清理以“同一次已解析接口响应”为边界，并逐个核对对应后端返回契约。

应移除：

1. `then`/`await` 成功返回后，针对 `successMsg(...)` 或 `success(data, msg)` 再手动展示的成功 Toast。
2. 已解析到 `{ code: 0, msg }` 后，直接展示 `res.msg`、`response.msg` 或同义硬编码文案的错误 Toast。
3. 同一次响应中与后端 `msg` 等价、会和全局提示连续出现的 `$message` 或 `ToastificationNotice` 调用。

应保留：

1. 后端使用 `success()`、`successData(...)` 且不返回 `msg` 时，页面原有的业务成功提示。
2. 表单校验、前端条件判断、用户操作限制等未发起接口请求的提示。
3. `catch` 中对前端运行异常的上下文提示。
4. WebSocket 任务提交、执行进度、完成或失败通知。
5. `code === 2000` 等非统一协议的页面提示。
6. 页面状态更新、数据刷新、弹窗关闭、Loading 回收和事件派发逻辑。

如果局部提示与后端 `msg` 文案不同，但属于同一次具有非空 `msg` 的响应，仍以后端 `msg` 为唯一用户提示。页面不得为了保留旧文案而再次弹 Toast。

## 错误处理

本次只处理 Axios 成功响应中的业务 `code/msg`：

- HTTP 401 继续执行退出登录和页面刷新；
- HTTP 403 继续跳转 401 页面；
- 超时继续提示“网络请求超时”；
- Blob 下载错误继续解析后端 `message`；
- 其他 HTTP 错误继续使用现有 `message` 或“接口请求失败”。

业务 `code === 0` 不主动 `Promise.reject`，仍按现状把响应交给调用方。全局错误 Toast 只补充用户反馈，不改变控制流，避免引发全站调用行为变化。

## 测试设计

### ToastificationNotice 单元测试

新增 `tests/unit/utils/vueToastificationNotice.spec.js`，至少覆盖：

1. `code === 200` 且 `msg` 非空时调用一次 success Toast。
2. `code === 0` 且 `msg` 非空时调用一次 error Toast。
3. `msg` 为 `null`、`undefined`、空字符串或纯空白时不调用 Toast。
4. `code === 2000`、未知响应码、字符串响应码时不调用 Toast。
5. `responseData` 为 Blob、数组、字符串、`null` 或 `undefined` 时不调用 Toast。
6. `msg` 首尾空白被去除，正文内容保持不变。
7. 现有 `notifySuccess`、`notifyError` 等组件 API 的行为和配置保持不变。

### 响应拦截器测试

新增或补充 `tests/unit/utils/request.spec.js`，通过 mock Axios 拦截器和 Toast 入口验证：

1. 成功响应只调用一次 `notifyResponse`。
2. `notifyResponse` 接收到 `Vue.$toast` 和处理下载后的 `response.data`。
3. 拦截器返回值与原始业务数据保持一致，不额外包裹或复制。
4. 下载响应不会触发成功或错误 Toast。
5. 现有 HTTP 异常分支不因本次改动发生回归。

### 页面回归测试

更新所有因移除局部重复提示而受影响的现有 Jest 测试。测试重点从“页面调用了成功/错误 Toast”调整为：

1. 页面不再针对同一 `msg` 调用局部 Toast。
2. 成功后的状态更新、刷新、弹窗关闭等业务逻辑仍执行。
3. 失败后的 Loading 回收和状态恢复仍执行。
4. 无后端 `msg` 的页面提示、前端校验提示和 WebSocket 提示保持原行为。

## 验收标准

1. 任意统一 Axios 请求返回 `{ code: 200, msg: 非空字符串 }` 时，只出现一次成功 Toast。
2. 任意统一 Axios 请求返回 `{ code: 0, msg: 非空字符串 }` 时，只出现一次错误 Toast。
3. 普通 `success()`、`successData(...)` 以及空白 `msg` 不产生全局 Toast。
4. `code === 2000`、下载响应和未知结构不产生全局 Toast。
5. 页面不再对同一次带 `msg` 的响应重复提示。
6. Axios 调用方收到的数据和当前版本一致。
7. HTTP 异常、鉴权跳转、下载、WebSocket 和前端校验行为保持不变。
8. 新增定向测试、受影响页面测试和 ESLint 全部通过。

## 影响范围

主要修改范围：

- `goto-web/src/utils/vueToastificationNotice.js`
- `goto-web/src/utils/request.js`
- `goto-web/tests/unit/utils/`
- 具有响应重复提示的页面及其现有测试

后端 `goto` 与公共库 `freedom-public` 仅作为返回契约依据，不在本次实现范围内。
